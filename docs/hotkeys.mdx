# Hotkeys and Keyboard Shortcuts

> **Document type:** Product specification.
> **Audience:** Engineering (implementation) and Product (key choices,
> conflict resolution).
> **Owners:** PM for key bindings and discoverability; CTO for the
> SwiftUI `.keyboardShortcut` + AppKit `NSResponder.keyDown` wiring and
> first-responder routing.
> **Status:** v1 spec for the Mac-native fork. Complements
> [`menus.mdx`](./menus.mdx) (which owns chorded `⌘`-prefixed
> shortcuts on menu items) and [`dir_ui.mdx`](./dir_ui.mdx) (which owns
> the Directory Panel selection model).

---

## 1. Purpose

This document specifies the **single-key and lightly-modified keyboard
shortcuts** of ImageGlass_Mac that are *not* attached to a menu-bar
item. These are the fast, in-flow keys a user presses while looking at
an image or scrolling the file list — the kind of muscle-memory keys
that make an image viewer feel quick.

The menu bar in `menus.mdx` owns the canonical `⌘`-prefixed shortcuts.
This document owns the unmodified letter keys (`C`, `N`, `Z`, `W`) and
the arrow-key navigation model. Where the two overlap, this document
is the source of truth for the bare-letter binding and `menus.mdx` is
the source of truth for the menu-bar twin.

---

## 2. Design Principles

1. **Single-letter keys are reserved for high-frequency, viewer-local
   actions.** Letter keys (no `⌘`) only fire when the **viewer** or
   the **Directory Panel** has focus, never when a text field or
   search field is active.
2. **Arrows do the same thing regardless of which panel has focus**,
   as long as that panel is one of {Directory Panel, Viewer}. Tree
   traversal and viewer advancement are the same action — the resolved
   file list is one model, navigated through one set of arrow keys.
3. **Modifiers add scrolling, not new actions.** `⌃` plus an arrow
   pans the visible image; the bare arrow advances the selection.
4. **Every binding here is also reachable through a menu or the
   command palette.** Discoverability is preserved — a user who has
   not memorized `W` can still reach Zoom to Width through View ▸
   Zoom to Width.
5. **No conflict with system-reserved shortcuts.** Bare letters never
   reach the system; `⌃`-arrow combinations are checked against macOS
   Spaces and Mission Control bindings and resolved in favor of the
   image viewer only while a viewer/panel has focus.

---

## 3. Focus Contexts

Hotkey routing depends on which UI surface has the keyboard focus.
Three contexts matter:

| Context | What "focus" means | Which hotkeys in this doc apply |
| --- | --- | --- |
| **Viewer focus** | The main image canvas (`MTKView`) is the first responder. This is the default after launching the app or clicking the image. | All of them. |
| **Directory Panel focus** | The Directory Panel tree (`OutlineGroup` / `NSOutlineView`) is the first responder. This is the state after the user clicks a row or presses `⌘1`. | All arrow-key bindings (§4). The zoom-letter bindings (§5) still fire because they target the linked viewer; the panel does not eat them. |
| **Text-field focus** | A search field, rename field, or settings text field is first responder. | **None** of the bindings in this doc. Letters type into the field; arrows move the text cursor. |

The first-responder check is done in a single dispatcher that wraps
the AppKit key-equivalent path; see §8 for the implementation note.

## 4. Arrow-Key Navigation

The four arrow keys drive a unified model: **navigate the resolved
file list**. That list is rendered both as the Directory Panel tree
(`dir_ui.mdx`) and as the viewer's current selection — moving in one
moves the other.

The semantics are designed to match the **Finder column view** mental
model: up/down moves the cursor within the current level; left/right
moves between levels (collapse / expand a folder, or step out / in).

### 4.1 Bare Arrow Keys

| Key | Selection state | Behavior |
| --- | --- | --- |
| **↑** | Any | Move the selection **up one row** in the resolved file list. The tree is traversed in display order: a collapsed folder is a single row; an expanded folder's children are walked depth-first. If the selection is already on the first row, the key is a no-op (audible beep is suppressed). |
| **↓** | Any | Move the selection **down one row**, same traversal order. Wrapping is off by default — at the last row, the key is a no-op. Settings ▸ Navigation ▸ *Wrap at ends* can enable wrap. |
| **←** | Selection is a **folder**, expanded | **Collapse** the folder. The folder row stays selected. |
| **←** | Selection is a **folder**, collapsed | Move selection to the **parent folder**. |
| **←** | Selection is a **file** | Move selection to the **parent folder** of that file. The folder row becomes selected; the viewer keeps showing the file the user was on until a new file is chosen. |
| **→** | Selection is a **folder**, collapsed | **Expand** the folder. The folder row stays selected. |
| **→** | Selection is a **folder**, expanded | Move selection to the **first child** of the folder. If the first child is a file, the viewer loads that file. If the first child is a sub-folder, that sub-folder becomes selected and remains collapsed. |
| **→** | Selection is a **file** | No-op. Files have no children. (Reserved for future use — e.g., entering a multi-frame image's frame list.) |

### 4.2 Behavioral Notes

* **Up/down always advances the viewer.** Whether focus is on the
  Directory Panel or the viewer, pressing `↓` makes the next image
  load. This is the single most-pressed key in the app and must feel
  instantaneous (see §10 for the performance budget).
* **Left/right only changes the viewer when crossing into a new
  file.** Collapsing or expanding a folder, or moving to a folder row,
  does **not** change which image is displayed. The viewer holds the
  last-viewed file until the user picks a new one.
* **Folder selection without a file shown.** When the user lands on a
  folder row that does not auto-load any file, the viewer shows a
  small folder-summary placeholder ("12 images, 3 sub-folders"). The
  status bar gives the same count.
* **Recursive vs. shallow folders.** The traversal in §4.1 walks only
  the **resolved** file list — i.e., the list Local Storage has
  already filtered through inclusion / exclusion criteria. A folder
  marked shallow (see `local_storage.mdx`) exposes only its own files;
  pressing `→` on it never enters its sub-folders.
* **Multi-select** is **deferred to v2** — see §11. v1 keeps the
  single-cursor model so `↑` and `↓` always advance the viewer, with
  no Shift-extends-selection semantics.

### 4.3 Arrow Keys With `⌃` (Page-Pan)

`⌃` plus an arrow key **pans the visible image** instead of advancing
selection. This is the keyboard equivalent of scrolling a long image
that does not fit in the viewer — most useful in **Zoom to Width**
mode (§6) on tall UI mockups, screenshots, and infographics.

| Key | Behavior |
| --- | --- |
| **⌃ ↓** | Scroll the visible viewport **down by 15%** of the viewport height. |
| **⌃ ↑** | Scroll the visible viewport **up by 15%** of the viewport height. |
| **⌃ →** | Scroll the visible viewport **right by 15%** of the viewport width. |
| **⌃ ←** | Scroll the visible viewport **left by 15%** of the viewport width. |

* Repeated taps stack: ten taps of `⌃ ↓` scrolls ~150% of a viewport,
  i.e., reads about one and a half pages.
* Holding the arrow key (key-repeat) continues to scroll smoothly at
  the system key-repeat rate.
* When the image fully fits the viewport (no overflow on the scrolled
  axis), `⌃`-arrow is a no-op on that axis.
* `⌃ ↓` and `⌃ ↑` never advance to the next/previous image — they
  scroll within the current image only. The bare arrow keys are the
  way to move between files.

### 4.4 Space Bar in the Directory Panel — Toggle Folder Expand/Collapse

When the **Directory Panel has focus** and the cursor row
(`treeNav.activeRow`) is a **folder**, pressing **Space** toggles the
folder's expand/collapse state. This is the keyboard twin of clicking
the disclosure triangle: same `treeNav.setExpanded(...)` call, same
persistence into `session.directory_panel.expanded_paths` per
[`dir_ui.mdx` §3.3.1](./dir_ui.mdx#331-persistence-across-launches).

The binding is **context-sensitive** — Space does not always mean
"toggle folder." Resolution is by focus and cursor row:

| Focus | Cursor row (`treeNav.activeRow`) | Space behavior |
| --- | --- | --- |
| **Directory Panel** | **Folder** | **Toggle expand/collapse** on that folder. The cursor stays on the same row. |
| **Directory Panel** | **File** (or no cursor) | Toggle the slideshow on the active scope (same as upstream and `dir_ui.mdx` §5.2). |
| **Viewer** | — | Focus-context handler in `ImageViewer.handleSpace()` (`videos.mdx` §11.2 / `svg.mdx` §10.2 / `slideshow.mdx` §10.2): play/pause for video, animation toggle for animated SVG, slideshow toggle for static images. |
| **Text field** | — | Space types a literal space character into the field. No binding fires. |

Behavioral notes:

* **Same call path as ← / →.** A folder-row `Space` is equivalent to
  pressing `→` on a collapsed folder, or `←` on an expanded folder.
  The arrow keys keep the §4.1 dual semantics (expand vs. step into,
  collapse vs. step out); `Space` is the single-key "just toggle"
  alternative that does not move the cursor under any circumstance.
* **Cursor never moves.** Whether expanding or collapsing, the cursor
  row remains on the toggled folder. `↑` / `↓` are still the way to
  walk into or out of the folder after toggling.
* **Viewer is not touched.** Toggling a folder does not load any new
  image; the viewer holds its last-viewed file, matching §4.2.
* **Tree view only.** This binding only fires when the panel is in
  tree view (`panelViewMode == .tree`). In list view, folders are not
  cursor-addressable, so `treeNav.activeRow` will be a file path and
  Space falls through to the slideshow toggle.
* **No-op when there is no row.** If the panel has focus but the user
  has not yet placed the cursor (no `activeRow`, no `selectedFile`),
  Space is a no-op — no folder to toggle and no scope file to begin a
  slideshow on.
* **Modifiers fall through.** `⌘ Space`, `⌥ Space`, `⌃ Space` are
  **not** captured here. Spotlight and Input Source switching keep
  their system bindings; menu chords that happen to use Space are
  unaffected.

This makes the file-tree feel like Finder's column / list view: a
disclosure triangle, an arrow key, and `Space` are three equivalent
ways to open or close a folder, and the user picks whichever matches
their hand position.

## 5. Zoom Hotkeys (Bare Letters)

The bare-letter zoom keys are the muscle-memory surface for the
viewer. They are deliberately short — single keys — because they are
the most-used viewer commands after arrow navigation.

| Key | Action | Notes |
| --- | --- | --- |
| **`+`** / **`=`** | **Zoom In by 20%.** | Multiplicative — each press multiplies the current zoom factor by `1.20`. `=` is the unshifted twin of `+` so the user does not need `⇧`. |
| **`-`** | **Zoom Out by 20%.** | Each press divides the current zoom factor by `1.20`. |
| **`C`** | **Center.** | Re-centers the image inside the viewport without changing the current zoom factor. Useful after `⌃`-arrow panning or after dragging the image off-center with the mouse. |
| **`M`** | **Normalize.** | Resets to the user's *default zoom mode*. The default is **Zoom to Fit** unless the user has changed it in Settings ▸ Viewer ▸ *Default zoom on open*; setting that to *Actual size (100%)* makes `M` snap to 100% instead. `M` is a one-press way to undo any ad-hoc zooming. *(Was bound to `N` prior to the slideshow-cursor work; `N` now navigates — see §6A.)* |
| **`N`** | **Next Image.** | Advance the global selection cursor to the next image in the file tree's top-down visible order (see [slideshow.mdx §10A](./use_cases/slideshow.mdx)). Wraps at the end when *Settings ▸ Viewer ▸ Wrap at ends* is on. Skips files whose include state is *exclude* or *inherit-exclude*. Works inside and outside a running slideshow — pressing `N` during a slideshow re-anchors the controller. |
| **`P`** | **Previous Image.** | Step the global selection cursor back to the previous image, same list and same skip rule as `N`. *(The earlier `P`-for-Copy-Path binding moved to `⇧P`.)* |
| **`Z`** | **Zoom to Fit.** | Explicit fit-to-screen, regardless of the default zoom mode. The image scales so the longer dimension exactly fits the viewport; the other dimension may have letterboxing. |
| **`W`** | **Zoom to Width.** | Switches to the new Zoom to Width mode (§6). The image is rescaled so its width exactly matches the viewport width, and the viewport scrolls to the top of the image. |
| **`⇧P`** | **Copy Path.** | Copies the **absolute POSIX path** of the currently displayed file (or the file selected in the Directory Panel when no image is loaded) to the system pasteboard as plain UTF-8. In-viewer twin of File ▸ Copy File Path (`⌃⌘C` in `menus.mdx`). See §5A for the full pasteboard contract. |
| **`L`** | **Load From Clipboard.** | Reads a file path or `file://` URL from the system pasteboard and loads it in the viewer. The path is **not** added to any scope — it is shown as an ad-hoc open. See §5B for the parse and error contract. |
| **`F`** | (Reserved) | Reserved for *Zoom to Fill* (the cover-style mode) in v2. v1 does not bind `F`. |

### 5.1 Mode vs. One-Shot

`Z`, `W`, and `M` set a **zoom mode** that persists until the user
zooms manually or switches mode. `+`, `-`, and `C` are one-shot
adjustments that put the viewer into a **custom-zoom** state. The
status bar shows the current mode (*Fit*, *Width*, *Actual*, *Custom*)
and the current percent.

### 5.2 Interaction With the Mouse

* Mouse-wheel zoom (when bound to zoom in Settings ▸ Viewer ▸ *Mouse
  wheel*) uses the same 20% multiplicative step.
* Magnify gesture (`MagnifyGesture` on the trackpad) is **smooth** —
  it does not snap to 20%. The status bar shows the resulting
  arbitrary percent.
* Pressing any bare-letter zoom key after smooth-magnifying overrides
  the gesture's zoom factor and switches back to the named mode.

## 5A. Clipboard Path Hotkeys (`P` / `L`)

`P` (Copy Path) and `L` (Load From Clipboard) are a paired set: `P`
puts a path **out** to the system clipboard so the user (or another
app) can act on it; `L` reads a path **in** from the system clipboard
and opens it in the viewer. Together they give the viewer a one-key
"send" and one-key "receive" surface for image paths, which is the
most-asked-for interop with Finder, Terminal, Claude Code, browser
downloads, and chat windows.

Both keys obey the focus rules in §3 (viewer or Directory Panel; not
text fields). Both are bare letters with no modifier.

### 5A.1 `P` — Copy Path

* **What gets copied.** The **absolute POSIX path** of the file that
  drives the viewer:
  * If an image is currently loaded, the path of that image.
  * Else, if the Directory Panel has a file selected, the path of
    that file.
  * Else (no image, no file selection), the key is a no-op and the
    status bar shows "No file to copy."
* **Pasteboard format.** Plain UTF-8 string on
  `NSPasteboard.PasteboardType.string`. Additionally,
  `NSPasteboard.PasteboardType.fileURL` is set to the file's
  `file://` URL, so paste targets that prefer the URL type (Finder,
  most Mac apps) accept the same paste. No HTML, no RTF, no other
  representations.
* **Symlinks and aliases.** The path is **not** resolved through
  symlinks or aliases. If the user is viewing the file via a symlink,
  the symlink path is what gets copied. (Reveal in Finder is the
  command that resolves; `P` is the command that preserves what the
  user sees.)
* **Sandbox-relative paths.** Are expanded to absolute paths before
  copy. Nothing relative ever lands on the clipboard.
* **Confirmation.** A 1.2-second status-bar toast confirms the copy:
  "Copied path: …/filename.jpg".

### 5A.2 `L` — Load From Clipboard

* **What gets read.** The system pasteboard is read in this order:
  1. `NSPasteboard.PasteboardType.fileURL` — preferred. The first
     `file://` URL is used.
  2. `NSPasteboard.PasteboardType.string` — if the string is a single
     line that parses as a POSIX path (absolute or `~`-prefixed) or
     as a `file://` URL, use it.
  3. If neither yields a usable path, `L` is a no-op and the status
     bar shows "Clipboard does not contain a file path."
* **What gets loaded.** The viewer attempts to display the file at
  that path. The same decoder path used for File ▸ Open is used here
  — ImageIO first, vendored decoders as a fallback per
  `automation.mdx`. Format support and error reporting are identical
  to drag-from-Finder.
* **Scope behavior.** The loaded file is **not** added to any scope
  and does **not** mutate `directories.yaml`. The Directory Panel
  selection is **cleared** for the duration of the ad-hoc view: the
  panel shows a one-row "(ad-hoc) /Users/…/filename.jpg" entry
  italicized in the active scope header area, and `↑` / `↓` are
  no-ops until the user picks a scope file again.
* **Path normalization.** `~` is expanded to the user's home
  directory. `file://` URLs are decoded (percent-decode + drop the
  scheme). Trailing whitespace and surrounding quotes (`"…"`, `'…'`)
  are stripped — these survive copy-paste from chat windows.
* **Errors.**
  * File does not exist → status-bar error "File not found: …" and a
    300 ms shake on the viewer canvas.
  * File exists but ImageIO cannot decode it → standard error card
    used by File ▸ Open.
  * Path resolves to a **directory** → the viewer does not open the
    directory as an image; instead it asks Local Storage to add the
    directory to the active scope (same gesture as drag-folder-onto-
    panel from `dir_ui.mdx`), and the status bar shows "Added folder
    to active scope."
* **Security.** No remote URLs are followed. `http://`, `https://`,
  and other non-`file` schemes are rejected with "Clipboard does not
  contain a file path." This is deliberate — `L` is for local files
  the user already trusts, not for downloading from the web.

### 5A.3 Round-trip Workflow

The canonical `P` → `L` round trip:

1. User is on image `~/Pictures/2026/03/IMG_1234.heic` in the
   viewer. Presses `P`. The path is on the clipboard.
2. User switches to Claude Code, pastes the path into a prompt:
   "rotate this image 90° and save the result next to it".
3. Later, the user pastes the result path (which Claude Code echoed
   back) into Finder, into Terminal, or back into ImageGlass_Mac.
   Inside ImageGlass_Mac, the user presses `L`, and the rotated
   version loads.

This loop — viewer → clipboard → external tool → clipboard →
viewer — is the primary reason `P` and `L` are single-key bindings.
Any extra modifier on this loop would defeat the muscle-memory
intent.

## 6. Zoom to Width Mode (New)

**Zoom to Width** is a new fork-specific zoom mode tailored to **tall
images that the user wants to read top-to-bottom**: UI page mockups,
long screenshots, web-page captures, infographics, vertical comics,
and product-design exports.

### 6.1 Behavior

1. The image is rescaled so its **width equals the viewport width**
   (minus any insets defined by the toolbar and panels).
2. The viewport's vertical scroll position is set to the **top** of
   the image (offset `0`).
3. The viewport is allowed to scroll **vertically** as the user pans
   downward — this is the natural reading motion. If the rescaled
   image is also wider than expected (e.g., the user has resized the
   window narrower than the image's natural aspect), horizontal
   scrolling is also permitted.
4. The zoom mode persists across image changes — pressing `↓` to load
   the next image keeps Zoom to Width active, rescales the new image
   to viewport width, and scrolls to the top.

### 6.2 Why Top of Image

Most tall mockups have a header / hero at the top. Starting the view
there matches the user's intent to read the image in document order.
Zoom to Fit's "centered" default is wrong for this workflow because
it puts the middle of a long mockup in the middle of the screen,
forcing the user to scroll up before they can read.

### 6.3 Pairing With `⌃`-Arrow

Zoom to Width is designed to be used with **`⌃ ↓`** and **`⌃ ↑`**:

* `W` → enter Zoom to Width; viewport at top.
* `⌃ ↓` repeatedly → scroll down by 15% of viewport per press; read
  the mockup section by section.
* Hold `⌃ ↓` → smooth continuous scroll for a long page.
* `↓` (bare) → advance to the **next image**, which re-enters Zoom to
  Width at the top of that next image.

This sequence — `W`, then `⌃ ↓ ⌃ ↓ ⌃ ↓ …`, then `↓` — is the
canonical "read a folder of tall mockups" flow.

### 6.4 Menu Surface

Zoom to Width appears in the **View** menu (`menus.mdx` §4.4)
alongside the existing zoom items. The full View ▸ Zoom group is, in
v1 with Zoom to Width added:

| Item | Shortcut | Notes |
| --- | --- | --- |
| Actual Size | ⌘ 0 | 100% zoom. Same as upstream. |
| Zoom In | ⌘ + | One step of 20% zoom-in. Bare `+` / `=` is the in-viewer twin. |
| Zoom Out | ⌘ - | One step of 20% zoom-out. Bare `-` is the in-viewer twin. |
| Zoom to Fit | ⌘ 9 | Fits the longer dimension. Bare `Z` is the in-viewer twin. |
| Zoom to Fill | ⇧ ⌘ 9 | Covers the viewport (crops on the shorter dimension). |
| **Zoom to Width** **(new)** | **`W`** *(bare)* / ⌥ ⌘ 9 *(menu twin)* | Fit image width to viewport width; scroll to top. |
| Lock Zoom | ⌃ ⌘ L | Toggle — keep the current zoom across image changes. |

The bare `W` is the canonical shortcut shown on the menu item; the
`⌥ ⌘ 9` chord is the menu-friendly twin so the binding works even
when a text field happens to be focused.

### 6.5 State Persistence

The user's last-used zoom mode (Fit, Fill, Width, Actual, Custom) is
remembered in `UserDefaults` under `viewer.lastZoomMode`. On relaunch,
if Settings ▸ Viewer ▸ *Default zoom on open* is set to *Restore last
mode*, the app reopens in the same zoom mode the user was last in,
including Zoom to Width.

## 8. Implementation Notes

### 8.1 Routing

All bindings in this document are routed through a single SwiftUI
`ViewModifier` — `ImageGlassHotkeysModifier` — applied to both the
viewer canvas and the Directory Panel via the
`.imageGlassHotkeys(state:viewer:)` extension. SwiftUI's
`.onKeyPress` inside the modifier guarantees that:

* Bare letters (`C`, `N`, `Z`, `W`, `+`, `=`, `-`) reach the handler
  only when the modifier's host is the focused surface — i.e., the
  viewer or the Directory Panel. A focused text field swallows the
  keystroke before it ever arrives.
* Arrow keys with no modifier route to `AppState.arrow*()`, which
  walks the visible-tree row model shared by both surfaces, so the
  viewer and panel always show the same cursor.
* `⌃`-arrow combinations route to the viewer pan handler, which is a
  no-op on an axis where the image already fits the viewport (the
  canvas computes overflow from its own bounds; see §4.3).

### 8.2 SwiftUI + AppKit Split

* The viewer canvas is `NSView`-backed (AppKit, `ImageCanvasView`).
  Mouse-wheel zoom, pinch zoom, drag-pan, and the color-picker
  readouts live on `NSResponder`.
* The Directory Panel is a SwiftUI `LazyVStack` rendering recursive
  `DesignTreeNode` rows, with expansion + cursor state held in a
  shared `TreeNavigator` (`@Observable`).
* Both surfaces attach the same `ImageGlassHotkeysModifier`
  (`.imageGlassHotkeys(state:viewer:)`). Each `.onKeyPress(...)`
  inside the modifier translates the arrow / letter key into a call
  on `AppState.arrow*()` or `ViewerState.zoom*()`, so the viewer and
  panel always agree about which file is selected and which folders
  are open.
* Focus routing relies on SwiftUI's first-responder model. When a
  text field (search field, rename field) is focused, the modifier's
  `.onKeyPress` blocks never receive the keystroke — the text field
  consumes it for typing, satisfying spec §3's text-field rule.

### 8.3 Conflict With System and Menu Bindings

`W` is unusually short and could collide with menu items that take
`⌘ W` (Close Window). The dispatcher distinguishes:

* `⌘ W` → Close Window (handled by AppKit menu validation, owned by
  `menus.mdx`).
* Bare `W` (no modifier) → Zoom to Width.
* `⇧ W` → No binding. Falls through to text input if a field has
  focus.

This pattern (bare-letter wins, `⌘`-modified routes to the menu)
applies uniformly to `C`, `N`, `Z`, `W`, `+`, `-`.

The same pattern covers the two clipboard-path letters added in §5A:

* `⌘ P` → Print, and `⇧ ⌘ P` → Page Setup (owned by `menus.mdx`
  §4.2).
* Bare `P` (no modifier) → Copy Path.
* `⌘ L` and any other modified-`L` chord → routed by the menu bar
  per `menus.mdx`. Bare `L` (no modifier) never reaches the menu;
  it is dispatched to Load From Clipboard.
* `⇧ P`, `⇧ L` → No binding. Fall through to text input if a field
  has focus.

### 8.4 Performance Budget

* `↓` → next image must produce the **first frame** of the next
  image within **80 ms** on Apple Silicon for images up to 50 MP,
  using the prefetched-thumbnail-first then full-decode strategy
  defined in `list_of_files.mdx`. Anything slower than that feels
  laggy in the muscle-memory loop.
* `W` → entering Zoom to Width is a `CATransaction`-grouped layout
  pass; it must complete in one frame (≤ 16 ms at 60 Hz).
* `⌃ ↓` panning at 60 Hz key-repeat is delivered at 15% of viewport
  per tick; the `CAScrollLayer` content offset animation is disabled
  (instantaneous) so the pan tracks the keypress rate without
  damping.

## 9. Settings Hooks

The following Settings ▸ Viewer / Settings ▸ Navigation fields
influence the behavior in this document. None of them rebind keys in
v1 (rebinding is a v2 feature, see `menus.mdx` §9).

| `settings.json` key | UI field | Effect |
| --- | --- | --- |
| `viewer.default_zoom_on_open` | Viewer ▸ *Default zoom on open* | Drives `N` (Normalize): `fit` (default), `actual`, or `restore_last`. |
| `viewer.zoom_step_percent` | Viewer ▸ *Zoom step (percent)* | Multiplicative step for `+` / `-` / ⌘+ / ⌘-. Default `20`. |
| `viewer.pan_step_percent` | Viewer ▸ *Pan step (percent of viewport)* | Per-keypress pan for `⌃`-arrow. Default `15`. |
| `viewer.zoom_to_width_scroll` | Viewer ▸ *Zoom to Width: scroll target* | `top` (default — v1 only honors this), `center`, or `last_viewed` (reserved). |
| `viewer.wrap_at_ends` | Viewer ▸ *Wrap at ends* | `↓` on the last row wraps to the first when `true`. Default `false`. |
| `ig.viewer.last_zoom_mode` *(UserDefaults)* | — | Persisted last mode (`fit` / `width` / `lock` / `actual` / `auto` / `fill` / `height`). Read by `restore_last`. |

All `viewer.*` fields persist in `settings.json` and are exposed to
MCP via the standard settings tool surface (see `settings.mdx`).
The last-mode breadcrumb is per-machine state, so it lives in
`UserDefaults`, not `settings.json`.

A v2 setting — *Navigation ▸ Arrow keys move focus to viewer after
selection* — is not in v1 because SwiftUI's responder chain already
lands the next keystroke in whichever surface most recently sent
selection through `state.selectedFile`.

## 10. Acceptance Criteria for v1

The hotkey surface is considered v1-complete when:

1. Every binding in §7 fires the listed action in the listed focus
   context and is a no-op outside that context (verified by an
   XCUITest matrix).
2. `↓` on a folder of 1,000 mid-size JPEGs advances the viewer in
   under 80 ms median, measured on an M-class Mac.
3. `W` rescales any image taller than the viewport so its width fits
   the viewport width and scrolls to offset 0 in a single frame.
4. `⌃ ↓` scrolls the viewport down by exactly 15% of viewport height
   per keypress and a hold scrolls smoothly at the system key-repeat
   rate.
5. View ▸ Zoom to Width appears in the menu with the `⌥ ⌘ 9` chord,
   and the menu item's enabled state is `true` whenever an image is
   loaded.
6. Help ▸ Keyboard Shortcuts… lists every binding in §7 alongside
   every `⌘`-chord from `menus.mdx`, pulled from the same source of
   truth that wires the dispatcher.

## 11. Non-Goals (v1)

* **Rebindable hotkeys.** The bindings in this document are fixed in
  v1. Settings ▸ Keybindings is the v2 surface that will let the user
  swap `Z` and `W`, change the zoom step, or assign hotkeys to MCP
  tools.
* **Multi-select with `⇧`.** Holding Shift on `↑`/`↓` does **not**
  extend a multi-row selection in v1. Mac batch-edit workflows are
  out of scope until v2, when the Directory Panel grows a real
  selection-set model. v1 keeps a single cursor row.
* **Vim-style chords.** No `gg` / `G` / `:` style multi-key chords.
  Mac users do not expect them in an image viewer.
* **Per-scope hotkeys.** Hotkeys are global to the app; they do not
  change with the active scope.
* **Gamepad / remote-control bindings.** Out of scope until there is
  a credible Mac Catalyst path.
