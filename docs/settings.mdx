# Settings

This page is the **complete product + engineering specification** for the
Settings subsystem of the ImageGlass Starbuck fork (ImageGlass_Mac). It is
the contract that the SwiftUI Settings scene, the on-disk configuration
files, the migration story from upstream `igconfig.json`, and the
MCP-driven settings tools all must satisfy.

It cross-references the sibling specs and does not duplicate them:

* [Panels and Window Layout](./panels.mdx) — the movable / dockable / floating
  panel framework. Settings related to layout, panel visibility, and panel
  defaults are owned by the **Layout** page below but the panel semantics
  themselves live in `panels.mdx`.
* [Crop](./crop.mdx) — crop tool semantics. Crop *defaults* live in the
  **Tools → Crop** page below; the editing UX lives in `crop.mdx`.
* [File List Panel](./list_of_files.mdx) — the file panel and scope engine.
  Scope JSON files are not "settings" in the strict sense; the **Gallery**
  page below covers gallery-style file panel defaults, and scope persistence
  is fully spec'd in `list_of_files.mdx`.
* [Automation](./automation.mdx) — overall fork charter. Read this
  first if you have not already.

The Mac stack used throughout this spec is the one declared in
`CLAUDE.md`: **Swift 6 + SwiftUI + AppKit interop + ImageIO + Core Image +
Metal**, persisted via `Codable` JSON and `UserDefaults`, fronted by an
in-process MCP server.

## 1. Audience and Goals

### 1.1 Who uses Settings

The Settings surface serves four audience tiers. Every page below is
annotated with the tier it primarily targets.

| Tier | Audience | What they expect from Settings |
| --- | --- | --- |
| T1 | **Casual viewer** | Sensible defaults; rarely opens Settings; if they do, they want General + Appearance only. |
| T2 | **Photographer / designer** | Color management, EXIF auto-rotate, embedded thumbnails, default editor app integration, lossless JPEG settings. |
| T3 | **Power user** | Custom keybindings, mouse actions, toolbar editing, slideshow scripting, advanced cache tuning, plugin management. |
| T4 | **Claude Code / MCP driver** | Read and modify the same settings the GUI uses, atomically, without launching the app. |

### 1.2 Goals

* **Discoverability.** Every persisted setting is reachable from the GUI;
  there is no "hidden" power-user-only setting that can only be edited by
  hand. The reverse is also true — every GUI setting is one named field
  in the on-disk JSON.
* **Round-tripability.** Edits made by the GUI, by hand, or by MCP are all
  equivalent. Settings written by MCP survive a GUI relaunch unchanged.
* **Native feel.** No WebView. No HTML form controls. Every page is a
  SwiftUI `Form` with native Mac controls (`Toggle`, `Picker`,
  `ColorPicker`, `Slider`, `Stepper`, `TextField`, `Menu`, `KeyboardShortcuts`-style
  capture fields). Settings respects the user's system appearance, accent
  color, and accessibility preferences.
* **Atomic durability.** Settings writes are crash-safe: atomic rename of a
  temporary file. A power outage cannot leave the user with a corrupt
  config.
* **Migration from upstream.** A user coming from Windows ImageGlass with an
  `igconfig.json` can drop it next to the Mac config and have their
  preferences honored.

### 1.3 Non-goals

* Settings is not a backup/restore product. iCloud sync, profile sync across
  machines, and team-shared configuration are explicitly out of scope for
  v1.
* Settings is not a scripting environment. The MCP surface is the
  sanctioned automation channel; we do not embed a JS engine or expose a
  Lua sandbox inside Settings.
* Settings is not where scopes are edited in detail. Scopes have their own
  panel and JSON model — see `list_of_files.mdx`.

## 2. Settings Architecture

### 2.1 The split: UserDefaults vs. settings.json

Two on-disk surfaces hold preferences. The split is deliberate:

| Surface | Backing store | Contents |
| --- | --- | --- |
| **UserDefaults** (`com.starbuck.imageglass`) | `~/Library/Preferences/com.starbuck.imageglass.plist` | Trivial UI prefs that the **app** owns: last window position and size, last selected sidebar tab in Settings, last sort order in the file panel, whether the welcome image is dismissed, the active layout preset name. Anything you would not want an MCP client to touch and anything that is intrinsically per-user-per-machine. |
| **settings.json** | `~/Library/Application Support/ImageGlass_Mac/settings.json` | The canonical config: viewer behavior, color management, keybindings, mouse actions, slideshow settings, gallery defaults, tools, plugins. This file is the contract for both the GUI and the MCP server. |

Justification:

* **UserDefaults is opaque and per-OS-user.** It is correct for trivial
  per-install state that should not survive a config export. Putting these
  in `settings.json` would pollute MCP diffs every time a window moved.
* **`settings.json` is plain text and inspectable.** It is the same
  philosophy as the Local Storage scope files (see `automation.mdx` §4):
  a binary blob would close off the MCP, hand-editing, and
  version-control workflows the fork explicitly wants to enable.

### 2.2 File layout

```
~/Library/Application Support/ImageGlass_Mac/
├── settings.json            # canonical preferences (this spec)
├── settings.json.bak        # last-known-good (rotated on successful load)
├── layout.json              # panel layout state (see panels.mdx)
├── keybindings.json         # OPTIONAL split-out of keyboard config; if absent, keybindings live inside settings.json
├── scopes/
│   ├── default.json         # see list_of_files.mdx
│   └── ...
├── themes/                  # user-installed theme packs
├── plugins/                 # user-installed plugin bundles
└── logs/
    └── settings.log         # append-only log of GUI/MCP edits with actor + timestamp
```

The directory is created on first launch with mode `0700`. If the directory
is missing the app creates it; if `settings.json` is missing the app writes
the defaults out and shows the Quick Setup flow (see §3.1).

### 2.3 The Codable schema

The on-disk schema is a single root struct `Settings` with one nested
struct per Settings page, plus a `version` field for migration. The schema
is the canonical definition of "what is a setting"; the SwiftUI pages bind
to fields of these structs.

```swift
struct Settings: Codable, Sendable {
    var version: Int = 1                  // migration sentinel
    var general: GeneralSettings = .init()
    var image: ImageSettings = .init()
    var viewer: ViewerSettings = .init()
    var appearance: AppearanceSettings = .init()
    var layout: LayoutSettings = .init()
    var slideshow: SlideshowSettings = .init()
    var edit: EditSettings = .init()
    var mouse: MouseSettings = .init()
    var keyboard: KeyboardSettings = .init()
    var gallery: GallerySettings = .init()
    var toolbar: ToolbarSettings = .init()
    var fileAssoc: FileAssocSettings = .init()
    var language: LanguageSettings = .init()
    var tools: ToolsSettings = .init()
    var plugins: PluginsSettings = .init()
    var advanced: AdvancedSettings = .init()
}
```

Every field below in the page-by-page tables maps to a leaf property of one
of these structs. The JSON key is the field's snake_case name. The whole
file is encoded with `JSONEncoder.OutputFormatting = [.prettyPrinted,
.sortedKeys]` so MCP-driven edits produce stable diffs.

### 2.4 Persistence pipeline

Reads:

1. App launch — `SettingsStore` (an `actor`) loads `settings.json` on a
   background `Task`.
2. If the file is missing or unreadable, defaults are written and the
   first-run Quick Setup is shown.
3. If `version` is behind the current schema version, `SettingsMigrator`
   runs in-place and rewrites the file before the GUI sees it.
4. Loaded settings are published via an `@Observable` `SettingsViewModel`
   that the SwiftUI Settings scene binds to.

Writes:

1. Every binding mutation goes through `SettingsStore.update(_:)` which
   debounces (200 ms) and serializes writes to a temp file, then atomic-
   renames to `settings.json`.
2. After a successful write, the previous `settings.json` becomes
   `settings.json.bak`.
3. An `FSEventStream` watches `settings.json`. If the file changes from
   outside the running app (i.e. MCP wrote it), the store reloads and the
   GUI updates live. Bindings that the user is currently editing are
   detected via a "dirty" flag and not overwritten until the user commits
   or backs out.

### 2.5 MCP surface

The MCP server exposes settings to external clients. Every setting that
appears in the page tables below has an `mcp` column: **R** (readable), **W**
(writable), or **—** (not exposed). Settings marked **—** are either
session-only (window positions), security-sensitive (file-association
ownership), or platform-managed (`Info.plist` UTIs) and cannot be
meaningfully changed at runtime.

MCP tools:

| Tool | What it does |
| --- | --- |
| `get_settings(section?)` | Return the entire settings tree or one section. |
| `get_setting(path)` | Return one value by dotted path, e.g. `viewer.zoom_mode`. |
| `set_setting(path, value)` | Set one value; validated against the schema. |
| `set_settings(patch)` | Apply a partial-tree patch in one atomic write. |
| `reset_setting(path)` | Restore one value to its default. |
| `reset_section(section)` | Restore one section to defaults. |
| `export_settings(out_path)` | Copy `settings.json` to a path the caller chose. |
| `import_settings(in_path, dry_run?)` | Apply a settings file from disk; `dry_run` returns the diff without writing. |
| `list_setting_paths()` | Return every dotted path with its type and current value. Useful for discovery. |

All `set_*` calls return the prior value so callers can implement undo
without a round-trip. All writes are journaled to `logs/settings.log` with
the actor (`gui`, `mcp:<client-name>`, `import`).

### 2.6 Migration from upstream `igconfig.json`

The upstream Windows app persists its settings to `igconfig.json` next to
`ImageGlass.exe` (or, for portable builds, in the same folder). The Mac
fork supports a one-shot import:

1. On first launch, the app looks for `~/Documents/igconfig.json`,
   `~/Downloads/igconfig.json`, and `~/Library/Application
   Support/ImageGlass_Mac/import.json`.
2. If found, the user is prompted ("Import settings from previous ImageGlass
   install?") with a diff preview.
3. The `UpstreamConfigImporter` translates the upstream field names to the
   fork's schema. Every upstream property has either a direct fork
   equivalent (table below), a translated equivalent (e.g. Windows
   `BackdropStyle.Mica` → Mac `AppearanceSettings.WindowMaterial.sidebar`),
   or is dropped with a logged reason (e.g. `UseWebview2ForSvg` — the fork
   does not use WebView2).

The mapping table is exhaustive (Appendix A) so a power user can audit
every translated field.

### 2.7 Quick Setup (first run)

Upstream has a Quick Setup wizard that runs once on first launch. The Mac
fork preserves the pattern but rewrites it as a SwiftUI sheet with three
steps:

1. **Welcome and theme** — pick light, dark, or system. Pick an accent
   color override (or accept the system accent).
2. **What do you do with images?** — three preset bundles the user can pick
   from. Each bundle sets a coherent collection of defaults:
   * *Just view photos* (T1) — `viewer.zoom_mode = autoZoom`,
     `gallery.show = true`, `slideshow.interval = 5s`, default panel layout
     "Viewer".
   * *Photography workflow* (T2) — color management on, EXIF auto-rotate
     on, embedded thumbnails preferred for RAW, default panel layout
     "Photographer".
   * *Designer / power user* (T3) — frameless window, gallery hidden, MCP
     server enabled by default, default panel layout "Power user".
3. **Optionally import upstream settings** — if `igconfig.json` was
   detected, offer the diff and let the user accept or skip.

`general.quick_setup_version` records that Quick Setup ran, so a future
breaking-change release can re-trigger it.

### 4.1 General

#### Goals

Get the user's app-level preferences (language, theme, startup behavior,
update channel, window scoping, confirmations) right with the minimum
clicks. This is the only Settings page the typical T1 user will visit.

#### Audience

T1 primary; T2/T3 visit briefly during setup.

#### UI layout

Three sections in one scrolling `Form`:

* **Application** — language, theme, accent color override, check-for-updates
  cadence, show new version indicator.
* **Window** — multi-instance, always-on-top, open in full screen, open
  with last seen image, frameless window, window fit, center window fit.
* **Confirmations** — delete confirmation, overwrite confirmation, preserve
  modified date, open Save As in current image directory.

Footer row: `[Restore Defaults]` button (scoped to this page) and a `?`
help link.

#### Settings exposed

| Setting | JSON key | Type | Default | MCP | Notes |
| --- | --- | --- | --- | --- | --- |
| Language | `language.code` | `String` (BCP-47) | system | R/W | See §4.13. |
| Theme override | `general.theme_override` | `enum { system, light, dark }` | `system` | R/W | Drives `NSApp.appearance`; system mode follows `NSAppearance.current`. |
| Accent color | `general.accent_color` | `RGBA?` | `nil` (use system) | R/W | `nil` means inherit the user's macOS accent. |
| Open with last seen image | `general.open_last_image` | `Bool` | `true` | R/W | Maps to upstream `ShouldOpenLastSeenImage`. |
| Allow multiple instances | `general.multi_instance` | `Bool` | `true` | R/W | Maps to upstream `EnableMultiInstances`. When `false`, second launch focuses the running app and forwards the file argument. |
| Always on top | `general.window_top_most` | `Bool` | `false` | R/W | `NSWindow.level = .floating` when true. |
| Open in full screen | `general.start_full_screen` | `Bool` | `false` | R/W | Honored on launch only; runtime fullscreen is a viewer action. |
| Frameless window | `general.frameless` | `Bool` | `false` | R/W | Uses `NSWindow.StyleMask.borderless` + custom traffic-light overlay. |
| Window fit to image | `general.window_fit` | `Bool` | `false` | R/W | Resizes the window to the image's natural size on load. |
| Center window-fit window | `general.window_fit_centered` | `Bool` | `true` | R/W | Only meaningful when `window_fit = true`. |
| Confirm before delete | `general.confirm_delete` | `Bool` | `true` | R/W | Maps to upstream `ShowDeleteConfirmation`. |
| Confirm before overwrite | `general.confirm_overwrite` | `Bool` | `true` | R/W | Maps to upstream `ShowSaveOverrideConfirmation`. |
| Preserve modified date | `general.preserve_modified_date` | `Bool` | `false` | R/W | When saving in place, restore the file's pre-edit mtime. |
| Save As opens current dir | `general.save_as_in_current_dir` | `Bool` | `true` | R/W | NSSavePanel default directory. |
| Update check cadence | `general.update_cadence` | `enum { never, daily, weekly, monthly }` | `weekly` | R/W | Uses `NSBackgroundActivityScheduler`. |
| Show new version indicator | `general.show_update_badge` | `Bool` | `false` | R/W | Dock badge + menu bar dot when an update is ready. |
| Show welcome image | `general.show_welcome_image` | `Bool` | `true` | R/W | Empty-state image when no file is open. |
| Show app icon in title bar | `general.show_app_icon` | `Bool` | `true` | R/W | When `false`, hide the proxy icon. |
| Quick Setup version | `general.quick_setup_version` | `Float` | `0` | R | App-managed; bumped when Quick Setup is rerun. |
| In-app message duration | `general.toast_duration_ms` | `Int` | `2000` | R/W | Toast / in-app notification dwell time. |
| Last seen image | `general.last_seen_image` | `String?` | `nil` | R/W | Path; cleared when `open_last_image` is false. |

#### Backend wiring

* `theme_override` drives `NSApp.appearance` via a `themeDidChange`
  notification; the entire app re-resolves color tokens. The viewer canvas
  and the panel chrome both observe this.
* `update_cadence` registers an `NSBackgroundActivityScheduler` job with
  the matching repeating interval. The check itself is a `URLSession` GET
  to the project's Sparkle-compatible appcast endpoint.
* `multi_instance` is enforced by a `NSDistributedNotification` round-trip
  on launch: if a peer answers, the launcher forwards `argv[1]` and exits.
* `open_last_image` reads `last_seen_image` on launch; the path is
  resolved against a security-scoped bookmark to survive sandbox.

#### Workflow

User installs ImageGlass_Mac → Quick Setup picks the General defaults →
user opens an image. The user revisits General only to tweak update cadence
or toggle confirmations. Casual users may never open Settings again.

### 4.2 Image

#### Goals

Control how images are decoded and how the viewer presents them
(interpolation, color management, EXIF rotation, embedded thumbnails,
async loading, image-info overlay).

#### Audience

T2 primary; T3 for cache tuning.

#### UI layout

Four sections:

* **Color management** — color profile, apply to all formats.
* **Decoding** — async loading, embedded thumbnails for RAW, embedded
  thumbnails for other formats, embedded thumbnail minimum dimensions.
* **Interpolation** — scale-down algorithm, scale-up algorithm.
* **Image info overlay** — toggle the overlay, choose which EXIF/file tags
  to show.

#### Settings exposed

| Setting | JSON key | Type | Default | MCP | Notes |
| --- | --- | --- | --- | --- | --- |
| Color profile | `image.color_profile` | `enum { currentMonitor, sRGB, displayP3, adobeRGB, custom }` | `currentMonitor` | R/W | `custom` reveals a file picker for a `.icc` file. Maps to upstream `ColorProfile`. |
| Apply profile to all formats | `image.color_profile_all_formats` | `Bool` | `false` | R/W | Otherwise applied only to formats with an embedded profile (JPEG, PNG, HEIC, TIFF). Upstream `ShouldUseColorProfileForAll`. |
| Async loading | `image.async_loading` | `Bool` | `true` | R/W | Decode on a background actor; show a placeholder until ready. Upstream `EnableImageAsyncLoading`. |
| Embedded thumb for RAW | `image.embedded_thumb_raw` | `Bool` | `false` | R/W | Use the embedded JPEG when present. Upstream `UseEmbeddedThumbnailRawFormats`. |
| Embedded thumb for others | `image.embedded_thumb_other` | `Bool` | `false` | R/W | Upstream `UseEmbeddedThumbnailOtherFormats`. |
| Embedded thumb min width | `image.embedded_thumb_min_width` | `Int` | `0` | R/W | Below this width, fall through to full decode. |
| Embedded thumb min height | `image.embedded_thumb_min_height` | `Int` | `0` | R/W | Same, height axis. |
| Scale-down interpolation | `image.interp_scale_down` | `enum { nearest, linear, lanczos, cubic, mitchell, gaussian }` | `lanczos` | R/W | Mac default switched from `MultiSampleLinear` to `lanczos` because Core Image's Lanczos is GPU-accelerated and is the right default for downscaling. |
| Scale-up interpolation | `image.interp_scale_up` | `enum { nearest, linear, lanczos, cubic, mitchell, gaussian }` | `nearest` | R/W | Default preserves pixel-art crispness, matches upstream. |
| Image info overlay | `image.info_overlay` | `Bool` | `false` | R/W | When true, overlay shows the tags below. |
| Image info tags | `image.info_tags` | `[String]` | see Appendix B | R/W | Subset of `name`, `size`, `dimensions`, `dpi`, `bit_depth`, `color_space`, `exif_camera`, `exif_lens`, `exif_iso`, `exif_shutter`, `exif_aperture`, `exif_focal_length`, `exif_date_taken`, `gps_lat`, `gps_lon`. Upstream `ImageInfoTags`. |
| Single-frame formats | `image.single_frame_formats` | `[String]` | `[avif, heic, heif, psd, jxl]` | R/W | Formats for which the viewer never tries to animate. Upstream `SingleFrameFormats`. |
| Use checkerboard background | `image.checkerboard` | `Bool` | `false` | R/W | Transparency background for PNGs etc. Upstream `ShowCheckerboard`. |
| Checkerboard image only | `image.checkerboard_image_only` | `Bool` | `false` | R/W | When true, only the image region is checkered; canvas remains the background color. Upstream `ShowCheckerboardOnlyImageRegion`. |

#### Backend wiring

* `color_profile` drives the `kCGImageSourceShouldCacheImmediately` +
  ColorSync transform applied during `CGImageSourceCreateImageAtIndex`.
  `currentMonitor` queries `CGDisplayCopyColorSpace(CGMainDisplayID())` on
  every image load and on `NSWindowDidChangeScreen`.
* `interp_scale_down/up` selects the Core Image `CIFilter`
  (`CIBicubicScaleTransform`, `CILanczosScaleTransform`, etc.) used by
  the Metal-backed viewer. `nearest` is implemented directly in the Metal
  vertex shader for zero-cost pixel-art zoom.
* `embedded_thumb_*` toggles the
  `kCGImageSourceCreateThumbnailFromImageIfAbsent` /
  `kCGImageSourceCreateThumbnailFromImageAlways` flags on the
  `CGImageSourceCopyPropertiesAtIndex` call.
* `info_overlay` and `info_tags` drive a SwiftUI overlay layered on top of
  the Metal canvas, populated from `CGImageSourceCopyPropertiesAtIndex`
  metadata.
* `async_loading = false` is supported for reproducible-rendering use cases
  (e.g. running visual snapshot tests) but is not recommended.

#### Workflow

A photographer turns on **Apply profile to all formats**, sets the profile
to **Display P3**, enables **EXIF auto-rotate** in §4.3, and turns on
**Image info overlay** with the `exif_camera`, `exif_lens`, `exif_iso`,
`exif_shutter`, `exif_aperture` tags.

### 4.3 Viewer

#### Goals

How the user interacts with the displayed image: zoom, pan, gestures,
EXIF rotation, navigation behavior, animation behavior.

#### Audience

T2 / T3 primary.

#### UI layout

Five sections:

* **Zoom** — default zoom mode, zoom speed, zoom lock value.
* **Pan** — pan speed.

#### Backend wiring

* Gesture settings each gate a SwiftUI gesture modifier attached to the
  viewer canvas. Disabled gestures are not attached, so they do not
  consume events.
* The image-booster cache is an `actor` with an `LRUCache<URL, DecodedImage>`
  sized by `cache_image_count`. `cache_max_dim` and `cache_max_mb` filter
  out images that would dominate the cache.
* `huge_image_threshold` switches the decoder pipeline from
  `CGImageSourceCreateImageAtIndex` to a `libvips`-backed tiled decoder for
  panoramas and gigapixel images.
* `real_time_file_update` registers `FSEventStream` watchers for the
  active scope's source directories. The debounce is in
  `list_of_files.mdx`.

#### Workflow

A user opens a 60 MP RAW. The viewer notices the dimension exceeds
`huge_image_threshold`, switches to the tiled decoder, and shows the file
in a few hundred milliseconds with progressive refinement. Pinch-to-zoom
on the trackpad triggers the GPU-accelerated zoom path.

## 6. Validation and Error Handling

* **Type validation**: every `set_setting(path, value)` call validates
  the value against the field's Swift type. A mismatch returns an MCP
  error and does not write.
* **Range validation**: numeric fields with declared ranges (see the
  tables) reject out-of-range values.
* **Enum validation**: enum-typed fields reject unknown cases.
* **Cross-field invariants**: a small set are enforced before write:
  * `slideshow.interval_to_seconds >= slideshow.interval_seconds` when
    `use_random_interval = true`.
  * `viewer.cache_max_dim >= 256` (degenerate caches break the renderer).
  * `gallery.thumb_size` is clamped to {32, 48, 64, 96, 128, 192, 256,
    384, 512}.
* **Conflict detection**: keybinding writes flag conflicts but do not
  prevent the write; the GUI surfaces a warning.

Invalid writes leave `settings.json` unchanged. The MCP error message
includes the offending path, the offered value, the expected type, and
the constraint that failed.

## 7. Testing and Acceptance Criteria

The Settings subsystem ships when all of the following pass:

* **Round-trip parity**: write every setting at least once from the GUI,
  read it back via `get_setting`; values must equal.
* **MCP parity**: same as above driven from MCP; the GUI must reflect
  the change without a relaunch.
* **Crash safety**: kill the process mid-write 100 times; on relaunch,
  either `settings.json` is the prior valid state or
  `settings.json.bak` is and we recover from it.
* **Migration**: load 10 representative upstream `igconfig.json` files
  from real users; every field that has a mapping must produce the
  documented Mac value.
* **Schema documentation**: every field in this spec has an entry in
  `Settings.schema.json` and `list_setting_paths()` returns it.
* **Localization**: every label in the spec is keyed in `en.iglang` and
  at minimum one non-English pack ships with the same key set.
* **Accessibility**: every control has a VoiceOver label;
  `KeyboardShortcutCaptureField` exposes its current chord to
  AT-SPI / VoiceOver.
