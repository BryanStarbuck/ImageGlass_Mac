# Crop

This page is the **complete product + engineering specification** for
the crop tool in ImageGlass_Mac.

It cross-references:

* [Automation](./automation.mdx) вҖ” automation, panels, scopes, and Local Storage.
* [Settings вҶ’ Tools вҶ’ Crop](./settings.mdx#414-tools--crop) вҖ” the
  persisted crop defaults.
* [Panels](./panels.mdx) вҖ” the framework the crop tool's controls
  live in (as a docked or floating panel).
* [File List Panel](./list_of_files.mdx) вҖ” when batch-cropping, the
  file panel drives the file iteration.

The Mac stack used throughout: **Swift 6 + SwiftUI** for the crop
panel's controls, **AppKit + Metal** for the overlay drawn over the
viewer canvas, **Core Image** for the standard re-encode crop,
**libjpeg-turbo** for the true lossless JPEG crop path, and
**ImageIO** (`CGImageDestination`) for writing output files with
preserved metadata.

---

## 1. Audience and Goals

### 1.1 Audiences

| Tier | Audience | What they want from crop |
| --- | --- | --- |
| T1 | Casual user | Quick rectangular crop, drag-and-confirm, save. Five seconds. |
| T2 | Photographer | Aspect-ratio locking (1:1, 4:3, 3:2, 16:9, custom), pixel-precise dimensions, lossless JPEG crop where possible, batch with a persistent selection across the file list. |
| T3 | Designer | Exact-pixel input fields, snap-to-grid, rule-of-thirds overlay, golden-ratio overlay, save to multiple sizes. |
| T4 | Power user / MCP | Drive crop from Claude Code via an MCP tool that takes `(path, x, y, w, h, output)` and returns the new file path. |

### 1.2 Goals

* **Match the standard crop tool UX** for cases that work well
  (eight handles, modifier-key constraints, aspect-ratio presets).
* **Add lossless JPEG crop.** ImageGlass_Mac uses a true lossless path
  via libjpeg-turbo's `tjTransform` for MCU-aligned rectangles. The
  user sees no quality loss for any JPEG crop with edges aligned to
  the 8 px / 16 px MCU grid (which is most crops).
* **Preserve metadata by default.** EXIF, IPTC, XMP, ICC profile, and
  GPS coordinates are carried through every write unless the user
  explicitly opts out.
* **Live preview.** The crop result is shown in real time as the user
  drags. No "preview" button.
* **Persistable selection.** A T2 batch-cropping a folder can lock the
  selection and walk the file list, applying the same crop to each
  image.
* **Programmable.** MCP can drive a crop without GUI input. A Claude
  Code workflow can crop a directory of screenshots to the same
  region.

### 1.3 Non-goals

* No content-aware crop (no AI fill, no seam carving). The crop tool
  produces a strictly-smaller rectangular subset of the source.
* No perspective correction or skew correction. Those are different
  tools.
* No multi-rectangle / non-rectangular crop in v1.
* No video cropping (motion JPEGs and Live Photos are rendered as a
  single still during crop; the user is informed).

## 2. The Crop UI

The crop UI has two parts: a **panel** (docked or floating, per
[panels.mdx](./panels.mdx)) and an **overlay** drawn over the viewer
canvas.

### 2.1 Overlay

The overlay is drawn by the Metal viewer renderer as a second pass
after the image. It consists of:

* A translucent dim mask covering the entire canvas (default 60 %
  black; respects `appearance.background` and the user's reduce-transparency
  accessibility setting).
* A sharp rectangular cutout (the selection) вҖ” fully transparent so
  the image shows through.
* The selection's border, drawn 1 pt wide with the system accent
  color.
* Eight resize handles at the corners and edge midpoints, drawn
  inside transparent AppKit `NSView` overlays so they receive mouse
  events directly. Handle visual: 10 pt Г— 10 pt rounded square,
  accent fill, 1 pt white outline. The corner handles are slightly
  larger (12 pt).
* An optional grid overlay (rule of thirds, golden ratio, golden
  spiral diagonals, 8 Г— 8 grid, or none) вҖ” drawn inside the
  selection.

Five grid modes:

### 2.7 Persistence modes

`CropController.persistent` (panel toggle, mirrored from
`tools.crop.persistent_selection` in Settings) controls what happens
when the active image changes (via Next, Previous, or a file-panel
selection change):

| Persistent | Behavior on image change |
| --- | --- |
| `false` (default) | Selection is cleared. The crop tool stays open. The user starts a new selection on the new image. |
| `true` | Selection is preserved in pixel coordinates. If the new image is smaller in either axis, the rect is clipped to fit. The user can immediately Save / Crop without re-drawing. |

Persistent mode is the batch-crop workflow described in В§6.2.

## 4. Output Pipeline

### 4.1 The four save paths

| Action | What it does |
| --- | --- |
| **Crop (in-app replace)** | Replaces the in-memory image with the cropped CGImage. No file write. Pushes an Undo entry. Useful before further edits or before saving. |
| **Save** | Writes the cropped image back to the source file using the source format. Atomic-rename write so a crash mid-save cannot corrupt the source. Preserves the file's mtime if `general.preserve_modified_date = true`. |
| **Save AsвҖҰ** | Opens `NSSavePanel`. The default name is `<basename>_cropped.<ext>`. The user can change the format here. |
| **Copy** | Writes the cropped CGImage to `NSPasteboard.general` (raw bitmap + the source-format file data + a PNG fallback). |

### 4.2 Per-format encoder

Selecting an output format selects an `ImageIO` writer or a
specialized writer:

| Format | Writer | Notes |
| --- | --- | --- |
| `jpeg` | libjpeg-turbo `tjTransform` (lossless path) OR `CGImageDestination` (re-encode path) | See В§4.3. |
| `png` | `CGImageDestination(public.png)` | Lossless by definition. |
| `webp` | `CGImageDestination(org.webmproject.webp)` | macOS 14+ writes natively. Quality respected. |
| `heic` | `CGImageDestination(public.heic)` | Quality respected. HDR preserved if source is HDR. |
| `avif` | `CGImageDestination(org.aomedia.avif)` | macOS 14+. |
| `tiff` | `CGImageDestination(public.tiff)` | Lossless; LZW compression on by default. |

The output quality (`tools.crop.default_output_quality`, default 90)
maps to `kCGImageDestinationLossyCompressionQuality` for JPEG / WebP /
HEIC / AVIF.

### 4.3 The lossless JPEG path

ImageGlass_Mac uses libjpeg-turbo's `tjTransform` for lossless JPEG
crops, preserving quality at every crop. The pipeline:

1. When the user clicks Save / Save As with a JPEG source and
   `tools.crop.lossless_jpeg = true`:
2. The pipeline rounds the crop rectangle outward to the nearest MCU
   boundary. Standard JPEG MCUs are 8 px (grayscale) or 16 px
   (4:2:0-chroma; the most common). The rounding may add up to 15 px
   on each side.
3. If the rounded rect equals the user's rect (the user's rect was
   already MCU-aligned), the path is a pure lossless transform: the
   pipeline calls libjpeg-turbo's `tjTransform` with `TJXOP_NONE`
   and `TJXOPT_CROP` set to the rounded rect. The result is bit-exact
   to the source minus the cropped DCT blocks.
4. If the rounded rect is larger than the user's rect (most common),
   the panel shows a one-line caption вҖ” "Lossless crop nearest the
   MCU boundary; the rectangle was expanded by N px" вҖ” and writes the
   lossless rect. The user can disable this and force re-encode via
   the panel's `Lossless JPEG` toggle.
5. EXIF, IPTC, XMP, ICC profile, and Adobe APP14 markers are
   preserved verbatim by libjpeg-turbo when `TJXOPT_COPYNONE` is
   *not* set.

For PNG / WebP / HEIC / AVIF / TIFF the standard `CGImageDestination`
path is used.

### 4.4 Metadata preservation

| Metadata | Source | How preserved |
| --- | --- | --- |
| EXIF | `kCGImagePropertyExifDictionary` | Copied with `CGImageDestinationCopyImageSource(_:image:options:)`. EXIF orientation is rewritten to `.up` because the cropped CGImage is already in display orientation. |
| IPTC | `kCGImagePropertyIPTCDictionary` | Same. |
| XMP | `kCGImagePropertyMetadataKey` | Same. |
| ICC profile | `kCGImagePropertyProfileName` + the actual color space of the CGImage | Set on the destination via `kCGImageDestinationEmbedThumbnail` / `kCGImagePropertyColorModel`. |
| GPS | `kCGImagePropertyGPSDictionary` | Copied. The user can opt out per-export in Save As (a "Strip GPS" checkbox). |
| Adobe APP14 | Only relevant to JPEG | Preserved by libjpeg-turbo. |

The user can opt out of metadata preservation globally via
`edit.preserve_exif = false` ([Settings В§ 4.7](./settings.mdx#47-edit)).

### 5.2 Overlay rendering

The overlay is rendered by `ViewerRenderer.renderCropOverlay(_:)` as
a second pass after the image pass in the Metal `MTKView`:

```swift
extension ViewerRenderer {
    func renderCropOverlay(in pass: MTLRenderCommandEncoder) {
        guard let rect = cropController.rect else { return }
        encodeDimMask(pass, excluding: rect)
        encodeGrid(pass, in: rect, mode: cropController.gridMode)
        encodeBorder(pass, around: rect, color: .accent)
        // handles are AppKit NSViews layered above MTKView вҖ” not in this pass
    }
}
```

Handles are eight transparent `NSView`s positioned via
`CALayer.frame` updates, parented to the Metal view's superview.
This is the simplest way to get reliable mouse-event delivery on a
Metal-rendered surface. Each handle has a `NSCursor` set
appropriately (corner-resize cursors, edge-resize cursors). An
alternative "Metal-sprite handles + manual hit-test" implementation
is documented for future consideration but is not v1.

### 5.3 Hit-test rules

For a click at point `p` in canvas coordinates:

* If `p` is inside a handle's frame вҶ’ start `resizing` with that
  handle.
* Else if `p` is inside `rect` вҶ’ start `moving`.
* Else (no selection or click outside the selection) вҶ’ start
  `drawing` a new rect from `p`.

### 5.4 Undo

Every `applyAndReplace()` pushes onto the standard `UndoManager` of
the active window. The undo entry holds the prior CGImage handle.
`Save in place` does *not* push to Undo (it is a file write; the
file write is reversible only via the backup file).

### 5.5 The crop.json side-file

For batch-crop workflows the user can persist a per-image crop in a
side-file:

`<dir>/.imageglass/crop/<basename>.json`

```json
{
  "rect": [x, y, w, h],
  "applied": false,
  "saved_to": null,
  "updated_at": "2026-06-02T18:34:21Z"
}
```

This is opt-in via `tools.crop.write_side_files = true` (default
`false`) and lets a MCP-driven batch worker resume where it left off
without re-reading the image to find the same selection. The
side-file directory is hidden by default by `viewer.show_hidden_files`.

## 6. Workflow Scenarios

### 6.1 Quick crop (T1, 5 seconds)

1. User has an image open. Presses вҢҳK (crop).
2. Overlay appears with a centered 50 % rectangle (per default
   `tools.crop.init_selection = select50Percent`).
3. User drags a corner to tighten.
4. User presses Return вҶ’ Crop applied in app.
5. User presses вҢҳS вҶ’ file written back, original replaced.

### 6.2 Photographer batch crop to 16:9 (T2)

1. Open the folder; the file panel lists 80 photos.
2. Open crop (вҢҳK). Set Aspect ratio = 16:9. Set Persistent selection
   = on.
3. Position the rectangle precisely on the first image.
4. Press вҢҳвҮ§S вҶ’ Save As, accept the default name, accept JPEG, choose
   quality 95. (The panel offers Lossless JPEG; for arbitrary
   16:9-not-MCU-aligned rects the panel will warn that lossless
   expand will be small or zero.)
5. Press вҶ’ (next image). The selection persists. The rectangle
   automatically clips to the new image's bounds.
6. Adjust if needed; вҢҳвҮ§S to save.
7. Repeat for the remaining photos.

### 6.3 Designer asset extraction (T3)

1. User opens a screenshot containing several UI elements.
2. вҢҳK. Sets Aspect ratio = Custom, W = 3, H = 2 (for a 3:2 thumbnail
   crop).
3. Types exact pixel values into X, Y, Width, Height for the first
   element.
4. вҢҳвҮ§S вҶ’ name it `dropdown.png` вҶ’ PNG вҶ’ save.
5. Types new X, Y for the second element. вҢҳвҮ§S вҶ’ name it
   `radio.png` вҶ’ save.

### 6.4 Claude Code drives crop via MCP (T4)

```
User to Claude Code:
  "Look at ~/screenshots/2026-06-02/*.png, crop just the dialog box
   from each one вҖ” top-left corner is around (40, 80), about 800 by
   500 вҖ” and save as <name>_dialog.png next to the originals."

Claude Code calls:
  list_files_in_scope("screenshots-2026-06-02")   // returns 12 files
  for path in paths:
    crop_image(
      input_path: path,
      x: 40, y: 80, width: 800, height: 500,
      output_path: path.replacing(".png", "_dialog.png"),
      format: "png"
    )
```

The user sees nothing in the GUI; the files appear on disk. No GUI
focus is stolen. Each call is journaled to `logs/crop.log`.

### 6.5 Lossless JPEG crop of a 50 MP photo

* Source: 8688 Г— 5792 px JPEG, 24 MB.
* User draws a 6000 Г— 4000 rect roughly centered. The rect is not
  MCU-aligned.
* The pipeline rounds outward to the nearest 16-px boundary вҶ’
  6016 Г— 4000 from (1344 вҶ’ 1344, 896 вҶ’ 896). Total expansion: 16 px
  on the right.
* libjpeg-turbo `tjTransform(..., TJXOP_NONE, TJXOPT_CROP, ...)` runs
  in ~80 ms on M3 Apple Silicon, producing a 12.4 MB JPEG.
* The output is bit-exact to the source minus the cropped DCT
  blocks вҖ” no quality loss, no re-encode artifacts.
* EXIF, IPTC, XMP, ICC profile are preserved verbatim.

Timing table (Apple M3 Pro, 12-core CPU):

| Source | Crop method | Time |
| --- | --- | --- |
| 50 MP JPEG | Lossless (`tjTransform`) | ~80 ms |
| 50 MP JPEG | Re-encode at Q=95 | ~620 ms |
| 50 MP HEIC | `CGImageDestination` re-encode at Q=90 | ~900 ms |
| 50 MP PNG | `CGImageDestination` lossless | ~1.4 s |

## 8. Open Questions

* **Aspect ratios for vertical video frames.** Photographers wanting
  to crop iPhone Live Photos to 9:16 want a single click; the
  `Ratio9_16` preset is included. Should the
  default preset list be reordered so 9:16 is more prominent on
  vertical-source images? Tentative: yes, but only if the source has
  an EXIF Orientation that implies portrait.
* **Multi-rectangle export.** A designer wanting to extract three
  thumbnails from one screenshot could benefit from a "queue of
  rectangles" workflow. Out of scope for v1; reasonable v2 work.
* **Snap to image features.** A "snap to detected edge" mode would
  be a meaningful crop-quality improvement and is a natural Vision /
  Core Image experiment. Out of scope for v1.
