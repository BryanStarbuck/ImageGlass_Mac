# Directory Panel UI

> **Document type:** Product specification.
> **Audience:** Engineering (implementation) and Product (behavior, interaction).
> **Owners:** PM for behavior and interaction; CTO for the
> SwiftUI / AppKit interop and performance budget.
> **Status:** v1 spec for the Mac-native fork. See `starbuck-fork.mdx` §2
> for the umbrella charter (new column, modular panels) and
> `local_storage.mdx` for the on-disk data model that this panel renders.

---

## 1. Purpose

The **Directory Panel** is the new column the Starbuck fork adds to the
ImageGlass main window. It renders, in a hierarchical tree view, the set
of files resolved from the active scope in `directories.yaml`. It is the
control surface through which the user picks which image the main viewer
displays.

Conceptually:

* `directories.yaml` declares **one or more root directories** (plus
  inclusion / exclusion criteria — see `local_storage.mdx`).
* The application walks the directory hierarchies under those roots,
  filters the files by the scope's criteria, and produces a **resolved
  file list**.
* The Directory Panel renders that resolved file list as a **tree** that
  mirrors the on-disk directory structure.
* Clicking an image file in the tree makes the main viewer display that
  image.
* Scrolling up and down through the tree (or using arrow keys) advances
  the main viewer's selection in turn.

The panel is the user-facing twin of the MCP server: the server lets
Claude Code edit scopes from outside the app, and this panel lets the
user see and navigate the resulting file list from inside the app.

---

## 2. Where It Fits in the Window

The fork's new column hosts **modular panels** (see `starbuck-fork.mdx`
§2). v1 ships with two panels in this column:

| Panel | Purpose | Can be toggled off? |
| --- | --- | --- |
| **Directory Panel** *(this spec)* | Hierarchical tree of resolved files. Drives the main viewer's current image. | Yes. |
| **Image Preview Panel** | Thumbnail / preview strip of the same resolved files. Complementary to the tree. | **Yes** — the user can turn this one on or off independently of the Directory Panel. |

Both panels read the **same** resolved file list from Local Storage. They
are two views of one data model, not two parallel systems. Selection
state is shared: clicking an entry in either panel updates the other,
and updates the main viewer.

The user can run with:

* Directory Panel only (default — the tree is the primary navigation).
* Directory Panel + Preview Panel (tree plus thumbnail strip).
* Preview Panel only (thumbnails as the primary navigation — for users
  who prefer image-led browsing).
* Neither (the new column hidden entirely — the viewer falls back to
  upstream behavior).

**Default-visible guarantee.** On first launch, and on every relaunch
where a `layout.json` does not explicitly place the Directory Panel
elsewhere, the panel is rendered on the left of the main window in
tree mode. The window's title-bar `sidebar.left` toolbar button (and
its `⌘L` keyboard twin) is the single control that hides or restores
it; see [panels.mdx §5.6.1](./panels.mdx#561-title-bar-toggle-for-the-left-file-panel)
for the title-bar button contract and [panels.mdx §6.5](./panels.mdx#65-bootstrap-reconciliation--guarantee-the-file-panel-is-visible)
for the bootstrap reconciliation pass that keeps a stale layout from
swallowing the panel.

**Resizable width.** The Directory Panel column carries an 8 pt
grippable divider on the side that faces the viewer (right edge
when docked left, left edge when docked right). Dragging it widens
or narrows the panel relative to the viewer; the new width
persists to `layout.json` via `PanelLayoutModel.setSize("file_panel",
…)` exactly the same way the framework's docked dividers do —
there is no parallel "inline width" store. Dragging the column
below 160 pt snaps it closed and runs the same code path as `⌘L`
(the title-bar toggle and MCP `hide_panel("file_panel")` route
here too), so the persisted state and the audit log are coherent
across all three controls. See
[panels.mdx §5.3.1](./panels.mdx#531-inline-file-panel-draggable-splitter)
for the full contract.

## 3. View Mode: Hierarchical Tree

The Directory Panel is **always hierarchical** in v1. It is **not** a
flat list. A flat list of filenames is offered by the Preview Panel
(thumbnails) and by upstream ImageGlass's existing gallery; this panel's
job is to expose the directory structure.

### 3.1 Tree Construction

* The roots of the tree are the **include directories** declared in
  `directories.yaml` for the active scope. If three root directories are
  declared, the tree has three top-level nodes.
* Under each root, the tree contains **only the subdirectories that
  contain at least one in-scope file**, transitively. Empty branches are
  pruned so the tree stays scannable.
* Leaf nodes are **files** that passed the scope's inclusion / exclusion
  criteria.
* If a root directory's `recursive` flag is `false`, only that root's
  direct children are considered; deeper subdirectories are not walked.

### 3.2 Node Display

Each node shows:

| Node kind | Primary text | Secondary text |
| --- | --- | --- |
| Root directory | Display name (last path component) | Full absolute path, dimmed, on hover or as a tooltip. |
| Subdirectory | Folder name | Optional file count of in-scope descendants. |
| File | Filename | Optional metadata badge (e.g., dimensions, file size) once the metadata pass has run. |

Icons follow the system icon (`NSWorkspace.shared.icon(forFile:)`) so
the tree feels native and the user gets the same file-type cues they
see in Finder.

### 3.3 Expand / Collapse

* Initial state on **first launch** (or after a reset): roots are
  **expanded**, immediate subdirectories are **collapsed**, except
  when a subdirectory contains the currently selected file (that
  path is auto-expanded so the selection is visible).
* Disclosure triangles follow standard `OutlineGroup` /
  `NSOutlineView` behavior.
* `⌥`-click on a disclosure triangle expands or collapses the entire
  subtree, matching Finder.

#### 3.3.1 Persistence Across Launches

Every directory the user expands or collapses is recorded in the
session-state block in `settings.yaml` (see `local_storage.mdx`
§4.2.1) under `session.directory_panel.expanded_paths`. The map is
keyed by absolute POSIX path; the value is `true` (expanded) or
`false` (collapsed). On the next launch the tree is materialized
with the same disclosure state — what the user opened stays open,
what they collapsed stays collapsed.

* A directory the user has **never touched** does not appear in
  `expanded_paths` and falls back to the §3.3 default rule.
* A path that no longer exists at launch time (e.g., the user
  deleted the folder) is silently dropped from the map on the next
  save.
* `⌥`-click on a disclosure triangle records the entire subtree
  state, not just the clicked row — every descendant directory
  whose disclosure state changed is updated in the map atomically.
* The expansion map is part of the **save-on-quit** session block.
  It is not journaled mid-session; a crash loses at most the
  disclosure changes the user made since the last debounced
  settings write.

## 4. Data Source and Refresh

* The panel reads the active scope from `directories.yaml` (see
  `local_storage.mdx`). The "active scope" is the scope named in
  `settings.yaml` as the currently selected scope.
* The panel never re-walks the file system on its own. It reads the
  **resolved file list** that was last written by the scope evaluator
  (see `mcp.mdx` §6, `evaluate_scope`).
* When the resolved file list changes — because the user added a
  directory, because Claude Code mutated the scope via MCP, or because
  the file-system watcher fired — the panel refreshes incrementally.
  It must not collapse the tree or lose the user's selection on
  refresh.

### 4.1 Live File-System Updates

The application's file-system watcher (`FSEventStream`) emits change
events for the directories named in the active scope. On a change:

1. The scope is re-evaluated in the background.
2. The new resolved file list is diffed against the displayed tree.
3. New files appear, removed files disappear, with a brief highlight to
   draw the user's eye.
4. The current selection is preserved if the file still exists; if it
   was removed, selection moves to the nearest surviving sibling.

## 5. Selection and Image Preview

### 5.1 Single Selection Drives the Viewer

* Selecting a **file** node makes the main viewer display that image.
* Selecting a **directory** node does **not** change the viewer — it
  only highlights the directory. (A future v2 may show a folder
  thumbnail mosaic; out of scope here.)
* Selection is **single-row** by default. Multi-select (⌘-click,
  ⇧-click) is supported for bulk actions (copy, move, exclude — see
  `menus.mdx`).

### 5.2 Scrolling and Sequential Browsing

The panel is the primary navigation surface for stepping through
images.

* `↓` and `↑` move to the next / previous **visible** node (skipping
  collapsed subtrees) and, if that node is a file, load it into the
  viewer.
* `→` expands a collapsed directory; on an expanded directory, moves
  to the first child. `←` collapses; on a collapsed or leaf row, moves
  to the parent.
* `Home` / `End` jump to the first / last visible node.
* `Space` toggles the slideshow on the current scope (same shortcut
  upstream uses).
* Scroll-wheel and trackpad scrolling are pass-through — they pan the
  panel, they do **not** change the selection. This matches Finder and
  avoids the "I scrolled and now I'm looking at a different image"
  surprise.

### 5.3 Linked With the Preview Panel

When the Preview Panel is also visible, the selection in the Directory
Panel and the Preview Panel is **the same selection**. Changing one
changes the other. Both reflect the viewer's current image at all
times.

### 5.4 Global Cursor — Viewer and Tree Are Always in Sync

There is exactly **one** "currently selected file" in the app at any
moment. That single cursor lives in `AppState.selectedFile` and is the
authority that drives both the viewer's loaded image and the highlighted
row in the Directory Panel tree. Whenever the cursor moves — for *any*
reason — the tree must scroll the row into view and highlight it.

The cursor moves through several channels, and the tree highlight is
required to follow every one of them:

| Cursor source | Tree must… |
| --- | --- |
| User clicks a file row in the tree | Highlight the clicked row (trivially — the click itself moved the cursor). |
| User presses `↑` / `↓` / `←` / `→` per §5.2 | Highlight the newly cursor'd row and ensure it is scrolled into view. |
| **Slideshow advances to the next image** (timer-driven, see `menus.mdx` View ▸ Slideshow) | Highlight the slideshow's new file. The tree advances row-for-row in lock-step with the slideshow — never lags, never sits on the previous file. |
| User clicks in the Preview Panel | Highlight the corresponding tree row (per §5.3). |
| MCP `select_file` call (`mcp.mdx` §6) | Highlight the row that the external tool selected. |
| File-system watcher removes the current file | Cursor moves to the nearest surviving sibling (§4.1); tree highlight follows. |
| Drag-and-drop or "Load From Clipboard" loads an ad-hoc file outside the scope (§5A.2 of `hotkeys.mdx`) | Tree clears its highlight and shows the italic "(ad-hoc)" header — the cursor is *not* in the tree's resolved list, so no row is highlighted. |

Implementation contract:

* The slideshow timer mutates `AppState.selectedFile` exactly the same
  way `↓` does. There is no separate "slideshow cursor." This is what
  guarantees the tree row, the Preview Panel, and the viewer all stay
  on the same image without per-surface bookkeeping.
* The tree subscribes to `AppState.selectedFile` via SwiftUI
  `@Observable` and reacts by (a) setting the row's selected state and
  (b) calling `proxy.scrollTo(id, anchor: .center)` when the row is
  outside the visible viewport. The auto-scroll target is **center**,
  not **top**, so the user can see what is coming next and what just
  played.
* The same `ScrollViewReader` contract applies in **List mode** — when
  `state.selectedFile` changes the flat list also scrolls the selected
  row into view at `.center`. The Directory Panel must never become a
  stale navigation surface in either mode, including during fast
  slideshows that cross folder boundaries (the slideshow timer just
  steps through `AppState.orderedNavigationFiles`; the panel reacts to
  the same `selectedFile` change either way).
* Implementation surface: each file row in tree mode is tagged with
  `.id(fullPath)` so `ScrollViewReader.scrollTo` can find it; in list
  mode the `ForEach(... id: \.self)` already keys rows by their path
  string. The auto-scroll is dispatched on the next runloop tick so
  the auto-reveal expansion of the containing folder (`DesignTreeNode`
  `onChange(of: selected)`) has time to materialize the row before
  `scrollTo` looks for its id. The animation matches the framework's
  200 ms `easeInOut` so the row glides rather than snaps.
* When the cursored file lives inside a currently-collapsed subtree
  (e.g., the slideshow crossed a folder boundary), the path from the
  active root down to the file is auto-expanded so the row can be
  highlighted. This is the same auto-expand rule used in §3.3 for first
  launch. Auto-expanded directories *are* recorded in
  `session.directory_panel.expanded_paths` per §3.3.1 — if the user
  collapses them again later, that collapse sticks.
* The auto-scroll is debounced at 60 Hz max. During a fast slideshow
  (sub-second intervals on a 10k-file scope) the tree still tracks but
  does not thrash `NSOutlineView` row layout.

This is the only acceptable behavior: a user who switches their gaze
from the image canvas to the Directory Panel mid-slideshow must see the
highlighted row exactly match what is on the canvas. Anything else
makes the tree a stale, untrustworthy navigation surface.

## 5A. Default Focus and Hover-Routed Pointer Gestures

§5.4 defines *which file* is selected. This section defines *which
surface* receives the next keystroke when the user is not editing text.

### 5A.1 The Three Focus Surfaces

At any moment, exactly one of the following surfaces holds the keyboard
first-responder:

1. **The image canvas** (`MTKView` / `ImageCanvasView`).
2. **The Directory Panel tree**, with the cursor on the currently
   selected file row.
3. **The filter text field** in the panel toolbar (`list_of_files.mdx`
   §3A — the `⌕ filter` input).

Other text fields (rename, search-in-settings) are folded into category
3 — any focused text input behaves the same way for the routing rules
below.

### 5A.2 Default Focus Is the Tree, Not the Filter Field

On app launch, after dismissing a sheet, after closing Preferences, and
after the user clicks anywhere in the main window that is **not** the
filter field, the focus defaults to **the Directory Panel tree row of
the currently selected file** — *not* the filter field, and *not* the
image canvas.

The reason is hotkey routing. Single-letter hotkeys — `S` (toggle
slideshow), `Z` (Zoom to Fit), `W` (Zoom to Width), `C` (Center), `N`
(Normalize), `+` / `-` (zoom in/out), `P` (Copy Path), `L` (Load From
Clipboard) — are the fast muscle-memory keys defined in
`hotkeys.mdx` §5. They only fire when the focused surface is the viewer
**or** the Directory Panel (see `hotkeys.mdx` §3). If the filter field
silently held focus, those bare letters would type into the filter
instead of firing the action, which is a confusing footgun.

By defaulting focus to the tree:

* `S` toggles slideshow, `Z` fits to viewport, `+` zooms in — they
  always fire, with no "click the image first" ritual.
* `↑` and `↓` walk the resolved file list per §5.2 — the most-used keys
  in the app — and the user does not have to switch focus before using
  them.
* The Directory Panel cursor row is visible and highlighted (per §5.4),
  so the user has a clear visual anchor for where focus lives.

The filter field acquires focus **only** when the user explicitly
clicks it, presses `⌘ F` (`list_of_files.mdx` §4.5), or tabs into it.
While focus is in the filter field:

* Letter keys type into the filter — including `S`, `Z`, `W`, `+`, etc.
  This is the standard text-input behavior and is correct.
* Arrow keys move the text caret within the filter string, not the
  file selection. This is also standard.
* **Escape** returns focus to the Directory Panel tree on the current
  cursor row. (The filter text is preserved; only focus moves.)
* **Return / Enter** commits the filter (no-op since filters apply
  live) and returns focus to the tree, same way Escape does.

After any of those exits, focus is back on the tree row, and bare
letter hotkeys work again.

### 5A.3 Pointer Hover Overrides Focus for Pointer-Originated Gestures

Keyboard focus and pointer events are different routing planes. The
rule is:

* **Keyboard events** route to the focused surface (per §5A.2).
* **Pointer events** — trackpad scroll, mouse-wheel scroll, pinch /
  `MagnifyGesture`, two-finger pan, click — route to the surface
  **under the pointer**, regardless of keyboard focus.

This matters specifically for **zoom via trackpad two-finger scroll**
(when the user has bound the scroll wheel to zoom in Settings ▸ Viewer
▸ *Mouse wheel*, per `hotkeys.mdx` §5.2) and **pinch-to-zoom**. When
the user's pointer is hovering over the image canvas, trackpad scroll
zooms the image — even though keyboard focus is on the Directory Panel
tree row.

Concretely, all of the following must work without the user clicking
the canvas first:

* Pointer over canvas, focus on tree row, **two-finger scroll up** →
  the viewer zooms in.
* Pointer over canvas, focus on tree row, **pinch-out** → the viewer
  zooms in smoothly (`MagnifyGesture`).
* Pointer over canvas, focus on tree row, **two-finger pan after
  zooming in** → the viewer pans.
* Pointer over canvas, focus on filter field (user just typed), pinch
  or scroll-zoom still fires on the canvas. (The filter field does not
  consume pointer events that originated outside its bounds.)
* Pointer over a tree row, focus on tree row, **two-finger scroll** →
  the tree scrolls. This is the inverse case: pointer over the tree
  scrolls the tree, not the viewer. Matches the §5.2 rule that scroll
  is pass-through and does not change the selection.

Implementation contract:

* The image canvas owns `NSView.scrollWheel(with:)` and the
  `MagnifyGesture` SwiftUI modifier. AppKit's native hit-testing
  delivers the event to the view under the pointer, independent of
  first-responder state, which is exactly what we want.
* No code path may funnel pointer events through the first-responder
  chain (e.g., do not promote `MagnifyGesture` to a window-level
  shortcut). That would re-introduce the "click the image first" bug.
* The tree's own scroll view (`NSScrollView` wrapping `NSOutlineView`)
  must not swallow trackpad events that started outside its frame —
  AppKit handles this correctly by default, so the requirement here is
  to *not* override it.

### 5A.4 Acceptance Criteria for §5A

The focus and pointer-routing model is considered v1-complete when:

1. Opening the app with a populated scope leaves focus on the tree row
   of `AppState.selectedFile`. Pressing `S` immediately toggles the
   slideshow; pressing `Z` immediately fits the image. Neither types
   into the filter field.
2. Clicking the filter field, typing, and pressing **Escape** returns
   focus to the tree row. Pressing `S` after Escape toggles the
   slideshow.
3. With focus on the tree row, moving the pointer over the canvas and
   two-finger scrolling zooms the image (when scroll-to-zoom is
   enabled). Pinch on the trackpad zooms the image smoothly via
   `MagnifyGesture`. Neither action changes which file is selected.
4. With focus on the tree row, moving the pointer over a tree row and
   two-finger scrolling scrolls the tree. The viewer's zoom does not
   change. The selected file does not change.
5. During a running slideshow, the tree row highlight advances in
   lock-step with the viewer (per §5.4). The user can take focus into
   the filter field, type a query, press Escape, and the tree row is
   still highlighting whichever file the slideshow most recently
   advanced to.

## 6. Interaction Surface (Summary)

Interaction details for the menu bar, context menus, and toolbar live
in `menus.mdx`. The Directory Panel itself exposes:

| Interaction | Effect |
| --- | --- |
| Click a file row | Load that image into the viewer. |
| Double-click a file row | Load + open in the OS default app (optional config). |
| Click a directory row | Highlight it; no viewer change. |
| Double-click a directory row | Expand / collapse. |
| Right-click any row | Open the context menu (see `menus.mdx` §Context Menus). |
| Drag a row | Initiate a standard Finder-style drag of the file (`NSFilePromiseProvider`). |
| Drop a folder onto the panel | Offer to add that folder to the active scope as a new include directory. |
| `↑` / `↓` / `←` / `→` | Navigate as in §5.2. |
| Type-to-search | Jumps to the first visible row whose filename starts with the typed prefix (1.5 s buffer). |

## 7. Performance Budget

The tree must remain responsive on scopes that resolve to **tens of
thousands of files**.

* The data model is built lazily: subdirectory contents are
  materialized when the row is expanded, not on initial load.
* Row rendering uses `NSOutlineView` (via `NSViewRepresentable`) rather
  than naive SwiftUI `OutlineGroup` for v1, because `NSOutlineView`
  cell reuse handles 50k-row scopes without dropping frames.
* Initial paint of a 10,000-file scope must complete in **under 250 ms**
  on Apple Silicon.
* Selection change → image visible in the viewer must complete in
  **under 100 ms** for any format that ImageIO decodes natively.

---

## 8. Visual Style

* Native AppKit / SwiftUI styling. The panel inherits the system
  light / dark theme via `@Environment(\.colorScheme)`.
* Row height matches Finder's default list view at the user's current
  text-size preference.
* No custom chrome, no skeuomorphic textures. The panel must feel like
  it shipped with the OS.
* The active scope's name is shown as a small header above the tree,
  with a popover trigger to switch scopes (see `menus.mdx`).

---

## 9. Non-Goals (v1)

* No in-panel renaming, deleting, or moving of files. Those happen
  through Finder integration (Reveal in Finder, drag out) or
  scope-level menus, not by editing the tree.
* No tag, rating, or flag editing in the tree. The tree shows
  structure; metadata edits live in a separate (future) panel.
* No multi-scope tabs in the tree. The tree always shows exactly one
  scope at a time; switching scopes is handled by `menus.mdx` and a
  scope-picker control.
* No upstream-style "favorites" sidebar inside this panel. Upstream
  favorites belong to the viewer chrome, not to this column.

---

## 10. Acceptance Criteria for v1

The Directory Panel is considered v1-complete when:

1. Opening the app with a valid `directories.yaml` renders the tree
   with the include directories as roots, pruned to in-scope files
   only.
2. Selecting a file row loads the image in the main viewer within the
   §7 latency budget.
3. The Preview Panel can be toggled on and off independently of the
   Directory Panel without losing the active selection.
4. Editing the scope through the MCP server (see `mcp.mdx`) and calling
   `evaluate_scope` causes the tree to refresh incrementally, with
   selection preserved when possible.
5. The tree handles a 10,000-file scope without UI stutter (60 fps
   scrolling on Apple Silicon).
6. Keyboard navigation (§5.2) matches the spec on a stock US keyboard
   and respects the user's "Full Keyboard Access" system setting.
