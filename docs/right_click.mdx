# Right-Click Context Menus

> **Document type:** Product + engineering specification.
> **Audience:** Engineering (implementation, wiring), Product (text,
> grouping, item order, discoverability), Design (chrome, separators,
> sub-menu depth), QA (acceptance criteria, multi-select matrix).
> **Owners:** CPO for surface inventory, item text, ordering, ellipsis
> rules, and submenu depth budget. CTO for the SwiftUI
> `.contextMenu { … }` vs AppKit `NSMenu` / `menu(for:)` bridging, the
> activation-event routing, the audit-log contract, and the
> single-source-of-truth invariants between menus and verbs.
> **Status:** v1 spec for the Mac-native fork. This document is the
> single source of truth for **right-click** behavior. It defers to
> `menus.mdx` for menu-bar text and to `use_cases/actions.mdx` for
> the verb implementations; this document only specifies where each
> verb appears in which context menu and how the context menu is
> brought up.

## 1. Purpose

A modern Mac image viewer earns its "feels native" claim partly in
its main viewer chrome and partly in the **context menus**. A
right-click on the wrong surface, with the wrong text, in the wrong
order, with the wrong disabled-state behavior, is the single most
common reason a user files a "this doesn't feel like a Mac app"
issue. This spec exists to eliminate that.

It enumerates every right-click surface in the fork, fixes the menu
that appears on each surface, names every item, sets the order, sets
the separators, sets the sub-menus, and pins down the exact Swift
file that hosts the wiring. A reviewer reading this document should
be able to:

* Walk every surface and predict the menu before opening it.
* Locate the source code that owns the menu in under one minute.
* Confirm the verb's canonical home is the menu bar (not the context
  menu) — context menus only mirror, never invent.
* Verify the disabled-state, multi-select, and missing-file
  edge-case behavior in one sitting.

This document inherits, and never duplicates, the following:

* The **canonical menu-bar text and ordering** of every verb — see
  [`menus.mdx`](./menus.mdx). When a context-menu item appears here
  with text different from the menu bar, that is a bug.
* The **keyboard-shortcut twin** of every menu item — see
  [`hotkeys.mdx`](./hotkeys.mdx). Right-click surfaces do not invent
  new shortcuts; they display the existing shortcut beside the
  item.
* The **file-action verbs** (Rename, Move to Trash, Copy Image,
  Copy File Path, Print, New Window) — see
  [`use_cases/actions.mdx`](./use_cases/actions.mdx). Context menus
  invoke these through `FileActions.*` with `source = .context`.
* The **include-checks column** (the per-row Include / Inherit /
  Don't Include state) — see
  [`use_cases/include_checks.mdx`](./use_cases/include_checks.mdx).
  Context menus expose the same three states the column toggles,
  with the same hotkeys (`⌃1` / `⌃2` / `⌃3`) shown beside the items.
* The **MCP tool surface** — see [`mcp.mdx`](./mcp.mdx). Every
  context-menu verb either has a direct MCP twin or is one of the
  small handful of GUI-only verbs (e.g. *Quick Look*) called out in
  §6 below.
* The **Local Storage on-disk model** — see
  [`local_storage.mdx`](./local_storage.mdx). Menu items that
  mutate scopes / overrides / criteria write through the same
  atomic-replace pipeline.
* The **audit-log shape** — see
  [`error_handling.mdx`](./error_handling.mdx) §4.2. Every menu
  item that fires a verb emits one `key=value` audit line with
  `source=menu:context`.

If any of the above and this document disagree, the linked document
wins for the verb implementation and ordering; **this document
wins** for the context-menu surface (which surface, which items,
which order, which separators).

## 2. Design Principles

These are the eight non-negotiables that govern every menu in §7.

1. **Convenience surface, not canonical home.** Context menus
   re-state items that already exist in the menu bar. They never
   introduce a new verb that lacks a menu-bar entry. The user can
   always do the thing without a right-click; the right-click is
   the shortcut.
2. **One verb, one phrasing, everywhere.** The item text in the
   context menu is **byte-for-byte identical** to the menu-bar
   item, except for the *Image* / *File* qualifier the menu bar
   adds for disambiguation (e.g., the menu bar says *Move Image to
   Trash*; the context menu on a panel row says *Move to Trash*,
   because the row is the disambiguator). The qualifier rules are
   spelled out in §4.4.
3. **Item order matches user intent frequency, not menu-bar
   alphabetical order.** The most likely next action goes at the
   top. Destructive items go at the bottom, behind a separator. A
   user habituating to right-click should be able to mouse to the
   common verb without scanning.
4. **Ellipses obey Apple's HIG.** An item that opens a sheet, an
   `NSAlert`, an `NSOpenPanel`, an `NSSavePanel`, or any other
   modal that asks for input gets `…` (U+2026, one character — not
   `...`). An item that acts immediately does not. Sub-menu
   parents get `▸` (rendered automatically by AppKit). This rule is
   enforced at the linter level; the build fails if `...` appears
   in a context-menu item name.
5. **Sub-menu depth is at most one level.** A right-click never
   leads to a sub-menu that itself contains another sub-menu. If a
   second level is needed (e.g., a chooser within a chooser), the
   item opens a sheet instead. Two levels of cascading menus is a
   Windows pattern that ages badly on Mac.
6. **Shortcut twins are always shown.** Every item that has a
   keyboard shortcut from `hotkeys.mdx` displays the shortcut on
   the right-hand side of the menu row. The shortcut display uses
   the system glyphs (`⌘`, `⇧`, `⌥`, `⌃`) — never the spelled-out
   words. This is what `NSMenuItem.keyEquivalent` produces by
   default; SwiftUI's `Button` inside `.contextMenu { … }` does
   **not** show the shortcut on macOS today, which is why the
   panel-row context menus are implemented with an AppKit
   `NSMenu` bridge (§7.6.7 / §9.2) rather than a pure-SwiftUI
   `.contextMenu` modifier.
7. **Activation surface is the right-click; selection is the
   right-click target.** A right-click on an unselected row
   **selects that row first** and then opens the menu against it.
   The user does not have to left-click and then right-click. Multi-
   select is preserved if the right-click lands on an already-
   selected row (§5).
8. **Every fire emits one audit line.** No matter which surface
   fired the verb, a single `MCPAuditLogger.shared.log([…])` line
   lands in `~/Library/Application Support/ImageGlass_Mac/log.log`
   with `source=menu:context`, the verb's `tool=…`, and a fresh
   correlation id. A second line for the resulting state change
   (e.g., a file rename emitting `tool=file.rename`) follows on
   the same correlation id so a debugger can collate them.

## 3. Activation Events

Right-click is not a single event on macOS. The OS routes four
different gestures into the same `menu(for:)` / `.contextMenu { }`
codepath, and the spec must handle all four.

### 3.1 The four activation channels

| Channel | Event | Notes |
|---|---|---|
| Right mouse button down | `NSEvent.EventType.rightMouseDown` | A two-button mouse, a trackpad with right-click enabled, or an Apple Magic Mouse with secondary-click set to right side. |
| `⌃`-click (Control-click) | `leftMouseDown` with `event.modifierFlags.contains(.control)` | The pre-multi-button-mouse Mac convention. Still the only way some users right-click. |
| Trackpad secondary-click | Native two-finger click on a trackpad with the macOS preference set | Routes through the same `rightMouseDown` synthesis as a physical right button. |
| Keyboard *Menu* key | `keyDown` with `event.charactersIgnoringModifiers == "\u{1B}"` *(reserved)*; in practice `⇧F10` is bound to the same handler when the user enables it | Some Apple keyboards (and most third-party Mac keyboards) have a Menu key. macOS does not bind it by default; we bind `⇧F10` as the documented alternative so the keyboard-only user can open the context menu on the focused row. |

All four channels enter the same handler. The handler does **not**
branch on which channel fired it — there is one menu per surface,
not four.

### 3.2 The menu hot-zone and the row that owns the menu

* In the Directory Panel tree, the hot-zone is the full **row**
  (depth-indented filename cluster + the include swatch). A right-
  click anywhere on the row opens the row's menu. The disclosure
  triangle is **not** a separate hot-zone; right-clicking the
  triangle still opens the row's menu (and the chevron's normal
  left-click expand behavior is preserved on left-click).
* In the file list panels (List / Grid / Strip / Details / Column),
  the hot-zone is the **cell**: in Grid it's the thumbnail plus its
  label; in Strip it's the thumbnail; in Details/Column it's the
  table row. The existing `.contextMenu { }` modifier on
  `FileListItemView` already follows this rule; the spec only
  expands its contents (§7.6).
* On the image canvas, the hot-zone is the **entire viewport** —
  every pixel under the canvas, including the area outside the
  drawn image. The picker overlay, color picker reticle, frame
  navigator and other transient overlays do **not** swallow the
  right-click; they pass it through to the canvas.
* On the empty area of a panel (between rows, below the last row),
  the hot-zone is the empty area and produces the **panel-empty**
  menu (§7.4).
* On the panel header strip (the "FILES" label + List/Tree picker
  + filter field), the hot-zone produces the **panel-header** menu
  (§7.5). A right-click in the filter field itself does **not**
  fire this menu — text fields use the standard system editing
  menu (Cut / Copy / Paste / …) provided by AppKit, never our
  custom menu.

### 3.3 Right-click on a row that is not currently selected

Apple's Finder behavior: a right-click on an unselected row
**selects that row** before opening the menu. Our panels follow
suit:

* The right-clicked row becomes the new selection (single-select)
  before the menu opens. The viewer canvas loads the row's file
  (if it's a file). The `treeNav.activeRow` cursor lands on the
  row. This happens **before** the menu is shown so the menu's
  disabled-state computation (e.g., "is the target a file or a
  folder?") sees the correct value.
* If the right-clicked row is already part of a multi-select
  group, the multi-select is preserved and the menu operates on
  the group (§5).
* The right-click does **not** scroll the row into view; if the
  user managed to right-click a row that is partially clipped at
  the panel edge, the menu still opens but the row's visible
  portion does not change. This is the conservative behavior —
  jumping the viewport on a right-click is surprising.

### 3.4 Right-click on an empty area inside a row's hot-zone

Some panel rows (e.g., a deeply indented file row) have empty
horizontal space to the right of the filename. A right-click in
that space still hits the row's hot-zone and opens the row's menu,
not the panel-empty menu (§7.4). The rule: empty space *between
rows* is panel-empty; empty space *within* a row is the row.

### 3.5 Cancelling the menu

* `Escape` dismisses the menu without firing anything. No audit
  log line is emitted (the user never crossed the commit
  boundary).
* Clicking outside the menu dismisses it the same way.
* Selecting a sub-menu parent and then `Escape` dismisses the
  sub-menu but keeps the parent menu open, then a second `Escape`
  dismisses both. This is standard `NSMenu` behavior — we do not
  customize it.

## 4. Item Text Conventions

These rules govern the exact text the user sees on every item.

### 4.1 Title case

Every item is **title case**: all major words capitalized
(*Open in New Window*, not *Open in new window*). Articles, short
prepositions, and conjunctions ("a", "an", "the", "in", "of",
"on", "to", "for", "and", "or", "by", "as") stay lowercase unless
they are the first or last word.

### 4.2 Ellipsis (`…`) policy

* Use `…` (U+2026) when the item opens a dialog, sheet, file
  picker, save panel, or any other surface that asks for further
  input.
* Do not use `…` when the item acts immediately, even if the
  action is destructive (a confirm dialog is **not** "further
  input" — it is a yes/no gate, not a parameter prompt).
* Sub-menu parents never get `…`; AppKit draws the `▸` glyph.
* The lint check at build time matches `Sources/**/*.swift` for
  the literal three-dot string `"..."` inside a context-menu
  builder and fails the build.

### 4.3 Verb form

* Item text is an **imperative verb phrase**: *Reveal in Finder*,
  *Remove Directory*, *Re-evaluate Scope*, *Move to Trash*.
* Avoid the noun form (*"Reveal in Finder Action"*) and the
  gerund (*"Revealing in Finder"*).
* Avoid "Click to …" prefixes; the user already knows clicking is
  what fires the verb.

### 4.4 Qualifier rules

The menu bar disambiguates by qualifying the verb (*File ▸ Rename
**Image**…*, *File ▸ **Move Image to Trash***) because the menu
bar has no row context. A context menu's surface *is* the context,
so the qualifier is dropped unless dropping it would create
ambiguity:

| Menu-bar text | Context-menu text on a file row | Context-menu text on a folder row |
|---|---|---|
| *Rename Image…* | *Rename…* | *Rename Folder…* (different verb — see §7.2) |
| *Move Image to Trash* | *Move to Trash* | *Move Folder to Trash* |
| *Copy Image* | *Copy Image* (qualifier kept — could be confused with *Copy File Path*) | *Copy as Image* (disabled on folder; see §7.2) |
| *Copy File Path* | *Copy File Path* | *Copy Folder Path* |
| *Print Image…* | *Print…* | n/a (disabled) |
| *New Window* | n/a (no per-row New Window verb) | n/a |
| *Reveal in Finder* | *Reveal in Finder* | *Reveal in Finder* |

The qualifier rule is enforced once at the menu-builder layer
(`ContextMenuBuilders.swift`) so it cannot drift between surfaces.

### 4.5 Right-aligned shortcut display

Every item that has a shortcut from `hotkeys.mdx` shows the
shortcut on the right side of the menu row. The shortcut is
drawn using the system modifier glyphs:

* `⌘` = Command
* `⇧` = Shift
* `⌥` = Option
* `⌃` = Control

Single-letter and arrow keys are drawn as plain glyphs (`Z`, `→`,
`Space`). The `Space` and `Esc` words are rendered as words; the
arrows are rendered as glyphs (`↑` `↓` `←` `→`); Return is `↵`,
Delete is `⌫`, Forward-Delete is `⌦`.

### 4.6 Sub-menu naming

A sub-menu parent ends in a space and then `▸` is drawn by
AppKit. The parent name is a **noun** describing what the sub-menu
contains (*Open With ▸*, *Show Channels ▸*, *Sort By ▸*, *Move to
Scope ▸*) — not a verb. The leaf items inside the sub-menu are
imperative verb phrases.

### 4.7 Disabled-item text

A disabled item shows the same text as enabled, in the system's
disabled foreground color. The item also publishes a `help`
property (tooltip) explaining why it is disabled (e.g.,
*"Disabled because no file is selected."*). Tooltips on disabled
menu items require the AppKit `NSMenu` path; the SwiftUI
`.contextMenu` path does not currently expose them, which is one
more reason the panel-row context menus go through AppKit.

## 5. Multi-Select Behavior

The Directory Panel tree is **single-select** in v1 (see
`dir_ui.mdx` §5.1). The file list panels (List / Grid / Strip /
Details / Column) support multi-select via `⌘`-click and
`⇧`-click. Multi-select context-menu rules:

* If the right-click lands on a row that is part of the current
  multi-select, the menu's verbs operate on the **whole
  selection**.
* If the right-click lands on a row that is *not* part of the
  current multi-select, the multi-select is **cleared**, the
  clicked row becomes the single selection, and the menu operates
  on just that row.
* The menu's item text **pluralizes** when the selection count is
  greater than 1: *Move to Trash* → *Move 3 Files to Trash*,
  *Reveal in Finder* → *Reveal 3 Files in Finder*. This pluralizes
  by setting the item's title at build time, not by toggling a
  separate enable/disable state.
* Verbs that operate naturally on one file at a time (Rename, Set
  As Wallpaper) are **disabled** on multi-select. The disabled
  tooltip reads "Disabled — pick a single file to rename." (or
  similar).
* The implementation reuses the `FileActions.*` verbs in a loop —
  see `use_cases/actions.mdx` §11.1 for the forward-compatible
  shape. v1 keeps the loop in the menu handler; v2 promotes it to
  a batch tool with a `paths: [String]` argument.

The Directory Panel tree's row context menu (§7.1) always operates
on the single right-clicked row; the multi-select rules above
apply only to the file list panels.

Each subsection below specifies one surface. The shape is the

#### 7.1.1 Sub-menu — Open With ▸

Built from `ExternalToolStorage.shared.list()` plus the system's
default-app list for the file's UTI.

| # | Item | Notes |
|---|---|---|
| 1 | <default app name> (default) | E.g., *Preview (default)*. Fired via `NSWorkspace.shared.open(url)`. The "default" suffix is grayed-but-readable. |
| — | *(separator)* | |
| 2…N | Other registered handlers for this UTI | One row per app from `NSWorkspace.shared.urlsForApplications(toOpen: url)`. Fired via `NSWorkspace.shared.open([url], withApplicationAt: appURL, configuration: …)`. |
| — | *(separator)* | |
| N+1…M | Tools registered with `register_external_tool` | One row per tool from `ExternalToolStorage.shared.list()`. Fired via `ExternalToolLauncher().launch(tool, filePath: path)`. |
| — | *(separator)* | |
| last | App Store… | `NSWorkspace.shared.open(URL(string: "macappstore://")!)` — Apple-standard pattern when no app is registered for the UTI. |
| last | Other… | Opens `NSOpenPanel` scoped to `.applicationBundle` so the user can pick any app on disk. The chosen app is **not** persisted as a default — that is a system-level setting reachable via Finder ▸ Get Info. |

The sub-menu is built lazily on hover (`NSMenuDelegate.menuNeedsUpdate`)
so the system app-list query (a slow IO operation on cold disk)
does not block the parent menu's open animation.

#### 7.1.2 Disabled-state rules

* If the file does not exist on disk (the panel is showing a
  stale row), items 6, 7, 12, 13, 14, 15, 16 are disabled with
  tooltip *"This file no longer exists on disk."* Items 1–5 and
  8 are still enabled — opening, opening with, and copying the
  path are reasonable verbs even on a missing file (Copy Path
  returns `ok=true note=missing` per `actions.mdx` §5.6).
* If the file's UTI is not `kUTTypeImage`, items 7 (Copy Image)
  and 14 (Set As Desktop Picture) are disabled.

convenience, not the primary teach-the-shortcut surface.

## 8. Selection Semantics — Worked Examples

### 8.1 Right-click an unselected file row

* **Before:** `state.selectedFile = "~/Pictures/a.jpg"`. The user

* **Item 16 *Move to Trash* fires.** `b.jpg` goes to Trash. The

  ```
  ts=… tool=file.trash from=…/b.jpg trash=…/.Trash/b.jpg

  ```

### 8.2 Right-click an already-selected row in a multi-select

* **Before:** Three files selected in the Grid view —
  `a.jpg`, `b.jpg`, `c.jpg`. The user right-clicks on `b.jpg`.

## 9. Implementation Notes

### 9.1 The single canonical menu builder file

All menu *contents* live in
`code/Sources/ImageGlass/ContextMenus/ContextMenuBuilders.swift`.
This is the only file that decides what items appear in each
menu. The hosts (DesignTreeNode, DirectoryFilenamePanel,
ImageCanvasView, etc.) call into one of:

```swift
ContextMenuBuilders.fileRow(state:path:) -> NSMenu
ContextMenuBuilders.folderRow(state:path:) -> NSMenu
ContextMenuBuilders.rootRow(state:rootPath:) -> NSMenu
ContextMenuBuilders.panelEmpty(state:) -> NSMenu
ContextMenuBuilders.panelHeader(state:) -> NSMenu
ContextMenuBuilders.imageCanvas(state:viewer:) -> NSMenu
ContextMenuBuilders.statusBar(state:) -> NSMenu
ContextMenuBuilders.fileListItem(model:entry:) -> some View
```

A SwiftUI host calls `fileListItem` and embeds the returned view
inside `.contextMenu { }`. An AppKit host calls one of the
`NSMenu` builders and returns it from `menu(for:)` (canvas) or
attaches it to the host view's `menu` property (rows, empty
space, header).

### 9.2 Wiring rows to a per-row NSMenu

SwiftUI views do not own AppKit `NSMenu`s natively. We bridge with
an `NSViewRepresentable` whose `NSView` subclass:

* Forwards `acceptsFirstMouse` / `mouseDown` / etc. to the SwiftUI
  parent (no event interception) — so left-click selection and
  the include-swatch button still work.
* Overrides `menu(for event: NSEvent) -> NSMenu?` to build and
  return the per-row menu.
* Calls a `preselect` closure before opening the menu so §3.3
  fires.

The bridge view is **invisible** — it sits as a `.overlay` on
each row view with `.allowsHitTesting(false)` everywhere except
the right-click path (which AppKit dispatches independent of
SwiftUI hit-testing). Concretely, `acceptsFirstMouse(for:)`
returns `false` so left-clicks pass through; `menu(for:)` runs
for right-clicks.

The bridge is implemented in
`code/Sources/ImageGlass/ContextMenus/ContextMenuBridge.swift`.

### 9.3 Wiring the image canvas

`ImageCanvasView` is already an `NSView` subclass. Override
`menu(for event: NSEvent) -> NSMenu?` directly:

```swift
override func menu(for event: NSEvent) -> NSMenu? {
    guard let state = hostState, state.selectedFile != nil,
          let viewer = hostViewer else { return nil }
    return ContextMenuBuilders.imageCanvas(state: state, viewer: viewer)
}
```

The `hostState` / `hostViewer` references are stamped on the
canvas by `CanvasHost.makeNSView` (see
`code/Sources/ImageGlass/Viewer/ImageViewer.swift`).

### 9.4 Audit logging

Every menu item handler ends with a single
`MCPAuditLogger.shared.log([…])` line. The handler also calls
through to the canonical verb (`FileActions.*`,
`IncludeStateController.*`, `DirectoriesStore.*`,
`viewer.zoomToFit()`, etc.) which itself emits its own audit
line. The convention:

* The menu handler emits **one** `tool=menu.click` line:
  ```
  ts=… tool=menu.click menu=<surface> item=<item-id>
   source=menu:context corr=<id> ok=true
  ```
* The verb it dispatches emits **one** line with the verb's own
  `tool=…` on the same `corr=<id>`. This pairs cleanly.
* The surface ids are the §6 surface numbers, written as
  `"directory_panel.file_row"`, `"directory_panel.folder_row"`,
  `"directory_panel.root_row"`, `"directory_panel.empty"`,
  `"directory_panel.header"`, `"file_list.item"`,
  `"viewer.canvas"`, `"viewer.video"`, `"viewer.svg"`,
  `"viewer.image_info_overlay"`, `"gallery.strip"`,
  `"window.floating_file_tree"`, `"panel.header_tab"`,
  `"status_bar"`. The item ids are kebab-case verb names
  (`"open"`, `"open-in-new-window"`, `"reveal-in-finder"`,
  `"copy-image"`, `"copy-file-path"`, `"include"`, `"inherit"`,
  `"dont-include"`, `"rename"`, `"move-to-trash"`,
  `"open-with.<bundle-id>"`, `"open-with.tool.<tool-id>"`, …).

### 9.5 Accessibility

* Every `NSMenuItem` carries an `accessibilityLabel` matching its
  title. Disabled items additionally publish
  `accessibilityHint` with the disabled-reason tooltip.
* `accessibilityIdentifier` is the same kebab-case id as the
  audit log (`"reveal-in-finder"`, `"move-to-trash"`, …) so
  XCUITest can address items by identifier without depending on
  user-visible text. The id is the *menu-item* id, not the menu
  surface id; the surface is derived from the menu's own
  identifier.
* VoiceOver pronounces the title and the shortcut. The system
  handles this natively for `NSMenuItem`; we do not customize.

### 9.6 Localization

* Every string in `ContextMenuBuilders.swift` goes through
  `NSLocalizedString(_:tableName:bundle:value:comment:)`.
* The strings live in
  `code/Sources/ImageGlass/Resources/en.lproj/ContextMenus.strings`.
* The `comment` argument is a one-line description of the
  surface and item id so a localizer can sort by surface.
* The pluralization in §5 uses `String.localizedStringWithFormat`
  with a `.stringsdict` file (`ContextMenus.stringsdict`) — not
  string concatenation. The plural rules for English live in the
  dict; other locales add their own entries.

### 9.7 Performance budget

* The menu must open within **16 ms** (1 frame at 60 Hz) of the
  right-click event. The bulk of that is the `NSMenu` `popUp`
  call; the menu *contents* construction must be **under 1 ms**.
* The *Open With ▸* submenu's app-list query
  (`NSWorkspace.urlsForApplications(toOpen:)`) is the only
  expensive call. It runs lazily in `menuNeedsUpdate(_:)` on
  hover, never on parent-menu open.
* The Quick Look preview opens through the system's own
  `QLPreviewPanel` infrastructure — we do not own its latency.

### 9.8 Linting

A SwiftSyntax-based linter at `tests/lint/ContextMenuLint.swift`
fails the build when:

* A literal `"..."` (three dots) appears inside a menu builder.
* A menu item title contains a leading or trailing whitespace.
* A menu item title is not in title case (heuristic: any "word"
  not in the stoplist starts with a lowercase letter).
* A new menu surface is added without updating the §6 inventory
  table (the linter parses this MDX file and cross-references the
  surface ids in `ContextMenuBuilders.swift`).

These rules are enforced because the human review window is too
narrow to catch text drift across fifteen surfaces.

## 12. Telemetry — `menu.open` and `menu.click`

Every successful right-click that produces a menu emits one
`menu.open` line. Every click on an item emits one `menu.click`
line. The two share a correlation id so the open-to-click latency
can be measured offline.

```
ts=2026-06-06T14:23:11.412Z tool=menu.open menu=directory_panel.file_row
 path=/Users/you/Pictures/2026/06/IMG_0001.heic items=16 selected=1
 source=event:right-click corr=R-ab12cd34 ok=true

ts=2026-06-06T14:23:12.001Z tool=menu.click menu=directory_panel.file_row
 item=move-to-trash source=menu:context corr=R-ab12cd34 ok=true

ts=2026-06-06T14:23:12.012Z tool=file.trash from=/Users/you/…/IMG_0001.heic
 trash=/Users/you/.Trash/IMG_0001.heic source=menu:context corr=R-ab12cd34
 ok=true
```

The `menu.open` line carries the **count of items** so a P95
opens-with-N-items dashboard is one query away. The `selected`
field is the multi-select count for surfaces that support it (§5).

## 13. Accessibility — Keyboard-Only Operation

A user who never touches a mouse must be able to open and use
every context menu via keyboard alone:

1. **Open the menu** — `⇧F10` is the documented binding. (macOS
   has no system-wide context-menu key; the spec adds `⇧F10`
   because most third-party Mac keyboards lack a Menu key.)
   `⇧F10` fires the same `menu(for:)` chain as a right-click,
   with the activation point set to the focused row's center.
2. **Navigate** — `↑` / `↓` move between items; `→` opens a
   sub-menu; `←` returns to the parent; `↵` (Return) commits the
   highlighted item; `Esc` dismisses.
3. **Mnemonic letters** — Apple deprecated `&`-style mnemonics
   long ago; we do not introduce them.
4. **VoiceOver** — handled automatically by AppKit / SwiftUI; the
   §9.5 identifiers are the stable hooks XCUITest scripts use.

The keyboard binding is registered globally on the main window
via `NSEvent.addLocalMonitorForEvents(matching: .keyDown)` in
`code/Sources/ImageGlass/ContextMenus/ContextMenuKeyboardOpener.swift`.

## 14. Edge Cases

### 14.1 Sandboxed builds

In sandboxed builds, `NSWorkspace.shared.urlsForApplications(toOpen:)`
returns only apps the user has previously authorized via the
power-box. The *Open With ▸* submenu surfaces *fewer* entries but
the menu still functions; the *Other…* item routes through the
power-box `NSOpenPanel`, which the sandbox does permit.

The *Set As Desktop Picture* item requires the
`com.apple.security.assets.user-selected.read-only` entitlement
plus a `NSDesktopFolderUsageDescription` Info.plist key. Without
either, the item is disabled with tooltip *"Disabled — desktop
access not authorized."*

### 14.2 Multi-window

Every window has its own menu surface; the menu always operates
on the **right-clicked window's** `AppState`, not the frontmost
one. The `WindowState`-aware `AppState` injection is already in
place via the `bindToFrontmostWindow(_:)` path (see
`code/Sources/ImageGlass/Window/WindowState.swift`).

### 14.3 Per-window scope dispatch

Mutations through `DirectoriesStore.shared` route to the right
window's per-window file via `frontmostResolver` (see
`code/Sources/ImageGlassCore/Storage/DirectoriesStore.swift` lines
25–36). A right-click on window B's panel mutates window B's
YAML, not window A's.

### 14.4 The clipboard does not contain a folder path (Add Directory From Clipboard)

The item is disabled. Tooltip: *"Disabled — the clipboard does
not contain a folder path."* The check is cheap
(`pb.string(forType: .string).flatMap { … isDir }`) and runs in
`menuNeedsUpdate(_:)` on hover so it does not delay the parent
menu's open.

### 14.5 The image canvas is in slideshow mode

Slideshow does **not** suppress the menu. The right-click pauses
the slideshow (same way a left-click does), opens the menu, and
the slideshow stays paused until the user clicks *Slideshow* in
the menu bar again or presses `Space`. The pause is logged with
`tool=slideshow.pause source=menu:context`.

### 14.6 The user right-clicks during a drag

`NSMenu.popUp(positioning:at:in:)` cancels the in-flight drag
(AppKit's documented behavior). The right-click then opens the
menu normally. No special handling needed; the spec confirms the
behavior so QA does not file a regression.

## 15. Non-Goals (v1)

* **Customizable menus.** v1 ships a fixed surface inventory and
  fixed per-surface menus. A future *Settings → Customize Context
  Menus* sheet that lets the user add or remove items is
  reserved.
* **Folder rename.** The §7.2 menu shows *Rename Folder…* in the
  matrix; the v1 ship has the item **disabled** with tooltip
  *"Coming in v2."* `FileActions.renameFolder` is the planned
  v2 verb.
* **Right-click for image regions.** A future *Crop Region* /
  *Save Region As* surface on the canvas after a marquee
  selection is reserved. v1 only has the canvas-wide menu.
* **Right-click on a panel divider.** Today's right-click on a
  divider is owned by `NSSplitView` and we do not override it.
* **Stable, rebindable menu order.** v1 fixes the order; a v2
  Settings sheet may allow per-user reordering.
* **Right-click cancel inside an open menu re-fires the previous
  menu.** Not a target; the menu just dismisses.

## 16. Acceptance Criteria for v1

The right-click surface is considered v1-complete when:

1. Each of the fifteen surfaces in §6 opens its specified menu
   on a right-click, on `⌃`-click, on trackpad secondary-click,
   and on the `⇧F10` keyboard binding.
2. Item order, text, separators, shortcut display, and ellipsis
   match §7 byte-for-byte; the linter passes.
3. Disabled-state rules (§7 individual subsections + §11
   matrix) match.
4. Multi-select pluralization (§5) fires on the File List
   panels.
5. §3.3 selection-on-right-click fires on every row surface.
6. Every menu fire emits the `menu.open` line; every commit
   emits the `menu.click` line plus the verb's own audit line on
   a shared correlation id (§9.4 / §12).
7. The Quick Look item opens `QLPreviewPanel` against the right
   file in under 100 ms (system budget; not our latency).
8. The image canvas menu is suppressed when no image is loaded
   (§7.7.1).
9. The on-canvas `WKWebView`'s native web-content menu is
   suppressed in favor of our menu when the SVG renderer is in
   web mode (§7.9).
10. XCUITest scripts can address every item by its kebab-case
    `accessibilityIdentifier` and the suite passes on a clean
    install.
