# macOS Menu Bar and Context Menus

> **Document type:** Product specification.
> **Audience:** Engineering (implementation) and Product (text,
> grouping, keyboard shortcuts).
> **Owners:** PM for menu text, grouping, and discoverability; CTO for
> the SwiftUI `CommandMenu` / AppKit `NSMenu` bridging and shortcut
> conflicts.
> **Status:** v1 spec for the Mac-native fork. Aligns with
> `starbuck-fork.mdx`, `dir_ui.mdx`, `mcp.mdx`, and `local_storage.mdx`.

---

## 1. Purpose

This document specifies the **full menu surface** of ImageGlass_Mac:
the macOS menu bar, the context menus that appear on right-click in
the Directory Panel and viewer, and the keyboard shortcuts attached to
each command.

The goal is to give the user a thoughtful, native-feeling set of menus
that exposes every interactive behavior the application supports —
especially the new fork-specific features: adding and removing
directories, switching scopes, toggling panels, and triggering scope
re-evaluation.

We design these menus to follow Apple's macOS Human Interface
Guidelines so the app feels like it shipped with the OS, not like a
ported Windows application.

---

## 2. Design Principles

1. **Native first.** Standard menus (App, File, Edit, View, Window,
   Help) appear in their standard positions with their standard
   shortcuts. The user's muscle memory works on day one.
2. **One verb per menu item.** Item text reads as an active verb
   phrase: *Add Directory…*, *Re-evaluate Scope*, *Reveal in Finder*.
3. **Ellipses (`…`) for any item that opens a dialog or asks for
   further input.** Items that act immediately do not get an ellipsis.
4. **Capitalization follows macOS title case** — all major words
   capitalized: *Show Preview Panel*, not *Show preview panel*.
5. **Group related commands with separators.** Dense menus get
   subdivided by intent (e.g., add vs. evaluate vs. switch).
6. **Shortcut conflicts with system-reserved shortcuts are not
   allowed.** Custom shortcuts never collide with Spotlight (`⌘ Space`),
   Mission Control, or Edit-menu standards.
7. **Every fork-specific menu item exists in only one place in the
   menu bar.** Context menus may repeat menu-bar items for convenience,
   but the menu bar is the canonical surface.

---

## 3. Menu Bar Layout

The menu bar contains the following menus, in order:

1. **ImageGlass_Mac** (application menu)
2. **File**
3. **Edit**
4. **View**
5. **Image**
6. **Scope**
7. **Panel**
8. **Go**
9. **Window**
10. **Help**

`Scope` and `Panel` are the two new fork-specific menus. They sit
between `Image` and `Go` so the user encounters them after the
upstream-style image commands but before the navigation menu.

## 8. Localization Notes

* All menu strings live in a single `Menus.strings` file so they can be
  localized without code changes.
* Ellipses are the Unicode `…` character (U+2026), not three dots.
* En dash (`–`) is used in shortcut display where applicable;
  modifier glyphs (`⌘`, `⇧`, `⌥`, `⌃`) come from the system.

---

## 9. Non-Goals (v1)

* Rebindable shortcuts. The shortcut set in this document is fixed in
  v1; configurable hotkeys are deferred.
* Touch Bar customization. The Touch Bar surface is small and
  declining; a minimal "scope-aware" Touch Bar layout may ship later.
* Per-scope menu customization. The menu bar is identical regardless of
  active scope.

---

## 10. Acceptance Criteria for v1

The menu surface is considered v1-complete when:

1. Every menu and item in §4 is wired to a working command. Disabled
   items show as grey and have a tooltip explaining why (e.g., "Delete
   Scope is disabled because this is the only scope.").
2. Every shortcut in §7 resolves to exactly one command in the active
   focus context, with no system-shortcut conflicts.
3. Context menus in §5 appear on right-click in their respective
   surfaces and contain the listed items in the listed order.
4. Scope ▸ Re-evaluate Scope and the MCP `evaluate_scope` tool produce
   identical resolved file lists for the same scope (see `mcp.mdx`
   §6).
5. Help ▸ Keyboard Shortcuts… lists every binding in this document,
   pulled from the same source of truth that wires the menu commands.
