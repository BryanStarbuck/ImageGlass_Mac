# Use Case — MCP File Panel Tour

This page is a **scenario-by-scenario tour** of the directory /
filename panel and the MCP calls that drive it. It is written as a
sequence of user actions: *do this first, then this, then this*, and
at each step it spells out the on-screen and on-disk result so a
human (or a test) can verify the step actually worked.

It is the user-facing companion to:

* [Automation](../automation.mdx) §2 (panels) and §4–§5 (Local
  Storage + MCP) — the charter being implemented.
* [Panels](../panels.mdx) — the framework the file panel lives in.
* [File List Panel](../list_of_files.mdx) §3A — the engineering
  spec for `directories.yaml`, the background walker, the in-memory
  tree, the filter shape, the Directories menu, and the directory
  MCP tools exercised here. Every behavior verified in this tour
  has its contract pinned down in §3A.
* [Error Handling](../error_handling.mdx) §4.2 — the `log.log`
  record format that every MCP call in this tour must produce.

Nothing in this tour adds new MCP tools. It exercises the existing
ones from
[list_of_files.mdx §3A.10](../list_of_files.mdx#3a10-mcp-tools-for-the-file-tree)
from a user's point of view and pins down the verifiable outcome of
each call.

## 0. Common Preconditions

These hold for every scenario unless a scenario explicitly overrides
them.

* The app is installed and signed; first launch has already happened
  once (so `~/Library/Application Support/ImageGlass_Mac/` exists).
* The MCP server is reachable. For Claude Code this is the
  `imageglass` MCP server registered in the user's MCP config; for
  the tour it does not matter which transport is used.
* The Local Storage layout is:

  ```
  ~/Library/Application Support/ImageGlass_Mac/
  ├── directories.yaml         # the file tree panel's root-directory list
  └── logs/
      └── log.log              # every MCP call lands here
  ```

* `directories.yaml` is the **only** file backing the file tree
  panel for this tour. Its schema, in-memory mirror, and lifecycle
  are spec'd in
  [list_of_files.mdx §3A](../list_of_files.mdx#3a-the-file-tree-panel--directoriesyaml-and-live-walker).
* The viewer canvas is in the center. The file panel docks on the
  **left** by default (see [Panels](../panels.mdx) §7.2 — the
  *Browser* preset).
* The panel only ever lists files of three kinds — images, SVG,
  video — per the built-in file-kind filter in
  [§3A.3](../list_of_files.mdx#3a3-built-in-file-kind-filter). Other
  files in the same directory (e.g. `README.txt`) are never visible.
* Test fixtures live under `~/Pictures/tour/`:

  ```
  ~/Pictures/tour/
  ├── beach/
  │   ├── 001.jpg
  │   ├── 002.jpg
  │   └── notes_old.png
  ├── mountains/
  │   ├── peak.heic
  │   └── peak.jpg
  └── README.txt          # never appears in the panel (not image/SVG/video)
  ```

* "Verify" means: read the named file with `cat`, look at the named
  pixel region on screen, or read the named log line. Each verify
  step is a check a human or an XCUITest run can perform.

## 2. Select a File — Image / SVG / Video All Load in the Viewer

**Rule, stated once:** any time a row is selected in the file panel
— by mouse click, keyboard arrow, MCP `select_file`, or the §10
auto-select — and the file's `kind` is `.image`, `.svg`, or
`.video`, the viewer canvas loads that file. This is the canonical
selection-drives-viewer contract from
[list_of_files.mdx §4.3](../list_of_files.mdx#43-selection-model).
The three kinds load through their dedicated preview surfaces:

* `.image` → decoded by ImageIO, rendered fit-to-window on the
  Metal canvas.
* `.svg` → SVG preview surface ([../svg.mdx](../svg.mdx)) — the
  canvas swaps to `WKWebView` for animated SVG, Core Graphics for
  the static fallback.
* `.video` → video preview surface ([../videos.mdx](../videos.mdx))
  — the canvas swaps to AVKit `AVPlayerView`. Playback does **not**
  start automatically on selection; the user presses Space to
  start. This avoids surprise audio.

The rule fires unconditionally for the three kinds. The auto-play
exclusion above is the only per-kind difference.

### 2.1 Preconditions

* Scenario 1 completed.
* Add a root directory by hand for this scenario (the MCP version is
  covered in §4). Two equivalent GUI paths:
  * Menu: **Directories → Add Directory…** (default hotkey ⌥⌘D, see
    [§3A.8](../list_of_files.mdx#3a8-directories-menu-and-hotkeys)),
    pick `~/Pictures/tour/beach`.
  * Drag `~/Pictures/tour/beach` from Finder onto the file panel.
  The panel walks; its list now shows `001.jpg`, `002.jpg`,
  `notes_old.png`. `README.txt` is **not** shown — it is dropped at
  the built-in file-kind filter.

### 2.2 Actions, in order

1. Click the row labeled `001.jpg` in the file panel.
2. Click `notes_old.png` — another `.image` file with a different
   UTI; the canvas re-loads, replacing (not overlaying) the prior
   image.
3. (Optional, if the fixture is extended with `beach/clip.mp4`)
   click `clip.mp4`. The canvas swaps to AVKit and shows the first
   frame with the play button enabled; pressing Space starts
   playback.
4. (Optional, if the fixture is extended with `beach/logo.svg`)
   click `logo.svg`. The canvas swaps to the SVG preview surface
   and renders the SVG fit-to-window.

### 2.3 Expected result

* The file panel highlights each clicked row with the system accent
  color.
* The viewer canvas loads the newly-selected file every time, never
  the previous one.
* The status bar reads the file name, pixel dimensions (or video
  duration / SVG `viewBox`), and file size of the active selection.
* The MCP server emits one `selection_changed` push event per
  selection
  ([list_of_files.mdx §8.1](../list_of_files.mdx#81-push-events))
  with payload `[<absolute path>]`.

### 2.4 Verify

* On screen: after each click the viewer shows that file, never the
  previous one; the status bar matches.
* On disk:

  ```sh
  tail -n 4 ~/Library/Application\ Support/ImageGlass_Mac/logs/log.log
  ```

  contains one `tool=panel.select_file path=<absolute path>` audit
  record per click. The GUI selection runs through the same audit
  path as MCP-driven `select_file`, so an external observer sees
  the same line shape regardless of which surface drove the change.

### 4.1 Preconditions

* Scenario 3 completed. `directories.yaml` contains one root with

### 4.2 Actions, in order

1. From Claude Code:

   ```
   add_directory(path="~/Pictures/tour/mountains")
   ```

   The optional `filter` argument is omitted; the new root inherits
   the empty filter (everything that passes the built-in file-kind
   filter is included).

### 4.3 Expected result

* The MCP server canonicalizes the path
  (`/Users/<you>/Pictures/tour/mountains`), validates that it
  exists, appends the new entry to `root_directories[]`, and writes
  `directories.yaml` atomically.
* The
  [background walker](../list_of_files.mdx#3a5-background-walker-thread)
  wakes from sleep, walks the new hierarchy, builds the in-memory
  tree, and registers a recursive `FSEventStream` watcher on the
  root.
* Within ~250 ms the panel's list (still in tree mode from §3)
  shows:

  ```
  ▾ /Users/<you>/Pictures/tour/beach
      001.jpg
      002.jpg
      notes_old.png
  ▾ /Users/<you>/Pictures/tour/mountains
      peak.heic
      peak.jpg
  ```

* `last_walked` on the new root advances to the wall-clock timestamp
  of the walk.
* Two new lines land in `log.log`: the MCP request, and the
  walk-completion event.

### 4.4 Verify

* On disk:

  ```sh
  cat ~/Library/Application\ Support/ImageGlass_Mac/directories.yaml
  ```

  contains:

  ```yaml
  schema_version: 1
  root_directories:
    - path: /Users/<you>/Pictures/tour/beach
      filter:
        items: []
      last_walked: 2026-06-02T10:05:00Z
    - path: /Users/<you>/Pictures/tour/mountains
      filter:
        items: []
      last_walked: 2026-06-02T10:05:11Z
  ```

* On screen: the panel shows both directory roots, expanded.
* On disk: `log.log` contains the two lines

  ```
  ts=… tool=mcp.add_directory path=…/mountains client=claude-code corr=ab12cd34 ok=true
  ts=… app=directory.walk path=…/mountains count=2 elapsed_ms=43 corr=ab12cd34
  ```

  (Both lines share the same `corr=` correlation id so a debugger
  can join them — see [error_handling.mdx](../error_handling.mdx)

## 6. MCP Applies a Per-Root Filter

Filters in `directories.yaml` are positive by default: items match
filenames, and `match: any` (OR) combines them. The same
[`update_directory_filter`](../list_of_files.mdx#3a10-mcp-tools-for-the-file-tree)
tool replaces the whole filter for one root in a single call. The
update is applied to the in-memory tree by
[§3A.7](../list_of_files.mdx#3a7-filter-re-evaluation--no-re-walk)
— **no filesystem walk** — so the panel updates within tens of
milliseconds even on a large root.

### 6.1 Preconditions

* Scenario 4 completed (both directories in scope). Re-add the
  `beach` root first if you ran scenario 5, so this scenario starts
  with five visible files.

### 6.2 Actions, in order

1. From Claude Code:

   ```
   set_global_filter(filter={
       "match": "any",
       "items": [
           { "pattern": "*.jpg",  "kind": "glob" },
           { "pattern": "*.jpeg", "kind": "glob" }
       ]
   })
   ```

   `set_global_filter` applies the same filter to every existing
   root in `directories.yaml`.

### 6.3 Expected result

* Each root's `filter` is replaced with the new value.
* The walker traverses the in-memory tree and flips `passesFilter`
  on every `.file` node: `001.jpg`, `002.jpg`, `peak.jpg` → `true`;
  `notes_old.png`, `peak.heic` → `false`. The walker does not touch
  the filesystem; this is the in-memory re-eval guarantee from §3A.7.
* The panel (still in tree mode) shows:

  ```
  ▾ /Users/<you>/Pictures/tour/beach
      001.jpg
      002.jpg
  ▾ /Users/<you>/Pictures/tour/mountains
      peak.jpg
  ```

### 6.4 Verify

* On disk: `directories.yaml` shows the same two-item glob filter on
  each root.
* On screen: `notes_old.png` and `peak.heic` are no longer visible.
* On disk: `log.log` shows

  ```
  ts=… tool=mcp.set_global_filter items=2 client=claude-code corr=… ok=true
  ts=… app=directory.refilter roots=2 visible_delta=-2 elapsed_ms=… corr=…
  ```

  Note `app=directory.refilter`, not `directory.walk` — the filter
  change did not re-walk the filesystem.

## 7. MCP Applies an Exclude-by-Negate Filter Item

The same filter shape supports exclusion via a `negate: true` flag
on any item. This is how the user (or MCP) writes "include all of
this kind **except** files whose name contains `_old`."

### 7.0 Mental model — positive vs. negative filter items

A root's filter is a flat list of items combined by `match: any`
(OR, the default) or `match: all` (AND). Each item carries a
`pattern`, a `kind` (`glob` / `substring` / `regex`), and a
`negate` flag.

* **Positive items** (`negate: false`, the default) declare what is
  *in scope*. A positive item that matches the filename contributes
  `true` to the combiner; one that does not match contributes
  `false`. Multiple positive items combined by `match: any` produce
  the **union** of their matches.
* **Negative items** (`negate: true`) declare what is *carved out*.
  A negative item is a **veto**: if it matches the filename, the
  file is excluded regardless of what any positive item said. If it
  does not match, it contributes nothing — it neither includes nor
  excludes.

The combined semantic, as implemented by the engine in
[list_of_files.mdx §3A.7](../list_of_files.mdx#3a7-filter-re-evaluation--no-re-walk):

```
included = positives_passed( positive_items, match )
         AND NOT any_negative_matched( negative_items )
```

with one **special case** that makes the most common real-world
intent expressible in a single item: if there are **no positive
items at all**, `positives_passed` is `true` — every file that
passed the built-in file-kind filter is in scope, and the negative
items carve out from that set. Without this special case,
`[ { *_old*, negate: true } ]` (the canonical "everything except
`_old`" filter) would be `OR-of-nothing = false` and the panel
would render empty.

This is also why `items: []` in §3A.2 means "include everything":
no positives + no negatives = `true AND NOT false = true` for every
file. The §10B rehearsal walks through the canonical
single-negate-item pattern against a real working directory.

### 7.0.1 Use-case → filter-item cookbook

The mappings below cover the intents that show up over and over.
Every row is a one-line MCP call where the `filter.items` array
matches the right-hand column.

| User intent (plain English) | `filter.items` |
| --- | --- |
| "Show every image / SVG / video." (the default) | `[]` |
| "Show only `.jpg` files." | `[ { pattern: "*.jpg", kind: glob } ]` |
| "Show `.jpg` *and* `.png`." | `[ { pattern: "*.jpg", kind: glob }, { pattern: "*.png", kind: glob } ]` *(combined by `match: any`, the default)* |
| "Show every image *except* files with `_old` in the name." | `[ { pattern: "*_old*", kind: glob, negate: true } ]` |
| "Show every image *except* files with `_DM_` in the name." | `[ { pattern: "*_DM_*", kind: glob, negate: true } ]` *(canonical §10B form, glob variant)* |
| "Show every image *except* `*_DM_*.jpg` specifically — keep `*_DM_*.png` visible." | `[ { pattern: ".*_DM_.*\\.jpg$", kind: regex, negate: true } ]` *(canonical §10B form, regex variant — the form actually stored in `directories.yaml`)* |
| "Show only JPEG / PNG **and** exclude `_old`." | `[ { pattern: "*.jpg", kind: glob }, { pattern: "*.png", kind: glob }, { pattern: "*_old*", kind: glob, negate: true } ]` |
| "Show only files whose name starts with `IMG_`." | `[ { pattern: "IMG_*", kind: glob } ]` |
| "Show only files matching a regex `^[0-9]{4}-shoot-.*$`." | `[ { pattern: "^[0-9]{4}-shoot-.*$", kind: regex } ]` |
| "Carve out two markers at once: `_DM_` **and** `_WIP_`." | `[ { pattern: "*_DM_*", kind: glob, negate: true }, { pattern: "*_WIP_*", kind: glob, negate: true } ]` *(two independent vetoes — either one excludes)* |
| "Hide every file under any `_archive/` or `.imageglass/` subtree." | path matching — see §10B.8 (v1.1) |

The cookbook is a contract: each row is a working call against the
engine. If a future change breaks any row's behavior, the §3A.7
unit tests should fail.

### 7.1 Preconditions

* Scenario 6 completed (glob filter `[*.jpg, *.jpeg]`).
* To show the `negate` item doing real work, first widen the filter
  back so PNGs are in scope again:

  ```
  set_global_filter(filter={
      "match": "any",
      "items": [
          { "pattern": "*.jpg",  "kind": "glob" },
          { "pattern": "*.jpeg", "kind": "glob" },
          { "pattern": "*.png",  "kind": "glob" }
      ]
  })
  ```

  Verify the panel now shows `notes_old.png` again before
  continuing.

### 7.2 Actions, in order

1. From Claude Code:

   ```
   update_directory_filter(path="~/Pictures/tour/beach", filter={
       "match": "any",
       "items": [
           { "pattern": "*.jpg",  "kind": "glob" },
           { "pattern": "*.jpeg", "kind": "glob" },
           { "pattern": "*.png",  "kind": "glob" },
           { "pattern": "*_old*", "kind": "glob", "negate": true }
       ]
   })
   ```

   `update_directory_filter` replaces the filter on a **single**
   root — here just `beach`. The `mountains` filter is unchanged.

### 7.3 Expected result

* `beach`'s filter records the four-item array. The fourth item's
  `negate: true` means a match excludes the file.
* In-memory refilter on the `beach` tree only: `notes_old.png` flips
  back to `passesFilter: false`. `mountains` is untouched.
* The panel updates within ~50 ms.

### 7.4 Verify

* On disk: `directories.yaml` shows the four-item filter on
  `beach`. The `mountains` root still has the three-item global
  filter from §7.1.
* On screen: `notes_old.png` is gone; `peak.jpg` and `peak.heic` are
  unaffected.
* On disk: `log.log` shows

  ```
  ts=… tool=mcp.update_directory_filter path=…/beach items=4 negate_items=1 corr=… ok=true
  ts=… app=directory.refilter roots=1 visible_delta=-1 elapsed_ms=… corr=…
  ```

## 8. Every MCP Call Is Journaled to `log.log`

This scenario is the union of the audit-trail checks above, stated
on its own so it is testable as one acceptance criterion.

### 8.1 Actions, in order

1. From a freshly truncated `log.log` (delete it; the next write
   recreates it), run scenarios 4 → 5 → 6 → 7 back-to-back.

### 8.2 Expected result

* `log.log` contains exactly the following kinds of lines, in the
  order produced, with no other MCP `tool=mcp.*` lines mixed in
  (because no other MCP calls happened):

  ```
  ts=… app=startup …                                              # if app restarted
  ts=… tool=mcp.add_directory          path=…/mountains corr=… ok=true   # §4
  ts=… app=directory.walk              path=…/mountains count=2  corr=…
  ts=… tool=mcp.remove_directory       path=…/beach     corr=… ok=true   # §5
  ts=… tool=mcp.set_global_filter      items=2          corr=… ok=true   # §6
  ts=… app=directory.refilter          roots=2          corr=…
  ts=… tool=mcp.update_directory_filter path=…/beach    corr=… ok=true   # §7
  ts=… app=directory.refilter          roots=1          corr=…
  ```

* Every `tool=mcp.*` line records `client=` (which MCP client
  called), `corr=` (8-char correlation id), `ok=true|false`, and on
  failure an `err=` field with the error code.
* Failed calls are also written. To test, send a deliberately
  malformed `add_directory` (e.g. `path` is not a string, or
  `filter.items` is missing `pattern`). The line reads
  `ok=false err=invalid_filter` and `directories.yaml` is **not**
  modified.
* `directory.walk` lines appear only for tools that mutate
  `root_directories[]` (add / remove). Filter changes produce
  `directory.refilter` lines instead — they do not touch the
  filesystem.

### 8.3 Verify

* On disk:

  ```sh
  grep -E "tool=mcp\\.(add|remove|set_global|update_directory)" \
      ~/Library/Application\ Support/ImageGlass_Mac/log.log \
      | wc -l
  ```

  prints `4` after the four §4–§7 calls.
* On disk:

  ```sh
  grep "app=directory.walk"    ~/Library/Application\ Support/ImageGlass_Mac/log.log | wc -l   # 1 (just §4)
  grep "app=directory.refilter" ~/Library/Application\ Support/ImageGlass_Mac/log.log | wc -l   # 2 (§6, §7)
  ```
* Every successful `tool=mcp.add_directory` line is followed (within
  ~500 ms by wall clock) by a matching `app=directory.walk` line
  with the same `corr=`. Every successful filter change is followed
  by a matching `app=directory.refilter` line with the same `corr=`.

## 9. No Image Previewed — Empty Viewer State

This covers the case where the panel has zero roots, or no file in
any root passes the active filter. The viewer must remain in a
defined empty state instead of holding a stale image.

### 9.1 Actions, in order

1. Starting from any scenario above, clear every root:

   ```
   clear_directories()
   ```

### 9.2 Expected result

* `directories.yaml` has `root_directories: []`. The walker tears
  down every `FSEventStream` watcher and drops every in-memory tree.
* The panel renders its empty state: a centered SF Symbol
  (`photo.on.rectangle.angled`), the label `No directories added`,
  and a secondary action `[+ Add directory]` that opens
  `NSOpenPanel` — the same action as the Directories menu's
  `Add Directory…` item.
* The viewer canvas renders its empty state: a centered placeholder
  reading `No image previewed` in the theme's secondary-label color.
  No stale image remains on the canvas.

## 10. Auto-Select First Image When Viewer Is Empty

When an MCP call adds a directory **and** the viewer is in the empty
`No image previewed` state from §9, the
[background walker](../list_of_files.mdx#3a5-background-walker-thread)
auto-selects the first image it finds. The selection lands in the
file-tree UI without the user touching the keyboard or mouse.

The rule exists so a Claude Code interaction like *"set up a
directory of my screenshots and show me one"* lands the user looking
at a real image without a second prompt.

### 10.1 Trigger conditions — all must hold

1. The viewer is in the empty state from §9 (no image loaded; either
   `0 files` or the prior selection is no longer in any root).
2. An incoming MCP call adds at least one new directory.
   Specifically any of:
   * `add_directory` with a valid `path`,
   * `set_active_scope` to a multi-scope target whose criteria
     resolve to ≥ 1 root (only relevant for users on the §3
     multi-scope feature; not exercised in this tour),
   * `refresh_directory(path)` that re-walks a previously empty
     root and now finds at least one file (e.g. the user just
     plugged in an external drive).
3. After the walk + filter eval, at least one file has
   `passesFilter: true` AND its `kind` is `.image`.

The auto-select rule deliberately fires for **images only**. The
reason is asymmetry with manual selection: §2's
selection-drives-viewer rule loads any of `.image / .svg / .video`,
but it does so because the user explicitly clicked the row.
Auto-select happens *without* the user clicking, so it must avoid
surprising side effects — an auto-loaded video could start playing
audio after Space is pressed for an unrelated reason, and an
auto-loaded animated SVG could begin a long animation the user did
not ask for. Restricting auto-select to still images keeps the
"add a directory, see an image" promise without any side effects
the user has not authorized. Manual selection of a video or SVG
remains fully supported (§2.2 steps 3–4).

If any condition fails the panel does **not** auto-advance — the
viewer stays empty and waits for an explicit `select_file`. In
particular:

* If the viewer already shows an image, the user is mid-task and
  auto-selection would yank their focus. Do nothing.
* If the new root contains only `.svg` or `.mp4` files, the viewer
  stays empty after the add. Those kinds populate the panel and the
  user can click any of them to load it (§2's manual selection rule);
  they are simply not auto-previewed by §10.
* If every file is filtered out (`passesFilter: false`), the viewer
  stays empty.

### 10.2 Walk + selection algorithm

1. The MCP call writes `directories.yaml` and returns immediately.
   It does **not** block on the walk.
2. The
   [DirectoryTreeWalker](../list_of_files.mdx#3a5-background-walker-thread)
   enumerates the new root on a background `Task` (off the main
   actor) using
   `FileManager.enumerator(at:includingPropertiesForKeys:options:)`
   with `.skipsHiddenFiles`. UTI is checked at the gate: only
   `public.image`, `public.svg-image`, `public.movie` pass.
3. **Order**: depth-first, children visited in lexicographic order
   by filename. This makes "first image" deterministic and
   reproducible.
4. As soon as the walker hits a `.file` node whose `kind == .image`
   and whose `passesFilter == true`, it posts a
   `firstImageFound(URL)` message to the main actor and **continues
   the walk in the background** to populate the rest of the tree.
5. On the main actor, the panel:
   * Re-checks the trigger conditions. If the viewer is no longer
     empty (the user clicked something while the walk ran), the
     auto-selection is abandoned and the walk continues silently.
   * Otherwise: switches the file panel to **tree** view if it is
     not already, expands the outline rows from the root down to
     the file's parent directory, sets `selection = [URL]`, scrolls
     the row into view, and loads the image in the viewer.
6. The background walk continues until the full in-memory tree is
   built. The panel updates as later batches arrive; the
   auto-selected file remains selected unless the user moves the
   selection.

### 10.3 Race and cancellation rules

* If two MCP calls arrive in quick succession (e.g. `add_directory`
  immediately followed by `add_directory`), only the most recent
  walk's `firstImageFound` is honored. Earlier walks are canceled
  via `Task.cancel()` and their pending callbacks are discarded.
* If the user clicks any row in the panel between the MCP call and
  `firstImageFound`, the auto-selection is canceled — the user is
  now driving.
* If the viewer is loaded via any other path (drag-from-Finder,
  Cmd-O, MCP `select_file`) before `firstImageFound` arrives, the
  auto-selection is canceled.

## 10B. Real-World Rehearsal — Carving Out `_DM_` Files via a Negative Filter Item

**Primary use case and primary acceptance test.** §10B is the
single end-to-end scenario this product is graded against. It is
the canonical "speak / type one sentence to Claude Code → the
panel updates in place" round-trip — the same shape every other
filter scenario in the codebase generalizes from. The 2026-06-06
verification run (corr `ed81f8bf`, see §10B.6) is the artifact a
reviewer should re-create to qualify a release candidate.

This scenario continues from §10A. The JFK/UX root is already in
`directories_window_<N>.yaml` (multi-window file naming: see
`docs/list_of_files.mdx` §3A.1 and `docs/multi_window.mdx` §6),
the walker has populated its in-memory tree, and the panel is in
tree mode showing every image / SVG / video the built-in
file-kind filter let through. Among those files are several whose
names carry an internal `_DM_` marker (used by an upstream tool
to flag "design markup" exports). The user wants those carved
out without removing the root, without touching the filesystem,
and without altering what the rest of the JFK/UX hierarchy
shows.

This is the canonical exercise of `add_filter_item` with a
single `negate: true` glob item — the smallest possible negative
filter, applied to a real working directory, that produces a
verifiable change in the panel and exactly one new entry in
`directories_window_<N>.yaml`. **Why `add_filter_item` rather
than `update_directory_filter`?** The user's request is
**append intent** (the existing filter on the root is empty;
they want to add a single rule, not replace the whole block).
The MCP manual cookbook (`code/mcp/mcp_manual.txt` §2 "Verb →
Field Cookbook") routes append verbs (*also*, *additionally*,
*one more rule*) to `add_filter_item`. The two calls produce
identical YAML when the prior filter was empty, but
`add_filter_item` is **idempotent** (the same pattern + kind +
negate + priority hashes to the same stable id — §4.5 — and
the second call returns `items_added=0` without re-writing
YAML), which is the correct shape for voice-driven workflows
that may retry.

### 10B.0 Case-sensitivity is part of the contract

The user's intent is **case-sensitive matching on `_DM_`** — the
upstream tool that emits the `_DM_` marker uses upper-case `DM`
deliberately to distinguish it from unrelated lower-case
sequences (`_dm` does NOT mean "design markup" in any upstream
producer this project sees today). Lower-case `_dm_` matches on
e.g. `admin/...` substrings are a known false-positive risk and
must not occur.

The engine offers three `kind` values for filter items and they
differ in case-sensitivity by construction. The choice of
`kind: "glob"` in §10B.3 is **not stylistic** — it is the
sole case-sensitive option that does not require the user to
hand-author a regex:

| `kind` | Case-sensitive? | Where it is decided in code |
| --- | --- | --- |
| `glob` | **Yes** | `code/Sources/ImageGlassCore/Glob.swift` — `Glob.match(_:_:)` (line 13) and the inner `matchSlice` literal-character compare `n[ni] != pc` (line 97) use raw `Character` equality, which is case-sensitive in Swift. |
| `substring` | **No** | `code/Sources/ImageGlassCore/FileList/DirectoryTree.swift` — `RootFilter.itemMatches` substring branch passes `options: .caseInsensitive` (line 298). Convenient for voice ("show me anything with hero in the name") but **wrong** for the `_DM_` marker because it would match `_dm_` too. |
| `regex` | Yes (by default) | Same file, `regex` branch (line 300) constructs `NSRegularExpression(pattern:)` with no options, which defaults to case-sensitive. Adding `(?i)` to the pattern body opts in to case-insensitivity. |

**Decision for this scenario:** `kind: "glob"`, pattern
`"*_DM_*"`. Case-sensitivity is delivered by the engine layer,
not by anything the user has to declare. If a future user
requests case-insensitive matching on this marker, the call
form switches to `kind: "substring"` and the LLM should
explicitly tell the user "switching to substring kind to
match case-insensitively" so the contract change is not
silent.

### 10B.1 The user's request, verbatim

The user types into Claude Code:

> In the JFK/UX root, don't show files whose path matches
> `".../*_DM_*.jpg"`.

That one sentence carries **three** independent pieces of
information; the MCP server must decompose them correctly because
each lands in a different place.

| Pattern fragment | What it means | Where it goes |
| --- | --- | --- |
| `.../` | "Anywhere under the root, at any depth." | **Implicit.** Every root in `directories.yaml` is walked recursively; nothing about `.../` is encoded in the filter item itself. The MCP server strips this prefix at the boundary. |
| `*_DM_*` | "Filename contains the literal substring `_DM_`." | The filter item's `pattern` (the substantive part). |
| `.jpg` | "And the file is a JPEG." | Anchored to the pattern's tail in regex form (`.*\\.jpg$`) or as a glob suffix (`*_DM_*.jpg`). Drop this fragment to widen the rule to *any* image / SVG / video with `_DM_` in its name (see §10B.7). |

The user said "don't show" — this is a **subtractive** intent,
which maps to `negate: true`. Nothing about the rest of the JFK/UX
hierarchy needs to change: the empty positive set + the §7.0 special
case still includes every other image, and the single negative item
simply vetoes the `_DM_` matches.

* `last_walked` is **not** updated. The walker did not re-walk;
  only the in-memory tree's `passesFilter` flags changed
  (changed-or-not changed depending on the include-override
  interaction in §10B.5.1).
* The existing `include_overrides:` block is **preserved
  byte-for-byte**. `add_filter_item` only touches `filter.items[]`
  on the target root; every other field on every other root
  (and every other field on this root) is left alone. This is
  enforced by `DirectoriesYAML.encode(_:)` re-emitting from the
  full `DirectoriesFile` value in
  `code/Sources/ImageGlassCore/Storage/DirectoriesYAML.swift`
  line 55, not by partial in-place editing.

### 10B.5 Expected result on the panel

* Every `.file` node under the JFK/UX root whose name matches
  the case-sensitive glob `*_DM_*` flips its `passesFilter`

  line 248). Every other file (PNG, HEIC, SVG, MP4, files
  *without* the upper-case `_DM_` marker — including any
  lower-case `_dm_` occurrences in unrelated names — etc.)

* The panel re-renders within ~50 ms. On a JFK/UX tree of
  ~5 000 files with ~80 `_DM_` JPEGs that are **not** under
  an excluded `include_overrides` subtree, the rendered file
  count drops by exactly those 80 rows; the count delta is

* The viewer canvas is **unaffected** unless the auto-selected
  file from §10A was one of the carved-out `_DM_` files. In that
  case the selection is invalidated and the panel auto-selects
  the next-lexicographic image (per §10.2 step 3); the viewer
  reloads to show that image.
* The other JFK/UX root state (`last_walked`, the `FSEventStream`
  watcher driven by
  `code/Sources/ImageGlass/FileWatcher.swift`, the in-memory
  directory skeleton built by
  `code/Sources/ImageGlassCore/FileList/DirectoryTreeWalker.swift`)
  is untouched — only per-`.file`-node `passesFilter` booleans
  changed.

### 10B.5.1 Interaction with `include_overrides`

The JFK/UX root carries a substantial `include_overrides:`
block (see §10B.4 YAML excerpt and the live
`directories_window_1.yaml` snapshot at
`~/Library/Application Support/ImageGlass_Mac/`). The override
layer and the filter-item layer are **independent gates** that
are evaluated in a fixed order by the panel's render pipeline:

1. **Built-in file-kind filter** (`FileKind.classify(path:)` in
   `DirectoryTree.swift` line 37). Drops `.txt`, `.zip`,
   non-media UTIs. A file that fails this gate never enters the
   in-memory tree and no further gate sees it.
2. **`RootFilter.evaluate(filename:)`** (`DirectoryTree.swift`
   line 248). Applies `filter.items[]` — the layer this scenario
   manipulates. Sets `passesFilter` per file.
3. **`RootDirectory.effectiveState(for:)`** (`DirectoryTree.swift`
   line 410). Walks the relative path looking for an explicit
   `include_overrides` ancestor; returns `.include` /
   `.exclude`. The panel hides any row whose resolved state is
   `.exclude` regardless of `passesFilter`.

`visible_delta=0` is the legitimate outcome whenever every
file the new filter item would have hidden was *already*
hidden by the `include_overrides` layer (i.e. it lived under
a subtree whose nearest explicit override is `.exclude`). In
the 2026-06-06 verification (§10B.3.1), every `*_DM_*` file
in the tree lived under `Posts_Download/`, which has
`state: exclude` in `include_overrides`. The filter item is
still **active** — it is the gate that will kick in the
moment a user toggles `Posts_Download` (or any other
currently-excluded ancestor that holds `_DM_` files) back to
`include`.

This independence is by design: a user can express "I never
want to see anything under `Posts_Download/`" (via the
override swatch UI in `code/Sources/ImageGlass/FileList/`)
**and** "I never want to see `_DM_` exports" (via this MCP
filter call). Removing one rule does not silently remove the
other.

### 10B.7 Equivalents and variants

The §10B.3 regex is the canonical stored form. Three useful
variants to know about:

* **Glob form** (easier to read, identical behavior for the JPEG
  case):

  ```
  { "pattern": "*_DM_*.jpg", "kind": "glob", "negate": true }
  ```

  Use this when the pattern fits a glob without ambiguity. The
  YAML stores `kind: glob` and the regex engine is never invoked.
  The MCP server accepts either form on input.

* **Widen to any kind** (drop the `.jpg` constraint — carve out
  `_DM_` from PNG, HEIC, SVG, MP4 too):

  ```
  { "pattern": "*_DM_*", "kind": "glob", "negate": true }
  ```

  This is the right form when the user says "don't show *any*
  files with `_DM_` in their name."

* **Stack a second negative** (e.g. also carve out `_WIP_`):

  ```
  filter={
    "match": "any",
    "items": [
      { "pattern": "*_DM_*",  "kind": "glob", "negate": true },
      { "pattern": "*_WIP_*", "kind": "glob", "negate": true }
    ]
  }
  ```

  Both vetoes apply independently; a file matching either is
  carved out. This is the canonical **multi-veto** shape — and it
  is why the engine (§7.0) treats negative items as standalone
  vetoes rather than as terms in the combiner.

### 10B.8 Filename vs. path matching — what the engine compares

Every item in `filter.items[]` matches against the **filename**
(the last path component), not the full path. See
[list_of_files.mdx §3A.2](../list_of_files.mdx#3a2-schema).

This is sufficient for the canonical `_DM_` case because the
marker lives in the filename: a file at
`screens/login/frame_42_DM_1.jpg` is recognized by the `_DM_` in
`frame_42_DM_1.jpg`, regardless of its parent directories. The
user-shorthand `.../*_DM_*.jpg` is normalized at the MCP boundary:
the `.../` prefix is stripped (it conveys "anywhere under the
root," which is already implicit), and what remains —
`*_DM_*.jpg` — is what the filename matcher sees.

A first-class `target: filename | path` field on each filter item
is a **planned v1.1 enhancement** for intents that genuinely need
directory-name matching ("carve out every file under any
`_archive/` directory"). Until v1.1 ships, two workarounds:

* If the parent-directory marker also appears in filenames
  (sometimes a build prepends parent names), match on that.
* Otherwise, add the offending subdirectory as its own root and
  immediately `remove_directory` it — except that this is
  exactly the inverse of what is wanted. The honest answer is
  that today the engine cannot express "carve out by parent
  directory." Don't pretend it can; emit
  `ok=false err=path_separator_in_pattern` (§10B.9) so the caller
  can rephrase.

### 10B.9 Failure modes the scenario must surface

* **Invalid regex.** If the pattern is `.*_DM_(.*\.jpg$`
  (unbalanced paren), the MCP call returns
  `ok=false err=invalid_regex`; `directories.yaml` is unchanged;
  the panel state is unchanged; one error record in `log.log`
  carries the offending pattern and the `corr=` id.
* **Unknown root path.** If `path` does not match any entry in
  `root_directories[]`, the call returns
  `ok=false err=path_not_found`.
* **Permission denied on YAML write.** If the application support
  directory is read-only (rare but possible), the call returns
  `ok=false err=storage_write_failed`; the in-memory filter is
  **not** updated (we never let memory and disk diverge on a
  write failure); the panel state stays as it was.
* **Pattern contains a path separator.** If a future MCP client
  sends a pattern like `*/foo/*_DM_*.jpg` (with `/` separators
  that are *not* the `.../` prefix shorthand), the call returns
  `ok=false err=path_separator_in_pattern` — v1 only supports
  filename matching (§10B.8). `directories.yaml` is unchanged.
