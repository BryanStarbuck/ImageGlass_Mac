# File List Panel

This page is the **complete product + engineering specification** for
the **file list panel** — ImageGlass_Mac's column, plus the
scope engine that feeds it.

This is goal #2 from [Automation](./automation.mdx)
(modular UI panels) combined with #3 (scope controls) and #4 (Local
Storage). It is also the largest single panel built on top of the
[panels framework](./panels.mdx).

It cross-references:

* [Automation](./automation.mdx) §2–§5 — the panel framework, scope controls, and Local Storage.
* [Panels](./panels.mdx) — the host framework the file panel lives
  in (movable / dockable / floating).
* [Settings → Gallery](./settings.mdx#410-gallery) — the small set of
  Settings that govern defaults.
* [Crop](./crop.mdx) — when batch cropping, the file panel drives
  the iteration.

The Mac stack used throughout: **Swift 6 + SwiftUI** (`Table`,
`List`, `OutlineGroup` for tree mode, `LazyVGrid` for the grid /
strip), **AppKit** (`NSCollectionView` for very large grids only;
`NSImageView` thumbnails) where SwiftUI hits performance limits,
**ImageIO** (`CGImageSourceCreateThumbnailAtIndex`) for fast
thumbnails, `FileManager.enumerator(at:includingPropertiesForKeys:options:)`
for the scope walker, **FSEventStream** for live file-change
notifications, and **`actor`** isolation for the caches and stores.

## 1. Audience and Goals

### 1.1 Audiences

| Tier | Audience | What they want from the file panel |
| --- | --- | --- |
| T1 | Casual viewer | A thumbnail strip at the bottom, like Preview.app. Click a thumbnail; the image loads. Nothing else. |
| T2a | Photographer | A grid of large thumbnails, sort by EXIF DateTimeOriginal, filter by extension, fast keyboard navigation. |
| T2b | Designer cataloging assets | A details list with Name / Size / Dimensions / Modified columns; resizable, sortable, reorderable. |
| T3 | Power user | Tree mode showing the directory hierarchy of a multi-directory scope; explicit scope editor next to it. |
| T4 | Claude Code / MCP | The panel binds to a scope. Claude Code edits the scope's JSON via MCP and the panel updates live; selecting an image in the panel loads it in the viewer. |

### 1.2 Goals

* **One panel, five rendering modes.** Strip, grid, details, tree,
  column — switchable from the panel toolbar. The data underneath is
  always the same resolved file list.
* **Scopes, not folders.** The panel never just "shows the contents
  of one folder." It always binds to a scope. A scope can be
  implicitly created from a single folder (the casual case) or
  explicitly defined as a multi-directory rule set.
* **Live.** When the underlying filesystem changes (files added,
  removed, renamed, modified), the panel reflects it within a
  debounced 250 ms.
* **Fast.** Thumbnails are generated on demand, cached on disk,
  paged for very large scopes (>50 000 files). The panel is usable
  on a NAS-backed scope of 250 000 RAW files.
* **Programmable.** Scopes are plain-text JSON on disk. The MCP
  server reads and writes them so Claude Code can swap, narrow, or
  rotate the active set without GUI input.
* **Native.** Drag from the panel to Finder via file promises. Drag
  from Finder into the panel to extend the scope. Space-bar Quick
  Look. SF Symbol badges. NSWorkspace icons for non-image files.

### 1.3 Non-goals

* No image editing in the panel. The panel is selection and
  navigation only; edits happen on the viewer canvas.
* No content-based search ("find me photos that look like this") in
  v1. The filter is filename + extension + date + dimensions only.
  Vision-backed search is a v2 candidate but is out of scope for
  this spec.
* No metadata editing in the panel (rotations, ratings, captions);
  those live in the dedicated Metadata panel.
* No two-way sync with Finder's sort order; macOS does not expose a
  general API for it. The panel has its own sort model.

## 3. The Scope

A **scope** is the source of truth for "what files are in this
panel." Scopes are described in [automation.mdx](./automation.mdx) §3–§4;
this section pins down the schema, semantics, and lifecycle.

### 3.1 Scope JSON

A scope is one JSON file under
`~/Library/Application Support/ImageGlass_Mac/scopes/<name>.json`.

Schema:

```json
{
  "$schema": "https://json-schema.org/draft/2020-12/schema",
  "title": "ImageGlass_Mac scope",
  "type": "object",
  "required": ["name", "schema_version", "criteria"],
  "properties": {
    "name":           { "type": "string", "pattern": "^[A-Za-z0-9._-]+$" },
    "schema_version": { "type": "integer", "const": 1 },
    "description":    { "type": "string" },
    "criteria": {
      "type": "array",
      "items": {
        "type": "object",
        "required": ["root"],
        "properties": {
          "root":           { "type": "string", "description": "Absolute path. Sandboxed builds require a saved security-scoped bookmark." },
          "bookmark":       { "type": "string", "description": "Base64-encoded NSURL bookmark data. Required in sandboxed builds." },
          "recursive":      { "type": "boolean", "default": true },
          "max_depth":      { "type": ["integer", "null"], "default": null },
          "include_globs":  { "type": "array", "items": { "type": "string" } },
          "exclude_globs":  { "type": "array", "items": { "type": "string" } },
          "include_exts":   { "type": "array", "items": { "type": "string" } },
          "exclude_exts":   { "type": "array", "items": { "type": "string" } },
          "include_hidden": { "type": "boolean", "default": false },
          "follow_symlinks":{ "type": "boolean", "default": false }
        }
      }
    },
    "sort": {
      "type": "object",
      "properties": {
        "by":        { "enum": ["name", "size", "modified", "created", "exif_date_taken", "extension", "random", "dimensions"] },
        "direction": { "enum": ["asc", "desc"] }
      }
    },
    "filter": {
      "type": "object",
      "properties": {
        "text":      { "type": "string" },
        "date_from": { "type": "string", "format": "date-time" },
        "date_to":   { "type": "string", "format": "date-time" },
        "min_width": { "type": "integer" },
        "min_height":{ "type": "integer" },
        "max_size":  { "type": "integer" }
      }
    },
    "last_evaluated": { "type": "string", "format": "date-time" },
    "resolved": {
      "type": "array",
      "items": {
        "type": "object",
        "required": ["path", "size", "modified"],
        "properties": {
          "path":     { "type": "string" },
          "size":     { "type": "integer" },
          "modified": { "type": "string", "format": "date-time" },
          "dim":      { "type": "array", "items": { "type": "integer" }, "minItems": 2, "maxItems": 2 }
        }
      }
    }
  }
}
```

The schema is the contract between the GUI, the MCP server, and any
external tool. Every required field above has a meaning; the
optional fields are progressive enhancements.

### 3.2 Worked example — "2026-shoots"

```json
{
  "name": "2026-shoots",
  "schema_version": 1,
  "description": "Active photo shoots from 2026, across local and external drives.",
  "criteria": [
    {
      "root": "/Users/bryan/Pictures/2026-shoots",
      "recursive": true,
      "include_exts": ["nef", "cr3", "dng", "jpg", "jpeg", "heic"],
      "exclude_globs": ["**/_archive/**", "**/.imageglass/**"]
    },
    {
      "root": "/Volumes/PhotoArchive/2026/exports",
      "recursive": false,
      "include_exts": ["jpg", "jpeg"]
    }
  ],
  "sort": { "by": "exif_date_taken", "direction": "desc" },
  "filter": { "min_width": 3000 },
  "last_evaluated": "2026-06-02T10:18:42Z",
  "resolved": [
    { "path": "/Users/bryan/Pictures/2026-shoots/Wedding/ceremony/001.dng",
      "size": 50331648, "modified": "2026-05-12T14:22:01Z", "dim": [8688, 5792] }
    // … 824 more entries
  ]
}
```

### 3.3 Staleness and re-evaluation

A scope is **stale** when any of:

* `last_evaluated` is older than the most-recent mtime of any
  source directory.
* An `FSEventStream` watching a source directory has signaled a
  change since `last_evaluated`.
* The user clicked **Refresh** in the panel toolbar.
* The user explicitly invoked `evaluate_scope` via MCP.

A stale scope is re-walked by `ScopeWalker` on a background task; the
panel shows a progress indicator on its toolbar refresh icon and
updates the file list as the walk completes. Subsequent reads use
the freshly-resolved list.

### 3.4 Sandbox + security-scoped bookmarks

When the app is sandboxed (App Sandbox enabled at build time), the
walker cannot enumerate a directory unless the app holds a
security-scoped bookmark for it. The scope JSON stores the bookmark
data as a base64 blob (`criteria[].bookmark`). The walker calls
`URL.startAccessingSecurityScopedResource()` before enumerating and
`stopAccessingSecurityScopedResource()` after.

The user grants the bookmark by picking the directory in an
`NSOpenPanel`, which produces a resolvable bookmark. If the user
later moves the source directory, the bookmark survives (it tracks
the directory by inode/volumeUUID, not by path).

### 3.5 Implicit scope (Cmd-O)

When the user opens a single file or folder with Cmd-O (or the
"Open With" Finder action), the panel creates an **implicit scope**
without writing it to disk:

* Single file → scope = "all images in the containing folder",
  active = the opened file.
* Single folder → scope = "all images in this folder, recursive=true,
  no extension filter".

The implicit scope lives only in memory and is replaced as soon as
the user opens another file. The user can promote an implicit scope
to a named scope with the panel toolbar's `[Save scope as…]` action.

## 3A. The File Tree Panel — `directories.yaml` and Live Walker

This section pins down the **file tree** rendering mode (§2.4) and
its dedicated storage file. The file tree is the only rendering mode
that shows the source directory hierarchy directly, so it is also
the mode that the MCP `add_directory` / `remove_directory` /
`update_directory_filter` calls drive most visibly.

Walked end-to-end as a user scenario in
[use_cases/mcp_file.mdx](./use_cases/mcp_file.mdx). That tour is the
**user-facing acceptance** for this section — every behavior
specified here must match the tour's verify steps.

### 3A.1 `directories.yaml`

The file tree panel's canonical storage is a single YAML file:

```
~/Library/Application Support/ImageGlass_Mac/directories.yaml
```

It maps **one-to-one** with what the file tree UI renders. Each
top-level row in the tree is one entry in `root_directories:`. Each
file shown underneath a row is a leaf in the recursive walk of that
root that passed the built-in file-kind filter *and* the root's
per-root filter.

The file is the source of truth for the panel. There is **no**
hidden binary state — the GUI and the MCP server read and write the
same file. Writes are atomic (temp file + rename); the panel reloads
via `FSEventStream` on the file itself.

Relationship to §3's `scopes/<name>.json`:

* `directories.yaml` is the file tree panel's **own** store. It is
  not a scope file. It exists independently of the multi-scope
  feature.
* When a multi-scope user is using a named scope, the file tree
  panel can be bound to it via the scope picker — at that point the
  panel renders the active scope's `criteria[]` and ignores
  `directories.yaml`. The default (and the case the MCP file tour
  covers) is the standalone `directories.yaml`.
* The data model in `directories.yaml` is a deliberate **subset** of
  the scope schema — only the fields the file tree panel needs.

### 3A.2 Schema

```yaml
schema_version: 1
root_directories:
  - path: /Users/bryan/Pictures/tour/beach
    filter:
      match: any           # 'any' = OR (default), 'all' = AND
      items:
        - pattern: "*.jpg"
          kind: glob       # 'glob' (default), 'substring', 'regex'
          negate: false    # true = exclude matches (default false = include)
        - pattern: "*.heic"
    last_walked: 2026-06-02T10:18:42Z
  - path: /Users/bryan/Pictures/tour/mountains
    filter:
      items: []            # empty = no per-root filter; everything that passes
                           # the built-in image/SVG/video filter is included
```

Field rules:

| Field | Required | Notes |
| --- | --- | --- |
| `schema_version` | yes | Integer. Currently `1`. |
| `root_directories` | yes | Array. Empty array is valid — see §3A.10. |
| `root_directories[].path` | yes | Absolute path. Must be canonicalized (no `~`, no `..`). The MCP server expands and canonicalizes before writing. |
| `root_directories[].filter` | no | If omitted, equivalent to `{ match: any, items: [] }` — i.e. include everything that passes the built-in file-kind filter. |
| `filter.match` | no | `any` (OR) is default. `all` (AND) is the alternative. |
| `filter.items[].pattern` | yes (per item) | The string the matcher tests against the filename (not the full path) by default. |
| `filter.items[].kind` | no | `glob` (default, uses `fnmatch(3)` semantics with `*`, `?`, `[abc]`), `substring` (case-insensitive substring match against the filename), `regex` (POSIX-extended). |
| `filter.items[].negate` | no | `false` by default. When `true`, a match means "exclude this file." |
| `last_walked` | written by walker | ISO-8601 instant. Set by the background walker on every completed walk. |

### 3A.3 Built-in file-kind filter

Independent of any per-root filter, only files of these kinds are
ever considered:

* **Images** — every UTI conforming to `public.image` (JPEG, PNG,
  HEIC, HEIF, WebP, AVIF, JXL, TIFF, GIF, BMP, RAW family, etc.).
* **SVG** — `public.svg-image`.
* **Video** — every UTI conforming to `public.movie` (MP4, MOV, M4V,
  HEVC, AVI, MKV — exact set bounded by what AVFoundation can decode
  on the running OS).

Everything else (`.txt`, `.pdf`, `.zip`, `.sketch`, …) is dropped at
the walker stage and is never inserted into the in-memory tree, so
it can never appear in the panel or be re-considered when a filter
changes. The built-in filter is not configurable from
`directories.yaml`; it is the panel's hard-wired contract.

### 3A.4 Root directory — in-memory data structure

```swift
struct RootDirectory: Sendable {
    let path: URL                       // canonical absolute path
    var filter: RootFilter              // per-root filter (§3A.2)
    var tree: DirectoryNode             // the recursive in-memory mirror
    var lastWalked: Date?
    var watcher: FileSystemWatcher?     // recursive FSEventStream
}

indirect enum DirectoryNode: Sendable {
    case directory(name: String,
                   children: [DirectoryNode])
    case file(name: String,
              uti: UTType,
              kind: FileKind,           // .image | .svg | .video
              passesFilter: Bool)       // result of applying RootFilter
}

struct RootFilter: Sendable {
    enum Match: Sendable { case any, all }
    enum ItemKind: Sendable { case glob, substring, regex }
    struct Item: Sendable {
        let pattern: String
        let kind: ItemKind
        let negate: Bool
    }
    var match: Match
    var items: [Item]
}
```

Key invariants:

* **The tree is recursive and complete** — every directory under
  the root that the walker visited is represented, including
  directories that contain no matching files. This is what the
  Outline view renders.
* **Only `.image`, `.svg`, and `.video` files appear in `.file`
  nodes.** Other files are dropped at walk time and never enter the
  tree at all.
* **`passesFilter`** is the cached result of evaluating `RootFilter`
  against the file's name. It is updated by `recomputePassesFilter`
  whenever the filter changes (§3A.7), without re-touching the
  filesystem.
* The tree is built once per walk and mutated in place by
  `FSEventStream` callbacks (§3A.6). It is never rebuilt from
  scratch unless the root directory itself becomes inaccessible.

### 3A.5 Background walker thread

Each file tree panel owns one background actor — the
**file-tree walker** — that runs for the lifetime of the panel,
**even when the panel is hidden or collapsed**. The walker is the
only thing that touches the filesystem on the panel's behalf.

```swift
actor DirectoryTreeWalker {
    private var roots: [URL: RootDirectory] = [:]
    private var sleeping: Bool = true        // see §3A.5.2

    func addRoot(_ path: URL, filter: RootFilter) async { /* §3A.5.1 */ }
    func removeRoot(_ path: URL) async { /* tear down watcher, drop from roots */ }
    func updateFilter(for path: URL, filter: RootFilter) async { /* §3A.7 */ }

    func snapshot() -> [RootDirectory] { Array(roots.values) }
}
```

The walker exists per-panel, but its work is `Task.detached(priority:
.utility)` so it does not contend with the UI. Mutations to the
in-memory `RootDirectory` graph are actor-isolated. The
@MainActor view model reads via `snapshot()` and renders the
SwiftUI `OutlineGroup` from that snapshot.

#### 3A.5.1 Adding a root

When `addRoot(path, filter)` is called (from a menu action, a drag,
or an MCP `add_directory`):

1. The walker enumerates `path` recursively with
   `FileManager.enumerator(at:includingPropertiesForKeys:options:)`,
   filtering at the UTI gate to **image / SVG / video only**.
2. As entries arrive, the walker builds the `DirectoryNode` tree
   and computes `passesFilter` for each `.file` node using
   `RootFilter`.
3. When the walk completes, the walker registers an
   **`FSEventStream` recursive watch** on `path` so subsequent
   add/remove/rename events under the hierarchy call back into the
   walker (§3A.6).
4. The walker sets `lastWalked = .now`, writes the new state into
   `directories.yaml` (atomic), and posts a `rootChanged` event so
   the panel re-renders.

The MCP call returns as soon as step 1 begins; the walk is async,
identical to the behavior covered in
[mcp_file.mdx](./use_cases/mcp_file.mdx) §10.

#### 3A.5.2 Sleep / wake

The walker is mostly idle. To reduce wake-ups on a battery:

* When all FS watchers are quiet for `idleTrigger = 30 s`, the
  walker marks itself `sleeping = true`. The actor task suspends on
  its message queue (no work; no CPU).
* Any of these wake it: an `FSEventStream` callback fires, a panel
  action arrives (menu / drag / MCP), or a periodic
  `NSBackgroundActivityScheduler` heartbeat (every 6 h) requests a
  reconciliation pass.
* Wake-up cost is < 1 ms; the walker does not re-walk the filesystem
  on wake — it just resumes processing whatever message woke it.

This sleep/wake behavior is the user-promised contract that the file
tree panel is "running even when the component is turned off."

### 3A.6 Recursive FS watching per root

Every `RootDirectory` carries one `FileSystemWatcher` wrapping an
`FSEventStream` started with `kFSEventStreamCreateFlagFileEvents`
and `kFSEventStreamCreateFlagWatchRoot`. The stream is created with
the root path and watches the entire hierarchy under it.

On a callback:

* For each changed path, the walker classifies the event:
  `.created`, `.removed`, `.renamed`, `.modified`.
* The walker updates the in-memory `DirectoryNode` tree in place:
  new `.image / .svg / .video` files are inserted with
  `passesFilter` computed; removed files are dropped; renames are
  modeled as remove+create.
* Events are debounced 250 ms (matches §6.6). A bulk `cp -R` of 5000
  files collapses into a single SwiftUI render pass.
* Non-image / non-SVG / non-video events are discarded immediately.

The watcher survives the panel being hidden — that is the point of
§3A.5's "runs even when the component is turned off." Closing the
panel does **not** tear the watcher down; only `removeRoot` does.

### 3A.7 Filter re-evaluation — no re-walk

When the user (or MCP) changes the filter for a root via
`updateFilter`, the walker **does not touch the filesystem**. It
walks the in-memory `DirectoryNode` tree and recomputes
`passesFilter` on every `.file` node:

```swift
func recomputePassesFilter(_ node: inout DirectoryNode, with filter: RootFilter) {
    switch node {
    case .directory(let name, var children):
        for i in children.indices { recomputePassesFilter(&children[i], with: filter) }
        node = .directory(name: name, children: children)
    case .file(let name, let uti, let kind, _):
        node = .file(name: name, uti: uti, kind: kind,
                     passesFilter: filter.evaluate(filename: name))
    }
}
```

This is what makes filter changes feel instant on a 250 000-file
scope — no I/O, just an in-memory traversal. The panel re-renders
only the rows whose `passesFilter` flipped.

`RootFilter.evaluate(filename:)`:

* If `items` is empty, returns `true` (no per-root filtering).
* Partition `items` into **positive** items (`negate: false`, the
  default) and **negative** items (`negate: true`).
* If there are **no positive items**, the positive result is `true`
  — i.e. every file that passed the built-in file-kind filter is
  in scope, and negative items will carve out from that set. This
  is the special case in
  [use_cases/mcp_file.mdx §7.0](./use_cases/mcp_file.mdx#70-mental-model--positive-vs-negative-filter-items)
  that makes "show everything *except* X" expressible as a single
  negate item.
* Otherwise, evaluate each positive item against the filename
  (`glob`, `substring`, or `regex` per the item's `kind`) and
  combine the results by `match`: `any` (default) = logical OR;
  `all` = logical AND. Call the combined boolean
  `positives_passed`.
* Evaluate each negative item against the filename. If **any**
  negative item matches, the file is excluded — regardless of
  `match` and regardless of `positives_passed`. This is the
  **veto semantic** spelled out in
  [use_cases/mcp_file.mdx §7.0](./use_cases/mcp_file.mdx#70-mental-model--positive-vs-negative-filter-items).
* Final result:

  ```
  included = positives_passed AND NOT any_negative_matched
  ```

The use-case → filter-item cookbook in
[use_cases/mcp_file.mdx §7.0.1](./use_cases/mcp_file.mdx#701-use-case--filter-item-cookbook)
is the contract that this engine must satisfy: every cookbook row
is a working call and a unit-test case.

### 3A.10.1 Per-node log events

Every background walk emits structured lines to `log.log` for each
node it discovers, making it possible to trace exactly what the walker
found without running a debugger. The three new `app=tree.*` events
appear in timestamp order for each walk:

| `app=` value | When emitted | Key fields |
| --- | --- | --- |
| `tree.walk_start` | Immediately before the filesystem walk begins | `path` = root dir, `corr` |
| `tree.node` | Once per node after the walk commits | `type` = `directory` / `image` / `svg` / `video`, `path` = full absolute path, `corr` |
| `tree.walk_failed` | When `walkSync` returns `nil` (path doesn't exist or isn't a directory) | `path` = root dir, `corr` |
| `directory.walk` | When the walk completes (existing event) | `path`, `count` = visible file count, `elapsed_ms`, `corr` |

A `tree.node type=directory` line appears for the root itself and for
every subdirectory discovered recursively. `tree.node type=image` (or
`svg` / `video`) lines appear for every file that passed the
`FileKind.classify` gate — both files that pass the current
`RootFilter` and files that do not (the `type=` field distinguishes
them; `passesFilter` is not recorded in this event).

Example walk sequence for root `/Users/bryan/Photos`:

```
ts=2026-06-02T10:30:01.123Z app=tree.walk_start path=/Users/bryan/Photos corr=a1b2c3d4
ts=2026-06-02T10:30:01.240Z app=tree.node type=directory path=/Users/bryan/Photos corr=a1b2c3d4
ts=2026-06-02T10:30:01.241Z app=tree.node type=directory path=/Users/bryan/Photos/2026 corr=a1b2c3d4
ts=2026-06-02T10:30:01.242Z app=tree.node type=image path=/Users/bryan/Photos/2026/img_01.jpg corr=a1b2c3d4
ts=2026-06-02T10:30:01.243Z app=tree.node type=video path=/Users/bryan/Photos/2026/clip.mp4 corr=a1b2c3d4
ts=2026-06-02T10:30:01.244Z app=directory.walk path=/Users/bryan/Photos count=2 elapsed_ms=121 corr=a1b2c3d4
```

If `tree.walk_start` appears in the log but neither `tree.node` nor
`tree.walk_failed` follows, the walk process crashed or deadlocked
between the filesystem scan and the commit step.

## 3B. Mac-Native Hierarchy Display Controls

The file tree panel requires a native macOS control that can render an
expandable/collapsible directory hierarchy. Apple provides exactly three
built-in controls for this job; no third-party library is needed. This
section describes each one, explains when to use it, and records the
architectural decision for this project.

### 3B.1 NSOutlineView (AppKit)

`NSOutlineView` is the primary AppKit tree-view control. It displays
indented rows with disclosure triangles and has been in macOS since
OS X 10.0. It is the control behind Finder's List view, Xcode's
Project Navigator, and Mail's mailbox sidebar.

**Strengths:**
* Mature, highly optimised for large hierarchies (virtual rendering —
  only visible rows are created).
* Full delegate/data-source API for fine-grained cell control.
* Drag-and-drop, multi-select, expand-all/collapse-all, persistent
  expand state all built in.
* Accessible by default (VoiceOver announces depth and expansion state).

**Weaknesses:**
* Requires `NSViewRepresentable` bridging in a SwiftUI app, which adds
  boilerplate and a `Coordinator` class.
* Cell layout is done in AppKit (XIBs or `NSTableCellView`), so mixing
  SwiftUI `View`s into cells is awkward.

**When to prefer it:** when the tree has > ~5 000 visible rows, or when
you need row-level drag-and-drop and Finder-exact visual behaviour.

### 3B.2 NSBrowser (AppKit)

`NSBrowser` is the column-based hierarchical browser that powers
Finder's Column view. The user navigates deeper by clicking into
progressively-appearing columns rather than expanding inline rows.

**Strengths:**
* Perfect "wide directory hierarchy" experience when the user spends
  time navigating depth rather than scanning breadth.
* Each column scrolls independently.

**Weaknesses:**
* Consumes a lot of horizontal space — not well-suited to a narrow
  side panel.
* Not the expected affordance for "show the full tree alongside the
  image canvas."
* Harder to show a multi-root scope where each root is a top-level
  peer.

**When to prefer it:** when the panel is wide (> 500 pt) and the user
is primarily *navigating* rather than *browsing a known set*. Deferred
to §2.5's Column mode (v1.1).

### 3B.3 List + OutlineGroup (SwiftUI)

`OutlineGroup` is SwiftUI's declarative equivalent of `NSOutlineView`.
It wraps a recursive data structure and renders an indented, expandable
tree inside a `List`. Available since macOS 11; mature in macOS 14+.

```swift
List(roots, children: \.children) { node in
    Label(node.name, systemImage: node.isDirectory ? "folder" : "photo")
}
```

Or, for a collection of top-level items each with optional children:

```swift
List {
    OutlineGroup(roots, children: \.children) { node in
        Label(node.name, systemImage: node.isDirectory ? "folder" : "photo")
    }
}
```

**Strengths:**
* Pure SwiftUI — no bridging layer, binds directly to `@Observable`
  view models, supports `@State` expand state automatically.
* Sidebar `List` style gives native macOS source-list appearance
  (translucent, grouped) for free.
* Row content is any SwiftUI `View` — thumbnails, badges, and labels
  can be mixed freely.
* `DisclosureGroup` gives per-node expand-state control when needed.

**Weaknesses:**
* Internally backed by `NSTableView`, but the implementation is opaque
  — some edge cases in virtual rendering for very large trees (> 50 000
  rows) still appeared in macOS 14 beta; resolved in 14.4+.
* Does **not** support drag between rows as elegantly as
  `NSOutlineView` yet (drag-within-list requires workarounds).

**When to prefer it:** the default choice for a SwiftUI-first app with
a side panel that shows up to ~50 000 nodes. Matches the app's overall
SwiftUI stack and avoids an `NSViewRepresentable` wrapper.

### 3B.4 Decision for ImageGlass_Mac

**The file tree panel uses `OutlineGroup` inside a `List` (SwiftUI).**

Reasons:
1. The app's primary UI layer is SwiftUI (see CLAUDE.md technology
   decision). Introducing `NSViewRepresentable` for the tree view would
   add bridging complexity across the entire panel.
2. The expected node count (< 50 000 in any real scope) is within
   `OutlineGroup`'s performance envelope.
3. `OutlineGroup` is data-binding-native — the `DirectoryNode` enum
   tree (§3A.4) maps directly to the `children:` parameter without a
   separate data-source object.
4. Row content is arbitrary SwiftUI, so thumbnail images, SF Symbol
   badges, and disclosure-state animations are all native.

The `NSOutlineView` path is kept available as a performance fallback:
if a scope exceeds 50 000 tree nodes the panel switches to an
`NSOutlineViewRepresentable` backed by a virtual data source. This
threshold is logged and covered in §6.7's memory budget table.

### 3B.5 Correct OutlineGroup usage

The single most common mistake when using `OutlineGroup` for a
directory tree — and the root cause of the recurring "tree appears
blank" bug (§3C) — is **passing the root directory node as the item**
instead of its children.

```swift
// WRONG — renders one collapsed "📁 Photos" row; nothing visible
OutlineGroup(rootNode, children: \.children) { node in
    Label(node.name, ...)
}

// CORRECT — renders the root's direct children as top-level rows
OutlineGroup(rootNode.children, children: \.children) { node in
    Label(node.name, ...)
}
```

When a `List` is in `.sidebar` style (which ImageGlass_Mac uses for the
file-tree panel), a single-item `OutlineGroup` collapses that item into
one disclosure row. The user sees only `▶ 📁 Photos` and has to click
to expand it — which looks like an empty list at first glance.

The correct pattern for a multi-root scope (where each root is a
top-level peer):

```swift
List {
    ForEach(walkerRoots) { root in
        Section(header: Text(root.displayPath)) {
            OutlineGroup(root.directChildren, children: \.children) { node in
                FileTreeRowView(node: node)
            }
        }
    }
}
.listStyle(.sidebar)
```

This renders each root's immediate children as top-level rows under a
section header that already names the root — no redundant disclosure
triangle for the root itself.

## 3C. Tree View Rendering — Known Failure Modes

Getting the file tree to render visibly has been the most persistent
implementation challenge in this project. This section is an
**engineering audit** of every failure mode that has been found, what
caused it, how it was diagnosed, and what the fix is. Future
contributors must read this before touching `DirectoryFilenamePanel.swift`
or `DirectoryTreeWalker`.

### 3C.1 Background: the failure pattern

The symptom is always the same: the file tree panel appears but shows
nothing (or shows one collapsed row that the user does not know to
click). The panel loads, the walker runs successfully, `directories.yaml`
is populated — but the UI control renders blank. This has led to
multiple rounds of "I thought it was fixed, but it wasn't" during
development.

The root cause in every instance has been one of three things:

1. `OutlineGroup` received the wrong starting node (single root instead
   of its children) — see §3C.3.
2. File path strings were constructed incorrectly, so rows rendered but
   selection/navigation was silently broken — see §3C.2.
3. The SwiftUI view checked the wrong state variable and showed a
   spinner or empty branch before the walk completed — see §3C.4.

### 3C.2 Bug: double-appended file paths (buildView)

**Symptom.** The tree renders directory structure and shows file rows,
but tapping/clicking a file does nothing. The file viewer does not load.
No error is logged.

**Root cause.** In `buildView`, the `.file` case constructed the path
as:

```swift
// BROKEN
let path = parentPath.appendingPathComponent(name).path
```

But the caller had **already** appended `name` to `parentPath` before
passing it into `buildView`. Every file's `path` string ended up as
`/Photos/img.jpg/img.jpg`. The `.tag` modifier and the selection binding
used this doubled path, so selection never matched any real file on
disk.

**Fix.**

```swift
// CORRECT — parentPath is already the full path of this node
let path = parentPath.path
```

**How to detect in the future.** If clicking a file in the tree does
nothing, add a temporary `print(path)` in the selection handler.
A doubled path segment (`/foo.jpg/foo.jpg`) is immediately visible.

**Test case.** Select any leaf file node in tree mode → the viewer
canvas must load the image within 300 ms. Run this on a directory with
at least two levels of nesting.

### 3C.3 Bug: tree showed one collapsed root row (walkerTreeView)

**Symptom.** The tree shows exactly one row: `▶ 📁 Photos`. Expanding
it reveals the files, but until the user clicks the disclosure triangle
the tree looks empty. Users do not click because they think "no photos
are loaded yet."

**Root cause.** `OutlineGroup` was called with the root `DirectoryNode`
as the single item:

```swift
// BROKEN — one collapsed root row
OutlineGroup(rootNode, children: \.children) { node in ... }
```

In a `.sidebar`-style `List`, this renders as a single disclosure row
for the root. The section header already shows the root path, making
the collapsed row redundant and confusing.

**Fix.** Extract the root's direct children and start `OutlineGroup`
one level below:

```swift
// CORRECT — children are the top-level rows
let topNodes = rootNode.children   // [DirectoryNode]
OutlineGroup(topNodes, children: \.children) { node in ... }
```

For a multi-root scope, the `ForEach + Section` pattern in §3B.5 is the
canonical form.

**Why this keeps recurring.** The SwiftUI documentation example for
`OutlineGroup` typically shows a single-item call where the item is a
folder. For `FileManager`-backed trees, that example maps naturally to
"pass the root," which is wrong. The fix is unintuitive because it
requires knowing that the `Section` header already carries the root
label.

**Test case.** Add one directory to the tree. Without clicking anything,
the tree must show that directory's immediate children as visible rows
(not one collapsed row). Assert `walkerRoots.first?.directChildren` is
non-empty and that those nodes appear rendered.

### 3C.4 Bug: blank tree during walk startup (treeView)

**Symptom.** On first launch (or after clearing all directories), the
tree shows blank for 2–5 seconds even though `directories.yaml` lists
valid roots. A spinner eventually appears, then the files load.

**Root cause.** The `treeView` property checked `walkerRoots.isEmpty`
and when true fell through to `legacyTreeView`. But `legacyTreeView`
also checked `resolvedFiles`, which was empty because the background
walk had not finished. The result: no spinner, no content, nothing.
The user saw a completely blank panel.

The condition that exposed this:
* `walkerRoots.isEmpty` — true (the background walk had not yet
  populated `walkerRoots`)
* `storedRootCount > 0` — true (roots exist in `directories.yaml`)
* `resolvedFiles.isEmpty` — true (walk not yet complete)

**Fix.** The `treeView` state machine now has an explicit "walk in
progress" branch:

```swift
var treeView: some View {
    if walkerRoots.isEmpty && storedRootCount > 0 {
        // Walk started but not yet delivered first results
        VStack(spacing: 12) {
            ProgressView()
            Text("Loading directory tree…")
                .foregroundStyle(.secondary)
        }
    } else if walkerRoots.isEmpty {
        emptyStateView          // no roots at all
    } else {
        walkerTreeView          // normal render path
    }
}
```

**Test case.** Quit the app with two valid roots in `directories.yaml`.
Relaunch. The tree must show a spinner + "Loading directory tree…" within
100 ms of the panel becoming visible, and must replace the spinner with
content within 5 s (on a local directory of up to 10 000 files).

### 3C.5 Minor fix: legacy tree single-node collapse

**Symptom.** `legacyTreeView` (the fallback code path) also showed one
collapsed `▶ /` row — the synthetic root node that `FileTreeNode.build`
wraps everything under.

**Root cause.** Same as §3C.3: `OutlineGroup(root, ...)` where `root` is
the synthetic "/" node.

Additionally, `.tag(node.fullPath ?? "")` mapped directory nodes to an
empty string `""`, so selecting a directory triggered selection of the
wrong item.

**Fix.**
* Use `OutlineGroup(topNodes, ...)` starting one level below "/".
* Use `.tag(node.fullPath as String?)` (the `Optional<String>` form) so
  directory nodes, which have no meaningful path to select, tag as `nil`
  and are not selectable.

### 3C.6 Diagnostic checklist

When the tree appears blank or broken, run through this list in order:

| # | Check | How to verify |
| --- | --- | --- |
| 1 | Is `directories.yaml` populated? | Read `~/Library/Application Support/ImageGlass_Mac/directories.yaml`. Must have ≥ 1 entry in `root_directories`. |
| 2 | Did the walk complete? | `walkerRoots` in the view model must be non-empty after ≤ 5 s for a local directory. Check `storedRootCount > 0 && walkerRoots.isEmpty` — if true, the spinner state (§3C.4) should be showing. |
| 3 | Is `OutlineGroup` receiving the root or its children? | Add `print(topNodes.count)` before the `OutlineGroup` call. Must be ≥ 1. If 0, the walk returned an empty children array — check the built-in file-kind filter (§3A.3). |
| 4 | Are file paths correct? | Print the `path` tag of any file node. Must not contain a doubled filename component (`img.jpg/img.jpg`). See §3C.2. |
| 5 | Is the `List` style `.sidebar`? | A `.inset` or `.plain` style suppresses disclosure triangles. The file tree panel must use `.listStyle(.sidebar)`. |
| 6 | Does clicking a file load the image? | If the tree renders but clicks do nothing, the tag/selection mismatch from §3C.2 is likely. |

### 3C.7 Acceptance bar for tree rendering

The tree view is considered working when **all** of the following are
true simultaneously, without the user taking any action:

1. Within 5 s of the panel becoming visible, at least one row is
   rendered for each root in `directories.yaml`.
2. Files appear as leaf rows without requiring the user to expand any
   disclosure triangle.
3. Clicking a file row loads that file in the viewer canvas within
   300 ms.
4. The correct file loads (path is not doubled, not truncated, not
   missing the extension).
5. On a fresh launch, a spinner is shown while the walk runs, and
   disappears when the first results arrive.

If any of these five conditions fails, the tree is not working — even
if it looks partially correct.

## 3D. How We Find and Link the Tree View Control

This section pins down exactly where the tree view control lives in the
source code, which Swift types are involved, and how the control connects
to the data model. It is the authoritative answer to "where does the
`OutlineGroup` actually live, and what feeds it?"

### 3D.1 The control used: List + OutlineGroup (SwiftUI)

The file tree panel uses option 3 from §3B: **`OutlineGroup` inside a
`List`**, pure SwiftUI throughout. There is no `NSOutlineView` and no
`NSBrowser` anywhere in the codebase. The backing AppKit control is an
`NSOutlineView` that SwiftUI creates and manages internally — it is never
referenced directly.

### 3D.2 Primary implementation: `walkerTreeView`

**File:** `code/Sources/ImageGlass/DirectoryFilenamePanel.swift`
**Lines:** approximately 224–251 (changes as the file evolves — use the
symbol names below to locate the current positions)

```swift
// walkerTreeView — the outer List that holds all roots
// Lines ~224–233
List(selection: $state.selectedFile) {
    ForEach(state.walkerRoots, id: \.path) { root in
        Section(header: rootHeader(root)) {
            walkerRootContent(root)   // ← calls OutlineGroup inside
        }
    }
}
.listStyle(.sidebar)

// walkerRootContent — the OutlineGroup for a single root
// Lines ~238–251
func walkerRootContent(_ root: WalkerRoot) -> some View {
    let views = rootChildViews(for: root)   // extracts root.directChildren
    return OutlineGroup(views, children: \.children) { node in
        row(for: node).tag(node.fullPath as String?)
    }
}
```

Key points:

* `walkerRootContent` calls `rootChildViews(for:)` — a helper that
  extracts the root's **direct children**, not the root node itself.
  This is the fix for Bug 2 (§3C.3): passing `views` (a collection)
  instead of `root` (a single node) prevents the single-collapsed-row
  anti-pattern.
* `.tag(node.fullPath as String?)` uses the `Optional<String>` form.
  Directory nodes have a `nil` `fullPath` and are therefore not
  selectable — only leaf file nodes carry a non-nil path tag that
  binds to `$state.selectedFile`.
* `.listStyle(.sidebar)` gives native macOS source-list appearance
  (translucent, grouped) and renders disclosure triangles automatically.

### 3D.3 Fallback: `legacyTreeView`

**Lines:** approximately 273–293

```swift
// legacyTreeView — used when walkerRoots is unavailable
// (e.g., file-tree walker not yet ready, or app still using legacy path)
List(selection: $state.selectedFile) {
    OutlineGroup(topNodes, children: \.children) { node in
        row(for: node).tag(node.fullPath as String?)
    }
}
.listStyle(.sidebar)
```

`topNodes` is `FileTreeNode.build(...)` stripped of its synthetic `/`
root, starting one level below. The fix for Minor Fix §3C.5 was to use
`topNodes` (a collection) rather than `root` (the single `/` node).

### 3D.4 The disclosure-triangle data contract

The children:` key path (`\.children`) on `NodeView` (or `FileTreeNode`)
is what drives the disclosure triangles:

```swift
struct NodeView: Identifiable {
    let id: String
    let name: String
    let fullPath: String?       // nil for directories
    let isDirectory: Bool
    let children: [NodeView]?   // nil = leaf (file); non-nil = expandable node (dir)
}
```

`OutlineGroup` renders a disclosure triangle **only** when `children` is
non-nil. A directory with `children: []` (empty array, not nil) still
renders a disclosure triangle — it just expands to nothing. The correct
representation for an empty directory is `children: []`, not `children:
nil` (which would render it as a non-expandable leaf, which is wrong).

### 3D.5 State machine that routes to the right view

`treeView` (the composite property that `body` calls) chooses between
`walkerTreeView`, spinner, and empty state:

```
walkerRoots.isEmpty && storedRootCount > 0  →  spinner + "Loading…"    (§3C.4)
walkerRoots.isEmpty && storedRootCount == 0 →  emptyStateView
otherwise                                   →  walkerTreeView            (§3D.2)
```

`storedRootCount` is the number of entries in `directories.yaml` at
startup — a value that is available immediately from disk before the
background walker has delivered its first results. This two-phase design
avoids the blank-panel problem: the spinner appears within < 1 ms of
panel layout, then is replaced by real content when the walk finishes.

## 3E. Debugging the Tree View UI

This section is the practical engineering guide for **inspecting the live
tree-view control at runtime** — from both inside and outside the Xcode
debugger. The goal: be able to observe exactly what UI is on screen, what
accessibility elements exist, and what SwiftUI thinks is in the view tree,
without relying on inference from source code alone.

Getting the tree to render has been difficult (§3C). Having real, working
diagnostic access to the live control is how we confirm it is actually
present, actually populated, and actually selectable — not just "probably
working."

### 3E.1 Layer model: what to inspect and where

The tree control spans three software layers. Different tools see different
layers.

| Layer | What you see | Best tools |
| --- | --- | --- |
| SwiftUI logical body | `List`, `OutlineGroup`, `Section`, `Label`, modifiers | Xcode View Hierarchy; `_viewDebugData()` SPI; ViewInspector |
| AppKit backing (`NSView`) | `NSHostingView > NSScrollView > NSClipView > NSOutlineView > NSTableRowView > NSTableCellView > NSHostingView (per cell)` | Xcode View Hierarchy; LLDB `recursiveDescription`; Chisel `pviews`; Reveal.app |
| Accessibility (AX) | `AXOutline > AXRow > AXText` (and other elements with accessibility identifiers) | Accessibility Inspector; `axcli`; `osascript`; XCUITest `debugDescription` |

For diagnosing the blank-tree bug: start at the AppKit layer to confirm
the `NSOutlineView` has rows. If it has rows but nothing renders, move to
the SwiftUI layer to check hosting views. If you need to test from
outside the process (automation, CI), use the AX layer.

### 3E.2 Xcode: Capture View Hierarchy (AppKit + SwiftUI)

**When to use:** the first thing to try. Gives a 3D exploded view of the
full `NSView` tree, pausing the app in place.

**How to trigger:**
1. Run the app in Xcode (Debug configuration).
2. Menu: `Debug > View Debugging > Capture View Hierarchy`
3. Or click the camera icon (⏱ icon) in the debug bar at the bottom of the
   Xcode window.

**What to look for in the tree:**
```
NSWindow
  NSVisualEffectView  (sidebar blur)
    NSHostingView                       ← SwiftUI root
      _UIHostingView or NSHostingView   ← inner hosting wrapper
        NSScrollView
          NSClipView
            NSOutlineView               ← THE CONTROL
              NSTableRowView            ← one per visible row
                NSTableCellView
                  NSHostingView         ← per-cell SwiftUI content
                    Label / Text
```

If `NSOutlineView` has **zero** `NSTableRowView` children, the data source
returned zero rows. Check `walkerRoots` and `rootChildViews` (§3D.2).

If `NSOutlineView` is **not present at all** in the hierarchy, the `List`
is not being rendered — check that `treeView` is selected (not
`emptyStateView` or the spinner) and that the panel is actually visible.

**Limitations:** Requires Debug build; app must be running in Xcode.

### 3E.3 Xcode Accessibility Inspector (AX layer)

**When to use:** verifying that the tree is accessible — that rows have
labels, that the outline is correctly identified, and that VoiceOver /
UI automation can find the elements.

**Launch:**
```bash
open "/Applications/Xcode.app/Contents/Applications/Accessibility Inspector.app"
```
Or: `Xcode > Open Developer Tool > Accessibility Inspector`

**Workflow:**
1. Start the app.
2. In Accessibility Inspector, select the app from the process picker
   (top-left).
3. Click the cursor icon (Inspection mode). Hover over the tree panel.
4. The inspector shows the element under the cursor: its role, label,
   value, frame, and children.

**What to look for:**
- The panel itself: `AXGroup` or `AXScrollArea`
- The outline: `AXOutline` (this is the `NSOutlineView`)
- Rows: `AXRow` — each file or directory node is one `AXRow`
- If the outline shows **AXOutline with 0 children**, the backing
  `NSOutlineView` has no rows — same conclusion as §3E.2 above
- If there are `AXRow` children but they are unlabeled ("AXRow (no label)"),
  add `.accessibilityLabel(node.name)` to the row view in `walkerRootContent`

**Limitations:** GUI-only; no batch export; requires Accessibility
permission granted to Xcode.

### 3E.4 LLDB — AppKit layer (in Xcode debugger)

These commands run in the LLDB console inside Xcode (the bottom-left
pane when paused or from the `Debug > Debugger Console` menu).

**Print the full NSView tree from the main window:**
```
(lldb) expression -l objc -O -- [[[NSApplication sharedApplication] mainWindow] contentView].recursiveDescription
```
This dumps every `NSView` subclass with its frame, `isHidden`, and
`wantsLayer` flags. The `NSOutlineView` and its child row views are all
visible here.

**In a Swift context (if you have a view variable):**
```
(lldb) po someView.value(forKey: "recursiveDescription")!
```

**Print SwiftUI state changes (inside a `body` breakpoint):**
```
(lldb) po Self._printChanges()
```
This shows which `@State` / `@Observable` property triggered the current
`body` evaluation. Use this when the tree is re-rendering unexpectedly
often (performance) or not re-rendering when it should (bug).

**Log SwiftUI changes to Console.app (Xcode 15+):**
```
(lldb) po Self._logChanges()
```
Output goes to `os_log` (subsystem `com.apple.SwiftUI`, category
`Changed Body Properties`). Filter in Console.app with
`subsystem:com.apple.SwiftUI`.

### 3E.5 Chisel — LLDB plugin (Homebrew)

Chisel is Facebook's set of LLDB extension commands. Its `pviews` command
is the fastest way to dump the NSView hierarchy without writing a
`recursiveDescription` expression.

**Install:**
```bash
brew install chisel
```

Add to `~/.lldbinit`:
```
command script import /opt/homebrew/opt/chisel/libexec/fbchisellldb.py
# Intel Mac: /usr/local/opt/chisel/libexec/fbchisellldb.py
```
Restart Xcode for the commands to load.

**Key commands in the LLDB console:**

| Command | What it does |
| --- | --- |
| `pviews` | Print the full NSView hierarchy for the key window |
| `presponder <view>` | Print the responder chain from a given view |
| `visualize <view>` | Render the view's bitmap and open it in Preview.app |
| `border --color red <view>` | Draw a live red border around the view on screen |
| `unborder <view>` | Remove the border |

**Most useful for the tree panel:**
```
(lldb) pviews
```
Find the `NSOutlineView` line in the output. Count its `NSTableRowView`
children — if zero, the `OutlineGroup` has no data.

**Limitation:** Chisel was written for iOS UIKit; the commands work on
macOS AppKit equivalents but some iOS-only commands will error. This is
fine — `pviews`, `presponder`, and `visualize` all work on macOS.

### 3E.6 SwiftUI-Introspect — access the live NSOutlineView from Swift

`SwiftUI-Introspect` is an SPM package that lets you reach through the
SwiftUI layer and grab the underlying `NSOutlineView` at runtime. Add it
to the Debug build only.

**Install (SPM, add to the app target under `#if DEBUG` guard):**
```swift
.package(url: "https://github.com/siteline/swiftui-introspect",
         from: "1.4.0")
```

**Usage inside `walkerTreeView` or a diagnostic view modifier:**
```swift
#if DEBUG
List(selection: $state.selectedFile) {
    // ... existing content ...
}
.listStyle(.sidebar)
.introspect(.list, on: .macOS(.v14, .v15)) { (outline: NSOutlineView) in
    print("Tree rows: \(outline.numberOfRows)")
    print("Is hidden: \(outline.isHidden)")
    print("Frame: \(outline.frame)")
    // Walk the rows:
    for row in 0 ..< outline.numberOfRows {
        let item = outline.item(atRow: row)
        print("  Row \(row): \(String(describing: item))")
    }
}
#endif
```

**What this tells you:**
- `outline.numberOfRows == 0` with `walkerRoots` non-empty → the `OutlineGroup`
  is receiving an empty collection, or `children: \.children` is returning `nil`
  for every node (meaning everything is treated as a leaf).
- `outline.isHidden == true` → something above the `List` in the view tree is
  hiding it — check the `treeView` state machine (§3D.5).
- `outline.numberOfRows > 0` but nothing is visible → the rows exist but are
  rendered off-screen, clipped, or with alpha = 0. Use `pviews` or Reveal to
  find the clipping ancestor.

Remove the introspect block before any release build — it is a debug-only
diagnostic, not part of the panel's production code.

### 3E.7 osascript — CLI AX tree dump (no Xcode required)

`osascript` is pre-installed on every Mac. It uses the same Accessibility
API as Accessibility Inspector and can run from any terminal.

**Setup (one-time):** Grant Accessibility permission to Terminal:
`System Settings > Privacy & Security > Accessibility > add Terminal.app`

**Dump top-level UI elements of the running app:**
```bash
osascript -e '
tell application "System Events"
  tell process "ImageGlass"
    get every UI element of window 1
  end tell
end tell
'
```

**Walk one level deeper — find the outline:**
```bash
osascript -e '
tell application "System Events"
  tell process "ImageGlass"
    set w to window 1
    repeat with elem in (every UI element of w)
      set kids to every UI element of elem
      log "role:" & (role of elem) & " subelements:" & (count of kids)
    end repeat
  end tell
end tell
'
```

**Count rows in the outline (AXOutline):**
```bash
osascript -e '
tell application "System Events"
  tell process "ImageGlass"
    -- Walk to find AXOutline
    set outline to first UI element of window 1 whose role is "AXOutline"
    set rowCount to count of (every UI element of outline whose role is "AXRow")
    log "AX rows: " & rowCount
  end tell
end tell
'
```

If `rowCount` is 0, the AX tree reports no rows — this aligns with
`NSOutlineView.numberOfRows == 0` from §3E.6. Both confirm the data
source has no items.

**Limitation:** AppleScript is serial and slow on deep trees. For large
hierarchies use `axcli` (§3E.8) instead.

### 3E.8 axcli — Fast CLI AX dump (Cargo-installed)

`axcli` is a Rust CLI tool that wraps the macOS Accessibility API. It
is faster than `osascript` for full-tree dumps and produces structured
text output.

**Install:**
```bash
# Requires Rust — install with:
curl https://sh.rustup.rs | sh
# Then:
cargo install axcli
```

**Grant Accessibility permission** to the terminal app (same as §3E.7).

**Dump the full AX tree of the running app:**
```bash
axcli snapshot --app "ImageGlass"
```

**Dump by PID (if you know it):**
```bash
axcli snapshot --pid $(pgrep -x ImageGlass)
```

**Filter to the outline element only:**
```bash
axcli snapshot --app "ImageGlass" --selector "[role=AXOutline]"
```

**List all running accessible apps:**
```bash
axcli list-apps
```

**Interpreting the output:** the snapshot is indented text. Look for the
`AXOutline` element and count its `AXRow` children. An `AXOutline` with
no `AXRow` children means the backing `NSOutlineView` has zero visible
rows.

**GitHub:** `https://github.com/andelf/axcli`

### 3E.9 XCUITest — programmatic AX dump for CI

In a `UITest` target, `XCUIApplication.debugDescription` prints the full
AX tree in one call. This is the machine-readable equivalent of
Accessibility Inspector, and it runs on any macOS test destination
without a display.

**Setup:** Add a UI test target to the Xcode project if one does not
exist (`File > New > Target > macOS UI Testing Bundle`).

**Dump the hierarchy in a test:**
```swift
import XCTest

final class TreeViewDiagnosticTests: XCTestCase {
    func testDumpTreeHierarchy() throws {
        let app = XCUIApplication()
        app.launch()
        // Add a directory via MCP or initial launch state here if needed
        print(app.debugDescription)  // full AX tree to the test log
    }

    func testOutlineHasRows() throws {
        let app = XCUIApplication()
        app.launch()
        let outline = app.outlineViews.firstMatch
        XCTAssert(outline.exists, "NSOutlineView must exist in the window")
        XCTAssertGreaterThan(outline.cells.count, 0, "Outline must have at least one row")
    }
}
```

**Run from command line:**
```bash
xcrun xcodebuild test \
  -project code/ImageGlass_Mac.xcodeproj \
  -scheme ImageGlass_Mac \
  -destination 'platform=macOS' \
  -only-testing "ImageGlass_MacUITests/TreeViewDiagnosticTests"
```

The `debugDescription` output goes to `xcresult` logs. View it in Xcode
(`⌘9 > Results`) or extract with:
```bash
xcrun xcresulttool get --path TestResults.xcresult --format json
```

**Automated accessibility audit (Xcode 15+):**
```swift
func testAccessibilityAudit() throws {
    let app = XCUIApplication()
    app.launch()
    try app.performAccessibilityAudit()  // fails on standard AX violations
}
```

### 3E.10 Reveal.app — 3D GUI inspector (paid)

Reveal is the industry-standard commercial view-hierarchy debugger. It
provides a 3D exploded view, live property editing, and a detailed

**Download:** `https://revealapp.com` — 14-day free trial, then paid

**Integration via LLDB breakpoint (zero source changes):**

Add a Symbolic Breakpoint in Xcode:

**For the tree panel specifically:** use Reveal to confirm that

### 3E.11 Recommended debugging workflow

Use this sequence when the tree appears blank or broken:

**Step 1 — Confirm data reaches the control (LLDB / Chisel)**
```
(lldb) pviews
```
Find `NSOutlineView` in the output. If it has zero `NSTableRowView`
children, the `OutlineGroup` has no data. Verify `walkerRoots` is
non-empty:
```
(lldb) po state.walkerRoots.count
(lldb) po state.walkerRoots.first?.directChildren.count
```

**Step 2 — Confirm the AX tree matches (osascript, no Xcode needed)**
```bash
axcli snapshot --app "ImageGlass" --selector "[role=AXOutline]"
```
Or use `osascript` from §3E.7. If `AXRow` count matches
`NSOutlineView.numberOfRows`, the two layers are in sync.

**Step 3 — Inspect the SwiftUI layer (Xcode View Hierarchy)**
Trigger `Debug > View Debugging > Capture View Hierarchy`. Locate the
`NSHostingView` that wraps the `List`. Confirm its `NSOutlineView`
child exists and has rows. If the `NSOutlineView` exists but is zero
height or zero width, the panel's layout constraints are wrong —
check that the panel's container gives the `List` a non-zero frame.

**Step 4 — Confirm paths are correct (breakpoint)**
Add a breakpoint in `walkerRootContent` on the `row(for:)` call.
Inspect `node.fullPath` for a file node — it must be a non-doubled,
accessible path. See §3C.2's diagnostic.

**Step 5 — CI regression test**
Add `testOutlineHasRows` from §3E.9 to the UI test target. Run it
in CI on every PR that touches `DirectoryFilenamePanel.swift` or
`DirectoryTreeWalker.swift`. A failing test surfaces the blank-tree
bug immediately instead of after a manual test cycle.

open or the app to be in a debug build. It:

1. Finds the running process by name
2. Walks the window hierarchy via the macOS Accessibility API
3. Locates the `AXOutline` (the tree control)
4. Dumps all visible row labels
5. Checks for the two key `JFK_Mar_c_files` rows
6. Exits 0 (pass) or 1 (fail) with a diagnostic message

**Prerequisites:**
- Grant Terminal Accessibility permission once:
  `System Settings > Privacy & Security > Accessibility > add Terminal.app`
- The app must be running with the `UX` root added to `directories.yaml`
- The tree panel must be in Tree view mode (not Strip / Grid / Details)

**Save as `scripts/verify_tree_jfk.sh`** in the project root and
`chmod +x` it:

```bash
#!/usr/bin/env bash
# verify_tree_jfk.sh
# Verifies that the JFK/UX directory tree is visible in the ImageGlass
# file tree panel by walking the live Accessibility hierarchy.
#
# Usage:  ./scripts/verify_tree_jfk.sh
# Exit:   0 = PASS   1 = FAIL

APP_NAME="ImageGlass"          # exact process name — adjust if bundle name differs

  echo "FAIL: '$APP_NAME' is not running. Launch the app first."
```

## 4. UI Mechanics

### 4.1 Panel chrome

The panel toolbar (top of the panel, ~36 pt high):

```
[Scope name ▾]  [⌕ filter]  [sort: Date ▾]  [▣ ░ ⊟ ┣ ⫯ view]  [↻ refresh]  [✎ edit scope]
```

* **Scope name** — popover with the list of saved scopes and an
  `[+ New scope…]` entry; clicking a name swaps the active scope.
* **Filter** — `TextField`; matches against filename and (in v1.1)
  date / size tokens.
* **Sort** — `Menu`; reuses `viewer.image_order` choices plus
  `dimensions`.
* **View toggle** — five buttons (or a `Picker`) for the five modes.
* **Refresh** — manual re-evaluation. Visually animated (spinning)
  while a re-walk is in progress.
* **Edit scope** — opens the `scope_editor` panel (which can be
  tabbed with this panel).

### 4.2 Item rendering

A `FileEntry` is rendered with:

```swift
struct FileEntry: Hashable, Sendable {
    var path: URL
    var size: Int64
    var modified: Date
    var created: Date
    var dimensions: CGSize?      // nil until resolved
    var exifDateTaken: Date?     // nil if not present or not yet resolved
    var typeIdentifier: UTType
    var isAnimated: Bool
    var isMultiFrame: Bool
    var isRAW: Bool
    var isHDR: Bool
    var isFavorite: Bool         // user-tagged via the panel
}
```

Visual elements:

* **Thumbnail**: 64 / 96 / 128 / 192 / 256 / 384 / 512 px square
  depending on `gallery.thumb_size` and the mode (strip uses one of
  the smaller; grid uses the configured size; details uses 24 pt;
  tree uses 18 pt).
  Generated via `CGImageSourceCreateThumbnailAtIndex` with
  `kCGImageSourceCreateThumbnailFromImageAlways = true` and
  `kCGImageSourceThumbnailMaxPixelSize = <target>`. Cached on disk
  under `~/Library/Caches/com.starbuck.imageglass/thumbnails/`
  keyed by source path + mtime + target size.

  ```swift
  func makeThumbnail(for url: URL, maxPixelSize: Int) throws -> CGImage? {
      let opts: [CFString: Any] = [
          kCGImageSourceCreateThumbnailFromImageAlways: true,
          kCGImageSourceThumbnailMaxPixelSize: maxPixelSize,
          kCGImageSourceCreateThumbnailWithTransform: true
      ]
      guard let src = CGImageSourceCreateWithURL(url as CFURL, nil) else { return nil }
      return CGImageSourceCreateThumbnailAtIndex(src, 0, opts as CFDictionary)
  }
  ```

* **Filename label** — truncated middle, system caption font.
* **Badges** — small SF Symbol indicators in the corners:

  | Badge | SF Symbol | Meaning |
  | --- | --- | --- |
  | Animated | `play.circle.fill` | Multi-frame animation (GIF / WebP / APNG) |
  | RAW | `camera.aperture` | Camera RAW |
  | HDR | `sun.max.circle.fill` | HDR JPEG / HEIC / AVIF |
  | Multi-frame | `square.stack.3d.up.fill` | Multi-page TIFF / multi-page PDF |
  | Favorite | `star.fill` | User-tagged |
  | Unread | `circle.fill` | New since last view (v1.1) |

  Badges are SF Symbols, not emoji, so they respect monochrome
  rendering, accessibility colors, and the user's "Increase contrast"
  setting.

* **Non-image files** — when the scope includes non-image files
  (PDF, text, video), the thumbnail is replaced by the file-type
  icon from `NSWorkspace.shared.icon(forFile:)`. The panel does not
  attempt to thumbnail unsupported formats; that prevents long
  stalls on huge files.

### 4.3 Selection model

* **Single-select** drives the viewer canvas: selecting any file
  whose `kind` is `.image`, `.svg`, or `.video` loads it into the
  viewer immediately. This is the canonical "click a thumbnail"
  behavior, and it applies to every selection mechanism — mouse
  click, keyboard arrow, MCP `select_file`, and the auto-select
  triggered by the walker (see
  [use_cases/mcp_file.mdx](../docs/use_cases/mcp_file.mdx) §2 and §10).
  Per-kind behavior on load:
  * `.image` — decoded via ImageIO and rendered fit-to-window on the
    `MTKView` canvas.
  * `.svg` — handed to the SVG preview surface ([svg.mdx](./svg.mdx))
    — `WKWebView` for animated content, Core Graphics static for the
    rest. The canvas swaps in the SVG view.
  * `.video` — handed to the video preview surface
    ([videos.mdx](./videos.mdx)) — AVKit `AVPlayerView` in the
    canvas. Playback does **not** start automatically on selection;
    the user presses Space (or the play button) to begin. Auto-play
    on selection would surprise the user with sound.
  Selection never silently fails: if the file is unreadable, the
  viewer renders its standard error placeholder and writes an
  `error.err` record with the file URL.
* **Multi-select** (Shift / Cmd-click) enables batch operations:
  * Delete (⌫) — moves to Trash via `FileManager.trashItem(at:resultingItemURL:)`.
  * Copy paths (⌘C in plain selection mode) — writes paths to the
    pasteboard.
  * Copy image data (⌥⌘C) — copies the first selection's image
    bytes.
  * Move / Copy to folder — opens a chooser.
  * Export… — passes to the export flow.
  * Batch crop — opens the crop tool with `tools.crop.persistent_selection = true` ([crop](./crop.mdx) §6.2).

### 4.4 Drag and drop

* **Drag out** — `NSItemProvider` file-promise items. The user can
  drag a selection into Finder, into another app, or into a
  multi-file-accepting upload field. Promises mean the file is not
  copied until the destination accepts.
* **Drag in** — accepts file URLs from Finder / other apps. Drop on
  the panel adds the URL(s) as a new source criterion to the active
  scope (with the user's confirmation if the scope is a saved
  named scope). Drop on an empty panel creates a new implicit
  scope.

## 5. Sort and Filter

### 5.1 Sort modes

The sort modes match `viewer.image_order` plus `dimensions`:

| Sort | Implementation |
| --- | --- |
| `name` | Natural sort (locale-aware, numeric-aware). Falls back to ASCII when filenames are not text. |
| `size` | `URLResourceKey.fileSize`. |
| `modified` | `URLResourceKey.contentModificationDate`. |
| `created` | `URLResourceKey.creationDate`. |
| `exif_date_taken` | EXIF `DateTimeOriginal` from `CGImageSourceCopyPropertiesAtIndex`. Files without EXIF sort after files with EXIF (configurable). |
| `extension` | UTI's preferred extension. |
| `random` | Stable random with the scope name as the seed (so the order is reproducible across launches until the scope is re-evaluated). |
| `dimensions` | Composite of width × height. Files without resolved dimensions are loaded asynchronously and re-inserted when ready. |

Direction is `asc` or `desc`. Headers in Details mode show the
direction with a `↑ / ↓` indicator.

### 5.2 Filter

The filter field is a free-form text input plus, in v1.1, a small
token language:

| Token | Effect |
| --- | --- |
| `*.jpg` | Filename glob. |
| `>5mb` | File size greater than 5 MB. |
| `wider:3000` | Dimensions width ≥ 3000 px. |
| `taller:2000` | Dimensions height ≥ 2000 px. |
| `after:2026-05-01` | Modified after this date. |
| `before:2026-06-01` | Modified before this date. |
| `kind:raw` | UTI conforms to `public.camera-raw-image`. |
| `kind:hdr` | EXIF / metadata flags HDR. |
| `fav:` | Favorite-tagged. |

Bare text (no `:` or `>`) is treated as a substring match against
the filename. Tokens compose with implicit AND.

## 6. Performance

### 6.1 Lazy decode

Only visible items have thumbnails decoded. Off-screen items use a
placeholder (a faint gray rectangle plus the file-type icon).
Visibility is tracked via the SwiftUI `onAppear` / `onDisappear`
hooks for grid/strip and via `NSCollectionView`'s prefetch APIs
for the rare cases where `LazyVGrid` is replaced by
`NSCollectionViewRepresentable`.

### 6.2 Bounded concurrency

Thumbnail decoding runs in a `ThumbnailCache` actor with a
`AsyncSemaphore` bounded by `ProcessInfo.processInfo.activeProcessorCount`:

```swift
actor ThumbnailCache {
    private let semaphore = AsyncSemaphore(value: ProcessInfo.processInfo.activeProcessorCount)
    private var memoryLRU: LRUCache<ThumbKey, CGImage> = .init(maxBytes: 256 * 1024 * 1024)
    private let diskRoot: URL

    func thumbnail(for entry: FileEntry, size: Int) async -> CGImage? {
        let key = ThumbKey(path: entry.path, mtime: entry.modified, size: size)
        if let hit = memoryLRU[key] { return hit }
        if let disk = await readDisk(key: key) {
            memoryLRU[key] = disk
            return disk
        }
        await semaphore.acquire()
        defer { Task { await semaphore.release() } }
        guard let img = try? makeThumbnail(for: entry.path, maxPixelSize: size) else { return nil }
        memoryLRU[key] = img
        Task.detached(priority: .utility) { await self.writeDisk(key: key, image: img) }
        return img
    }
}
```

### 6.3 LIFO queue

When the user scrolls fast, the queue of pending thumbnails
prioritizes the **most recently requested** items so the visible
viewport always wins. Scrolled-past items are dropped from the
queue without being decoded.

### 6.4 LRU eviction

* **Memory LRU** — capped by total bytes (256 MB default).
* **Disk LRU** — capped by total bytes (`gallery.disk_cache_mb`,
  default 400 MB; absolute cap `advanced.thumb_cache_mb`, default
  1024 MB). Eviction runs on a background `NSBackgroundActivityScheduler`
  job every 6 hours.

### 6.5 Paged scope walking

Scopes with > 50 000 files:

1. The walker yields the first 5 000 entries as soon as they are
   enumerated; the panel renders them while the walker continues.
2. Subsequent batches of 5 000 are appended; the panel uses a
   diff-friendly `IdentifiedArray` so SwiftUI does not re-render the
   whole list.
3. The resolved file list is streamed to disk in chunks, not held
   entirely in memory.
4. The panel toolbar shows a thin progress bar across the bottom
   of the toolbar until the walk completes.

### 6.6 FSEvent debounce

`FSEventStream` events are debounced 250 ms before being dispatched
to the panel. Bulk file operations (e.g., a `cp -R` of 500 files)
collapse to a single re-evaluation rather than 500.

### 6.7 Memory budget

Targets, measured on an M3 Pro:

| Scope size | Idle memory | Active-scroll memory |
| --- | --- | --- |
| 100 files | ~40 MB | ~80 MB |
| 5 000 files | ~80 MB | ~200 MB |
| 50 000 files | ~150 MB | ~400 MB |
| 250 000 files | ~300 MB | ~600 MB |

Above 250 000 files the panel shifts to `NSCollectionView` for the
grid mode automatically.

### 7.2 ScopeWalker

```swift
actor ScopeWalker {
    func walk(_ scope: Scope) -> AsyncThrowingStream<FileEntry, Error> {
        AsyncThrowingStream { continuation in
            Task.detached(priority: .utility) {
                let keys: [URLResourceKey] = [.isDirectoryKey, .fileSizeKey,
                                              .contentModificationDateKey,
                                              .creationDateKey,
                                              .typeIdentifierKey]
                for criterion in scope.criteria {
                    guard let enumerator = FileManager.default.enumerator(
                        at: criterion.root,
                        includingPropertiesForKeys: keys,
                        options: criterion.includeHidden ? [] : [.skipsHiddenFiles]
                    ) else { continue }

                    for case let url as URL in enumerator {
                        if Task.isCancelled { continuation.finish(); return }
                        if matches(url, criterion: criterion) {
                            continuation.yield(makeEntry(url, keys: keys))
                        }
                    }
                }
                continuation.finish()
            }
        }
    }
}
```


The panel's SwiftUI view binds to this `@Observable` view model.
External clients (the MCP server, the viewer canvas selection
synchronizer) call methods on it; all mutations are main-actor
isolated.

## 8. MCP Integration

The MCP server exposes these tools. All read tools are R-only; all
write tools mutate scope JSON via atomic-rename writes and
notify the panel via `FSEventStream`.

| Tool | What it does |
| --- | --- |
| `list_scopes` | Return every scope name + summary. |
| `get_active_scope` | Return the name of the currently active scope (or `null` for implicit). |
| `set_active_scope(name)` | Set the active scope. |
| `get_scope(name)` | Return the full scope JSON. |
| `create_scope(name, criteria, sort?, filter?, description?)` | Create a new scope file. |
| `update_scope(name, patch)` | Patch fields of an existing scope. |
| `delete_scope(name)` | Delete a scope. Refuses to delete the active scope; caller must `set_active_scope` first. |
| `evaluate_scope(name)` | Force re-walk. Returns the resolved count and the wall-clock duration. |
| `list_files_in_scope(name, offset?, limit?)` | Return the resolved file list, paginated. |
| `select_file(path)` | Set the panel's selection to this single file (also loads it in the viewer). |
| `select_files(paths)` | Set the panel's selection to a list. |
| `set_view_mode(mode)` | Switch the panel between strip / grid / details / tree / column. |
| `set_sort(by, direction)` | Set the sort. |
| `set_filter(text)` | Set the filter text. |

Each tool's full JSON-RPC schema lives in
`Source/MCPSchemas/file_list/*.json` and is registered with the
MCP server at panel registration time.

### 8.1 Push events

The MCP server also pushes events that clients can subscribe to:

* `scope_evaluated` — payload: scope name, file count, elapsed_ms.
* `scope_changed` — payload: scope name; emitted on any field change.
* `selection_changed` — payload: list of paths.
* `view_mode_changed` — payload: new mode.

Clients use these to keep their own UI / state in sync without
polling.

### 8.2 Example — Claude Code narrowing a scope

```
User to Claude Code:
  "From this scope, just show me the HEIC files smaller than 5 MB."

Claude Code calls:
  get_active_scope() -> "2026-shoots"
  update_scope("2026-shoots", {
    "criteria": [
      { "root": "/Users/bryan/Pictures/2026-shoots",
        "recursive": true,
        "include_exts": ["heic"] }
    ],
    "filter": { "max_size": 5242880 }
  })
  // ScopeStore writes the file; FSEventStream fires; the panel
  // detects, re-walks, and renders the narrowed list within ~300 ms.
```

## 9. Workflow Scenarios

### 9.1 Preview.app refugee (T1)

User installs the app, opens `~/Desktop/photo.heic` with Cmd-O.
* Implicit scope created from `~/Desktop`.
* Panel docks at the bottom in `strip` mode (Settings default).
* User scrolls horizontally; arrow keys advance through photos.
* No file panel work required from the user.

### 9.2 Photographer with multi-source scope (T2a)

* Photographer creates scope `2026-shoots` via Settings (`[+ New
  scope…]` from the scope picker), pointing at one local and one
  external-drive directory.
* Sets extension filter to `nef, cr3, dng`.
* Picks sort = EXIF Date Taken (descending).
* Switches the panel to `grid` mode, size L.
* When the external drive is unplugged, the panel shows the local
  files only; reconnecting the drive triggers a 250 ms-debounced
  re-walk that re-includes them.

### 9.3 Claude Code rotates scopes (T4)

* User says "use my review queue."
* Claude Code calls `set_active_scope("review-queue")`.
* Panel switches; viewer loads the first file.
* User finishes review; tells Claude Code "now switch to last
  week's exports."
* Claude Code: `set_active_scope("exports-2026-w22")`.
* Zero GUI clicks.

### 9.4 Designer drag-out to Sketch (T2b)

* Designer in Details mode, sorted by Modified descending.
* Multi-selects three icon PNGs (Cmd-click).
* Drags the selection out of the panel onto a Sketch artboard.
* Sketch accepts the file promises; the files are not duplicated
  until Sketch reads them.

### 9.5 Tree-mode disambiguation (T3)

* User has two folders named `IMG_0001.jpg` in different source
  directories.
* In `details` mode the rows look identical.
* User switches to `tree` mode; each file is visibly under its
  source directory.
* User right-clicks the wrong one → Move to Trash.

## 10. Open Questions

* **Vision-backed search (v2).** "Find photos that look like this"
  via on-device Vision embeddings is a natural addition once the
  thumbnail cache is in place. Out of scope for v1.
* **Per-file rating UI.** Should ratings live in the file panel or
  in the metadata panel? Tentative: metadata. The file panel renders
  the rating as a small badge but does not allow editing.
* **Auto-discovered scopes.** Should the app surface a "Recents"
  scope or a "Photos library" scope automatically? Tentative: no
  automatic scopes in v1 — but the Quick Setup flow ([Settings §2.7](./settings.mdx#27-quick-setup-first-run))
  could offer to create them.
* **NAS / S3 / cloud scopes.** Scopes today assume local filesystem
  (or mounted volumes). A future "Cloud" scope kind would walk
  remote storage. Out of scope for v1; spec'd here for awareness.

## 11. Acceptance Criteria

The file panel ships when all of the following pass:

* **Five modes load correctly** for a 5 000-file scope with mixed
  formats (JPEG, HEIC, PNG, DNG, MP4, PDF, TXT).
* **Live updates**: copying 500 files into a watched directory
  triggers exactly one re-walk within 500 ms.
* **Thumbnail throughput**: ≥ 200 thumbnails / second on M3 Pro for
  cached items, ≥ 50 / second for first-time generation.
* **Memory cap**: a 250 000-file scope holds idle at ≤ 350 MB.
* **MCP round-trip**: `create_scope` → `evaluate_scope` →
  `set_active_scope` → `list_files_in_scope` returns the expected
  list and the GUI reflects every change within 500 ms.
* **Sandbox**: a sandboxed build with a security-scoped bookmark
  correctly enumerates a user-granted directory across an app
  relaunch.
* **Crash safety**: kill the process mid-scope-write 100 times; on
  relaunch every scope is either the prior valid state or its
  `.bak` and we recover.
* **Accessibility**: VoiceOver announces each item by filename, size,
  and modified date; the view-mode toggle is keyboard-traversable.
