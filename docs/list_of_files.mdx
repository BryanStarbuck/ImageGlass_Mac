# File List Panel

This page is the **complete product + engineering specification** for
the **file list panel** — ImageGlass_Mac's column, plus the
scope engine that feeds it.

This is goal #2 from [Automation](./automation.mdx)
(modular UI panels) combined with #3 (scope controls) and #4 (Local
Storage). It is also the largest single panel built on top of the
[panels framework](./panels.mdx).

It cross-references:

* [Automation](./automation.mdx) §2–§5 — the panel framework, scope controls, and Local Storage.
* [Panels](./panels.mdx) — the host framework the file panel lives
  in (movable / dockable / floating).
* [Settings → Gallery](./settings.mdx#410-gallery) — the small set of
  Settings that govern defaults.
* [Crop](./crop.mdx) — when batch cropping, the file panel drives
  the iteration.

The Mac stack used throughout: **Swift 6 + SwiftUI** (`Table`,
`List`, `OutlineGroup` for tree mode, `LazyVGrid` for the grid /
strip), **AppKit** (`NSCollectionView` for very large grids only;
`NSImageView` thumbnails) where SwiftUI hits performance limits,
**ImageIO** (`CGImageSourceCreateThumbnailAtIndex`) for fast
thumbnails, `FileManager.enumerator(at:includingPropertiesForKeys:options:)`
for the scope walker, **FSEventStream** for live file-change
notifications, and **`actor`** isolation for the caches and stores.

## 1. Audience and Goals

### 1.1 Audiences

| Tier | Audience | What they want from the file panel |
| --- | --- | --- |
| T1 | Casual viewer | A thumbnail strip at the bottom, like Preview.app. Click a thumbnail; the image loads. Nothing else. |
| T2a | Photographer | A grid of large thumbnails, sort by EXIF DateTimeOriginal, filter by extension, fast keyboard navigation. |
| T2b | Designer cataloging assets | A details list with Name / Size / Dimensions / Modified columns; resizable, sortable, reorderable. |
| T3 | Power user | Tree mode showing the directory hierarchy of a multi-directory scope; explicit scope editor next to it. |
| T4 | Claude Code / MCP | The panel binds to a scope. Claude Code edits the scope's JSON via MCP and the panel updates live; selecting an image in the panel loads it in the viewer. |

### 1.2 Goals

* **One panel, five rendering modes.** Strip, grid, details, tree,
  column — switchable from the panel toolbar. The data underneath is
  always the same resolved file list.
* **Scopes, not folders.** The panel never just "shows the contents
  of one folder." It always binds to a scope. A scope can be
  implicitly created from a single folder (the casual case) or
  explicitly defined as a multi-directory rule set.
* **Live.** When the underlying filesystem changes (files added,
  removed, renamed, modified), the panel reflects it within a
  debounced 250 ms.
* **Fast.** Thumbnails are generated on demand, cached on disk,
  paged for very large scopes (>50 000 files). The panel is usable
  on a NAS-backed scope of 250 000 RAW files.
* **Programmable.** Scopes are plain-text JSON on disk. The MCP
  server reads and writes them so Claude Code can swap, narrow, or
  rotate the active set without GUI input.
* **Native.** Drag from the panel to Finder via file promises. Drag
  from Finder into the panel to extend the scope. Space-bar Quick
  Look. SF Symbol badges. NSWorkspace icons for non-image files.

### 1.3 Non-goals

* No image editing in the panel. The panel is selection and
  navigation only; edits happen on the viewer canvas.
* No content-based search ("find me photos that look like this") in
  v1. The filter is filename + extension + date + dimensions only.
  Vision-backed search is a v2 candidate but is out of scope for
  this spec.
* No metadata editing in the panel (rotations, ratings, captions);
  those live in the dedicated Metadata panel.
* No two-way sync with Finder's sort order; macOS does not expose a
  general API for it. The panel has its own sort model.

## 3. The Scope

A **scope** is the source of truth for "what files are in this
panel." Scopes are described in [automation.mdx](./automation.mdx) §3–§4;
this section pins down the schema, semantics, and lifecycle.

### 3.1 Scope JSON

A scope is one JSON file under
`~/Library/Application Support/ImageGlass_Mac/scopes/<name>.json`.

Schema:

```json
{
  "$schema": "https://json-schema.org/draft/2020-12/schema",
  "title": "ImageGlass_Mac scope",
  "type": "object",
  "required": ["name", "schema_version", "criteria"],
  "properties": {
    "name":           { "type": "string", "pattern": "^[A-Za-z0-9._-]+$" },
    "schema_version": { "type": "integer", "const": 1 },
    "description":    { "type": "string" },
    "criteria": {
      "type": "array",
      "items": {
        "type": "object",
        "required": ["root"],
        "properties": {
          "root":           { "type": "string", "description": "Absolute path. Sandboxed builds require a saved security-scoped bookmark." },
          "bookmark":       { "type": "string", "description": "Base64-encoded NSURL bookmark data. Required in sandboxed builds." },
          "recursive":      { "type": "boolean", "default": true },
          "max_depth":      { "type": ["integer", "null"], "default": null },
          "include_globs":  { "type": "array", "items": { "type": "string" } },
          "exclude_globs":  { "type": "array", "items": { "type": "string" } },
          "include_exts":   { "type": "array", "items": { "type": "string" } },
          "exclude_exts":   { "type": "array", "items": { "type": "string" } },
          "include_hidden": { "type": "boolean", "default": false },
          "follow_symlinks":{ "type": "boolean", "default": false }
        }
      }
    },
    "sort": {
      "type": "object",
      "properties": {
        "by":        { "enum": ["name", "size", "modified", "created", "exif_date_taken", "extension", "random", "dimensions"] },
        "direction": { "enum": ["asc", "desc"] }
      }
    },
    "filter": {
      "type": "object",
      "properties": {
        "text":      { "type": "string" },
        "date_from": { "type": "string", "format": "date-time" },
        "date_to":   { "type": "string", "format": "date-time" },
        "min_width": { "type": "integer" },
        "min_height":{ "type": "integer" },
        "max_size":  { "type": "integer" }
      }
    },
    "last_evaluated": { "type": "string", "format": "date-time" },
    "resolved": {
      "type": "array",
      "items": {
        "type": "object",
        "required": ["path", "size", "modified"],
        "properties": {
          "path":     { "type": "string" },
          "size":     { "type": "integer" },
          "modified": { "type": "string", "format": "date-time" },
          "dim":      { "type": "array", "items": { "type": "integer" }, "minItems": 2, "maxItems": 2 }
        }
      }
    }
  }
}
```

The schema is the contract between the GUI, the MCP server, and any
external tool. Every required field above has a meaning; the
optional fields are progressive enhancements.

### 3.2 Worked example — "2026-shoots"

```json
{
  "name": "2026-shoots",
  "schema_version": 1,
  "description": "Active photo shoots from 2026, across local and external drives.",
  "criteria": [
    {
      "root": "/Users/bryan/Pictures/2026-shoots",
      "recursive": true,
      "include_exts": ["nef", "cr3", "dng", "jpg", "jpeg", "heic"],
      "exclude_globs": ["**/_archive/**", "**/.imageglass/**"]
    },
    {
      "root": "/Volumes/PhotoArchive/2026/exports",
      "recursive": false,
      "include_exts": ["jpg", "jpeg"]
    }
  ],
  "sort": { "by": "exif_date_taken", "direction": "desc" },
  "filter": { "min_width": 3000 },
  "last_evaluated": "2026-06-02T10:18:42Z",
  "resolved": [
    { "path": "/Users/bryan/Pictures/2026-shoots/Wedding/ceremony/001.dng",
      "size": 50331648, "modified": "2026-05-12T14:22:01Z", "dim": [8688, 5792] }
    // … 824 more entries
  ]
}
```

### 3.3 Staleness and re-evaluation

A scope is **stale** when any of:

* `last_evaluated` is older than the most-recent mtime of any
  source directory.
* An `FSEventStream` watching a source directory has signaled a
  change since `last_evaluated`.
* The user clicked **Refresh** in the panel toolbar.
* The user explicitly invoked `evaluate_scope` via MCP.

A stale scope is re-walked by `ScopeWalker` on a background task; the
panel shows a progress indicator on its toolbar refresh icon and
updates the file list as the walk completes. Subsequent reads use
the freshly-resolved list.

### 3.4 Sandbox + security-scoped bookmarks

When the app is sandboxed (App Sandbox enabled at build time), the
walker cannot enumerate a directory unless the app holds a
security-scoped bookmark for it. The scope JSON stores the bookmark
data as a base64 blob (`criteria[].bookmark`). The walker calls
`URL.startAccessingSecurityScopedResource()` before enumerating and
`stopAccessingSecurityScopedResource()` after.

The user grants the bookmark by picking the directory in an
`NSOpenPanel`, which produces a resolvable bookmark. If the user
later moves the source directory, the bookmark survives (it tracks
the directory by inode/volumeUUID, not by path).

### 3.5 Implicit scope (Cmd-O)

When the user opens a single file or folder with Cmd-O (or the
"Open With" Finder action), the panel creates an **implicit scope**
without writing it to disk:

* Single file → scope = "all images in the containing folder",
  active = the opened file.
* Single folder → scope = "all images in this folder, recursive=true,
  no extension filter".

The implicit scope lives only in memory and is replaced as soon as
the user opens another file. The user can promote an implicit scope
to a named scope with the panel toolbar's `[Save scope as…]` action.

## 3A. The File Tree Panel — `directories.yaml` and Live Walker

This section pins down the **file tree** rendering mode (§2.4) and
its dedicated storage file. The file tree is the only rendering mode
that shows the source directory hierarchy directly, so it is also
the mode that the MCP `add_directory` / `remove_directory` /
`update_directory_filter` calls drive most visibly.

Walked end-to-end as a user scenario in
[use_cases/mcp_file.mdx](./use_cases/mcp_file.mdx). That tour is the
**user-facing acceptance** for this section — every behavior
specified here must match the tour's verify steps.

### 3A.1 `directories.yaml`

The file tree panel's canonical storage is a single YAML file:

```
~/Library/Application Support/ImageGlass_Mac/directories.yaml
```

It maps **one-to-one** with what the file tree UI renders. Each
top-level row in the tree is one entry in `root_directories:`. Each
file shown underneath a row is a leaf in the recursive walk of that
root that passed the built-in file-kind filter *and* the root's
per-root filter.

The file is the source of truth for the panel. There is **no**
hidden binary state — the GUI and the MCP server read and write the
same file. Writes are atomic (temp file + rename); the panel reloads
via `FSEventStream` on the file itself.

Relationship to §3's `scopes/<name>.json`:

* `directories.yaml` is the file tree panel's **own** store. It is
  not a scope file. It exists independently of the multi-scope
  feature.
* When a multi-scope user is using a named scope, the file tree
  panel can be bound to it via the scope picker — at that point the
  panel renders the active scope's `criteria[]` and ignores
  `directories.yaml`. The default (and the case the MCP file tour
  covers) is the standalone `directories.yaml`.
* The data model in `directories.yaml` is a deliberate **subset** of
  the scope schema — only the fields the file tree panel needs.

### 3A.2 Schema

```yaml
schema_version: 1
root_directories:
  - path: /Users/bryan/Pictures/tour/beach
    filter:
      match: any           # 'any' = OR (default), 'all' = AND
      items:
        - pattern: "*.jpg"
          kind: glob       # 'glob' (default), 'substring', 'regex'
          negate: false    # true = exclude matches (default false = include)
        - pattern: "*.heic"
    last_walked: 2026-06-02T10:18:42Z
  - path: /Users/bryan/Pictures/tour/mountains
    filter:
      items: []            # empty = no per-root filter; everything that passes
                           # the built-in image/SVG/video filter is included
```

Field rules:

| Field | Required | Notes |
| --- | --- | --- |
| `schema_version` | yes | Integer. Currently `1`. |
| `root_directories` | yes | Array. Empty array is valid — see §3A.10. |
| `root_directories[].path` | yes | Absolute path. Must be canonicalized (no `~`, no `..`). The MCP server expands and canonicalizes before writing. |
| `root_directories[].filter` | no | If omitted, equivalent to `{ match: any, items: [] }` — i.e. include everything that passes the built-in file-kind filter. |
| `filter.match` | no | `any` (OR) is default. `all` (AND) is the alternative. |
| `filter.items[].pattern` | yes (per item) | The string the matcher tests against the filename (not the full path) by default. |
| `filter.items[].kind` | no | `glob` (default, uses `fnmatch(3)` semantics with `*`, `?`, `[abc]`), `substring` (case-insensitive substring match against the filename), `regex` (POSIX-extended). |
| `filter.items[].negate` | no | `false` by default. When `true`, a match means "exclude this file." |
| `last_walked` | written by walker | ISO-8601 instant. Set by the background walker on every completed walk. |

### 3A.3 Built-in file-kind filter

Independent of any per-root filter, only files of these kinds are
ever considered:

* **Images** — every UTI conforming to `public.image` (JPEG, PNG,
  HEIC, HEIF, WebP, AVIF, JXL, TIFF, GIF, BMP, RAW family, etc.).
* **SVG** — `public.svg-image`.
* **Video** — every UTI conforming to `public.movie` (MP4, MOV, M4V,
  HEVC, AVI, MKV — exact set bounded by what AVFoundation can decode
  on the running OS).

Everything else (`.txt`, `.pdf`, `.zip`, `.sketch`, …) is dropped at
the walker stage and is never inserted into the in-memory tree, so
it can never appear in the panel or be re-considered when a filter
changes. The built-in filter is not configurable from
`directories.yaml`; it is the panel's hard-wired contract.

### 3A.4 Root directory — in-memory data structure

```swift
struct RootDirectory: Sendable {
    let path: URL                       // canonical absolute path
    var filter: RootFilter              // per-root filter (§3A.2)
    var tree: DirectoryNode             // the recursive in-memory mirror
    var lastWalked: Date?
    var watcher: FileSystemWatcher?     // recursive FSEventStream
}

indirect enum DirectoryNode: Sendable {
    case directory(name: String,
                   children: [DirectoryNode])
    case file(name: String,
              uti: UTType,
              kind: FileKind,           // .image | .svg | .video
              passesFilter: Bool)       // result of applying RootFilter
}

struct RootFilter: Sendable {
    enum Match: Sendable { case any, all }
    enum ItemKind: Sendable { case glob, substring, regex }
    struct Item: Sendable {
        let pattern: String
        let kind: ItemKind
        let negate: Bool
    }
    var match: Match
    var items: [Item]
}
```

Key invariants:

* **The tree is recursive and complete** — every directory under
  the root that the walker visited is represented, including
  directories that contain no matching files. This is what the
  Outline view renders.
* **Only `.image`, `.svg`, and `.video` files appear in `.file`
  nodes.** Other files are dropped at walk time and never enter the
  tree at all.
* **`passesFilter`** is the cached result of evaluating `RootFilter`
  against the file's name. It is updated by `recomputePassesFilter`
  whenever the filter changes (§3A.7), without re-touching the
  filesystem.
* The tree is built once per walk and mutated in place by
  `FSEventStream` callbacks (§3A.6). It is never rebuilt from
  scratch unless the root directory itself becomes inaccessible.

### 3A.5 Background walker thread

Each file tree panel owns one background actor — the
**file-tree walker** — that runs for the lifetime of the panel,
**even when the panel is hidden or collapsed**. The walker is the
only thing that touches the filesystem on the panel's behalf.

```swift
actor DirectoryTreeWalker {
    private var roots: [URL: RootDirectory] = [:]
    private var sleeping: Bool = true        // see §3A.5.2

    func addRoot(_ path: URL, filter: RootFilter) async { /* §3A.5.1 */ }
    func removeRoot(_ path: URL) async { /* tear down watcher, drop from roots */ }
    func updateFilter(for path: URL, filter: RootFilter) async { /* §3A.7 */ }

    func snapshot() -> [RootDirectory] { Array(roots.values) }
}
```

The walker exists per-panel, but its work is `Task.detached(priority:
.utility)` so it does not contend with the UI. Mutations to the
in-memory `RootDirectory` graph are actor-isolated. The
@MainActor view model reads via `snapshot()` and renders the
SwiftUI `OutlineGroup` from that snapshot.

#### 3A.5.1 Adding a root

When `addRoot(path, filter)` is called (from a menu action, a drag,
or an MCP `add_directory`):

1. The walker enumerates `path` recursively with
   `FileManager.enumerator(at:includingPropertiesForKeys:options:)`,
   filtering at the UTI gate to **image / SVG / video only**.
2. As entries arrive, the walker builds the `DirectoryNode` tree
   and computes `passesFilter` for each `.file` node using
   `RootFilter`.
3. When the walk completes, the walker registers an
   **`FSEventStream` recursive watch** on `path` so subsequent
   add/remove/rename events under the hierarchy call back into the
   walker (§3A.6).
4. The walker sets `lastWalked = .now`, writes the new state into
   `directories.yaml` (atomic), and posts a `rootChanged` event so
   the panel re-renders.

The MCP call returns as soon as step 1 begins; the walk is async,
identical to the behavior covered in
[mcp_file.mdx](./use_cases/mcp_file.mdx) §10.

#### 3A.5.2 Sleep / wake

The walker is mostly idle. To reduce wake-ups on a battery:

* When all FS watchers are quiet for `idleTrigger = 30 s`, the
  walker marks itself `sleeping = true`. The actor task suspends on
  its message queue (no work; no CPU).
* Any of these wake it: an `FSEventStream` callback fires, a panel
  action arrives (menu / drag / MCP), or a periodic
  `NSBackgroundActivityScheduler` heartbeat (every 6 h) requests a
  reconciliation pass.
* Wake-up cost is < 1 ms; the walker does not re-walk the filesystem
  on wake — it just resumes processing whatever message woke it.

This sleep/wake behavior is the user-promised contract that the file
tree panel is "running even when the component is turned off."

### 3A.6 Recursive FS watching per root

Every `RootDirectory` carries one `FileSystemWatcher` wrapping an
`FSEventStream` started with `kFSEventStreamCreateFlagFileEvents`
and `kFSEventStreamCreateFlagWatchRoot`. The stream is created with
the root path and watches the entire hierarchy under it.

On a callback:

* For each changed path, the walker classifies the event:
  `.created`, `.removed`, `.renamed`, `.modified`.
* The walker updates the in-memory `DirectoryNode` tree in place:
  new `.image / .svg / .video` files are inserted with
  `passesFilter` computed; removed files are dropped; renames are
  modeled as remove+create.
* Events are debounced 250 ms (matches §6.6). A bulk `cp -R` of 5000
  files collapses into a single SwiftUI render pass.
* Non-image / non-SVG / non-video events are discarded immediately.

The watcher survives the panel being hidden — that is the point of
§3A.5's "runs even when the component is turned off." Closing the
panel does **not** tear the watcher down; only `removeRoot` does.

### 3A.7 Filter re-evaluation — no re-walk

When the user (or MCP) changes the filter for a root via
`updateFilter`, the walker **does not touch the filesystem**. It
walks the in-memory `DirectoryNode` tree and recomputes
`passesFilter` on every `.file` node:

```swift
func recomputePassesFilter(_ node: inout DirectoryNode, with filter: RootFilter) {
    switch node {
    case .directory(let name, var children):
        for i in children.indices { recomputePassesFilter(&children[i], with: filter) }
        node = .directory(name: name, children: children)
    case .file(let name, let uti, let kind, _):
        node = .file(name: name, uti: uti, kind: kind,
                     passesFilter: filter.evaluate(filename: name))
    }
}
```

This is what makes filter changes feel instant on a 250 000-file
scope — no I/O, just an in-memory traversal. The panel re-renders
only the rows whose `passesFilter` flipped.

`RootFilter.evaluate(filename:)`:

* If `items` is empty, returns `true` (no per-root filtering).
* Partition `items` into **positive** items (`negate: false`, the
  default) and **negative** items (`negate: true`).
* If there are **no positive items**, the positive result is `true`
  — i.e. every file that passed the built-in file-kind filter is
  in scope, and negative items will carve out from that set. This
  is the special case in
  [use_cases/mcp_file.mdx §7.0](./use_cases/mcp_file.mdx#70-mental-model--positive-vs-negative-filter-items)
  that makes "show everything *except* X" expressible as a single
  negate item.
* Otherwise, evaluate each positive item against the filename
  (`glob`, `substring`, or `regex` per the item's `kind`) and
  combine the results by `match`: `any` (default) = logical OR;
  `all` = logical AND. Call the combined boolean
  `positives_passed`.
* Evaluate each negative item against the filename. If **any**
  negative item matches, the file is excluded — regardless of
  `match` and regardless of `positives_passed`. This is the
  **veto semantic** spelled out in
  [use_cases/mcp_file.mdx §7.0](./use_cases/mcp_file.mdx#70-mental-model--positive-vs-negative-filter-items).
* Final result:

  ```
  included = positives_passed AND NOT any_negative_matched
  ```

The use-case → filter-item cookbook in
[use_cases/mcp_file.mdx §7.0.1](./use_cases/mcp_file.mdx#701-use-case--filter-item-cookbook)
is the contract that this engine must satisfy: every cookbook row
is a working call and a unit-test case.

## 3B. Debugging the File Tree Control from the Command Line

This section catalogs the command-line tools used to inspect the
**running** ImageGlass_Mac process — its windows, the file tree
panel's `NSOutlineView` / SwiftUI `OutlineGroup` hierarchy, and the
per-row properties (frame, role, value, children, accessibility
attributes). It exists because reproducing a tree-control bug almost
always requires reading the live state of the control without
attaching Xcode and without modifying the running build.

The short answer up front: **there is no single `brew install <tool>`
that does all of this for a native AppKit / SwiftUI app.** The
upstream ImageGlass project on Windows has WinAppDriver / Inspect.exe
to lean on; macOS has the Accessibility API plus the Objective-C
runtime, and the practical CLI workflow is a small set of cooperating
tools — none of them a Homebrew formula by themselves.

### 3B.1 What we want from a debugger-like CLI

| Capability | Why it matters for the file tree panel |
| --- | --- |
| Attach to a running process by PID or bundle ID | The bug is in the live `directories.yaml`-driven tree (§3A); we cannot reproduce it by reading code. |
| Enumerate the process's windows | Confirm the panel is in the expected `NSWindow`; rule out a stale floating panel ([panels.mdx](./panels.mdx)). |
| Recursively walk the native UI control hierarchy | The whole point — see every `OutlineGroup` row, its parent root, its children. |
| Extract all properties per control / element | Role, title, value, frame, `passesFilter` (when bridged), AX children, identifier. |
| Print to stdout (no GUI) | Pipe into `grep`, `jq`, snapshot diffs, CI scripts. |

No Homebrew formula covers all five. The Apple-shipped GUI tool that
*does* is **Accessibility Inspector** (ships inside Xcode → Open
Developer Tool → Accessibility Inspector). It is not CLI and not
brew-installable; it is the reference for what the CLI tools below
approximate.

### 3B.2 Tool matrix — closest options

| Tool | Install | Attach by | Walks native UI? | Output | Verdict for the tree control |
| --- | --- | --- | --- | --- | --- |
| `macapptree` | `pip install macapptree` (Python; **not** brew) | Bundle ID, `--all-apps` | Yes — full AX tree | JSON file (or stdout via redirect) | **Best read-only option.** Mirrors the panel hierarchy via the Accessibility API; structured JSON is easy to diff. |
| Frida | `pip install frida-tools` (**not** a brew formula) | PID or process name | Yes — via Obj-C runtime + AppKit | stdout (script-controlled) | Best when AX is insufficient; reaches private internals like `_subtreeDescription`. Requires a short JS script. |
| lldb | `xcode-select --install` (Xcode CLT; **not** brew) | PID | Yes — via Obj-C `po` expressions | stdout | Always available on a dev Mac. Clunky for batch dumps but unbeatable for "what is this view *right now*." |
| Accessibility Inspector | Ships in Xcode | GUI pick | Yes — full AX tree | GUI | Not CLI; useful as the **reference** for what AX exposes. |
| `xctree` | `brew install xctree` exists | iOS Simulator processes only | Yes | stdout / JSON | **Does not apply** to a native macOS app process. Listed only to prevent confusion — the name implies otherwise. |
| Appium + Mac driver | `brew install node` + `npm i -g appium` | WebDriver session | Yes | JSON over WebDriver | Overkill for a snapshot dump; appropriate for full UI automation, not for one-off inspection. |
| Chromium `ax_dump_tree` | Source build | Chromium processes only | Yes | stdout | **Does not apply** to AppKit / SwiftUI apps. |

The verdict, condensed: read-only snapshot → **macapptree**; runtime
inspection of internals → **Frida**; ad-hoc debugger sessions →
**lldb**.

### 3B.3 macapptree — the read-only AX snapshot

`macapptree` walks the macOS Accessibility tree for an app (by bundle
ID) and emits structured JSON. The tree it produces is the same tree
the file tree panel renders, because both the panel and AX read from
the same `NSAccessibility` protocol implementation.

Install (no brew formula; this is a Python package):

```sh
python3 -m pip install macapptree
```

Snapshot the running ImageGlass_Mac and dump to stdout:

```sh
python3 -m macapptree.main \
    -a com.starbuck.imageglass \
    --oa /dev/stdout \
    | jq '.children[] | select(.role == "AXOutline")'
```

The `jq` filter narrows to the `AXOutline` element, which is the
`NSOutlineView` backing the SwiftUI `OutlineGroup` in §2.4. Each
child is one tree row; properties include `AXRole`, `AXTitle`,
`AXValue`, `AXFrame`, `AXChildren`, and `AXIdentifier`.

Caveats:

* Targets by **bundle ID**, not PID. If multiple instances are
  running, pick the one whose AX tree contains the expected root via
  `jq`.
* Reads only what the AX API exposes. Per-row internal state like
  `passesFilter` (§3A.4) is **not** in AX unless we explicitly bridge
  it via `accessibilityValue` or a custom AX attribute.
* Requires the user to grant the running Python interpreter
  Accessibility permission once (System Settings → Privacy & Security
  → Accessibility).

### 3B.4 Frida — runtime introspection of internals

When the AX tree is not enough — typically when a tree row's behavior
depends on `passesFilter`, `RootFilter.evaluate`, or another property
not exposed via AX — Frida attaches to the live process and reads the
Objective-C runtime directly.

Install:

```sh
python3 -m pip install frida-tools
```

Attach and dump the main window's view hierarchy:

```sh
frida -p "$(pgrep -f ImageGlass_Mac)" -e '
  const NSApp = ObjC.classes.NSApplication.sharedApplication();
  const windows = NSApp.windows();
  for (let i = 0; i < windows.count(); i++) {
      const w = windows.objectAtIndex_(i);
      console.log(`Window: ${w.title()}`);
      console.log(w.contentView()._subtreeDescription().toString());
  }
'
```

`_subtreeDescription` is a private `NSView` method (the same one
Xcode's "Debug View Hierarchy" calls under the hood) that prints
every `NSView` with its frame, autoresizing mask, and class. For the
outline view specifically:

```sh
frida -p "$PID" -e '
  // find every NSOutlineView in the process
  ObjC.choose(ObjC.classes.NSOutlineView, {
      onMatch(o) {
          console.log(`AXOutline: ${o.numberOfRows()} rows`);
          for (let r = 0; r < o.numberOfRows(); r++) {
              const item = o.itemAtRow_(r);
              console.log(`  row ${r}: ${item}`);
          }
      },
      onComplete() {}
  });
'
```

Notes:

* Frida attaches **as a debugger** — the running process must allow
  it. Disable hardened-runtime debugger protections in the
  `Debug`-configuration build, or pass `--allow-debugger` flags when
  signing. Production / notarized builds will refuse the attach.
* Frida sees private state but does not respect Swift-only APIs;
  pure-Swift structs (`DirectoryNode`, `RootFilter`, see §3A.4) are
  not directly enumerable from the Obj-C runtime. To inspect Swift
  values from a CLI, bridge them through an Obj-C-visible
  property on the view model (e.g., a `@objc dynamic var
  debugDescription: String`).

### 3B.5 lldb — debugger sessions when AX and Frida fall short

`lldb` ships with the Xcode Command Line Tools and is always
available on a dev Mac. For one-off questions about the tree
control's live state:

```sh
lldb -p "$(pgrep -f ImageGlass_Mac)"

(lldb) po [[NSApplication sharedApplication] windows]
(lldb) po [[[NSApplication sharedApplication] mainWindow] contentView]
(lldb) po [(NSView *)0x600003af0000 _subtreeDescription]
(lldb) expression -l swift -- import ImageGlass_Mac
(lldb) expression -l swift -- print(FileTreeViewModel.shared.snapshot())
```

The Swift `expression -l swift` form is the only way to read pure-Swift
state (`FileTreeViewModel.snapshot()` from §3A.5) without bridging.
It requires the running binary to have its full DWARF debug
information present — i.e., a `Debug` build, not the stripped Release
binary.

### 3B.6 Workflow — picking the right tool

| Symptom | Start with |
| --- | --- |
| "The panel is missing a row I expect to see" | `macapptree` snapshot → `jq` for the path → confirm AX has it. If AX is missing it, the bug is in §3A.5's walker, not in the renderer. |
| "The panel shows a row but it looks wrong (wrong filter result, wrong indent)" | Frida `_subtreeDescription` → confirm the `NSView` hierarchy matches expectations. Then bridge `passesFilter` to AX or read it via lldb Swift expressions. |
| "I want a CI regression check that the tree renders as expected" | `macapptree --oa` JSON → diff against a snapshotted baseline. This is the only option that produces stable, diffable output. |
| "I need to step through `RootFilter.evaluate`" | `lldb -p` + Swift breakpoints. Frida cannot break on pure-Swift code reliably. |

### 3B.7 What we are *not* doing

* **No bundled GUI inspector.** We are not shipping a custom AX
  inspector inside ImageGlass_Mac; Accessibility Inspector already
  exists and the engineering team uses it during development.
* **No `brew tap` for a custom CLI.** We considered packaging a small
  Swift CLI (`igmac-inspect`) that wraps the AX API and prints the
  tree to stdout. It is out of scope for v1 — `macapptree` covers the
  same ground without us shipping a separate binary. Revisit if
  `macapptree` proves insufficient.
* **No AX bridging of every internal field by default.** Bridging
  `passesFilter`, `kind`, and other §3A.4 fields into AX makes the
  panel observable to *every* AX client, including screen readers,
  with unclear UX consequences. Bridge selectively, behind a
  `DEBUG`-only build flag, when an investigation needs it.

## 3C. Floating File Tree Window

This section pins down a **detached, floating window** that hosts the
file list and file tree — independent of the main viewer window. The
goal is twofold: give the user a persistent navigator that stays put
even when the main window is moved, resized, or full-screened, and
give Claude Code / MCP a stable surface for driving file selection
without competing for chrome in the main viewer.

### 3C.1 Behavior

* **Detached.** The window is a separate `NSWindow` owned by a shared
  `FloatingFileTreeWindowController`. It is not a child window, not a
  sheet, and not a `WindowGroup` member of the main `App` scene. It
  has its own title bar, traffic lights, minimize button, and resize
  grip.
* **Default on.** The window opens automatically on every cold launch
  after `applicationDidFinishLaunching`. The user can dismiss it (red
  traffic light or the menu item toggle); the next launch re-opens it.
  Persisting "user explicitly closed it" across launches is out of
  scope for v1 — default-on is the contract.
* **Independent lifecycle.** Closing the main viewer window does
  **not** close the floating window. Closing the floating window does
  **not** quit the app or close any viewer window.
  `applicationShouldTerminateAfterLastWindowClosed` continues to fire
  only when the user closes the *last* window of any kind.
* **Title.** *File Tree* (no app prefix; the system already shows the
  app name in the menu bar). The dock menu lists it as `File Tree`.
* **Content.** A `FileTreeFloatingView` that renders the same
  `RootDirectory` graph the in-window `DirectoryFilenamePanel`
  renders (§3A.4 + the panel's `walkerTreeView`) — i.e. the file list
  on top and the recursive file tree below, both driven by
  `state.walkerRoots`. Selection in the floating window is the same
  `AppState.selectedFile` binding the main viewer uses, so clicking
  in the floating window loads the image in the main viewer canvas
  (§4.3).
* **Toggle without selection side effects.** Opening or closing the
  floating window must not change `selectedFile`, `panelLayout`,
  `panelViewMode`, or `walkerRoots`. The window is a *render* of those
  values; toggling its visibility is purely a window-management
  action.

### 3C.2 Menu placement and item name

The floating window's toggle lives in the **View** menu — the same
menu that already owns *Show Panel Column*. Adding it next to its
sibling makes the relationship obvious: *Show Panel Column* toggles
the **inline** file panel docked inside the main window, *Show
Floating File Tree* toggles the **detached** floating file tree
window. They are independent toggles and can be on or off in any
combination.

| Menu | Item | Shortcut | Toggle behavior |
| --- | --- | --- | --- |
| View | Show Floating File Tree | ⌃⌥⌘F | When the window is open, the item reads *Hide Floating File Tree*. Selecting it closes the window. When closed, the item reads *Show Floating File Tree*; selecting it opens (or brings forward) the window. |

The hotkey is `⌃⌥⌘F` (Control-Option-Command-F). It avoids the
already-bound `⌃⌘F` (View ▸ Enter Full Screen) and `⇧⌘F` (View ▸
Frameless Window) in [menus.mdx](./menus.mdx) §4.4 and the
`⌥⌘F` (Directories ▸ Edit Filter… in §3A.8). The triple-modifier
form is appropriate for a window-management toggle the user rarely
rebinds.

The item is a single SwiftUI `Button` whose label flips between *Show
Floating File Tree* and *Hide Floating File Tree* depending on
`FloatingFileTreeWindowController.shared.isVisible`. We do **not**
use a `Toggle` because SwiftUI's `Toggle` in a menu renders a
checkmark, which is a weaker affordance than a verb that flips —
"Show" / "Hide" reads as an action while a checkmark reads as a
status (HIG §Menus).

### 3C.3 Background color — bright yellow

The file tree's background is **bright yellow** (`#FFFF00`,
`Color(red: 1.0, green: 1.0, blue: 0.0)`). The yellow fills the
content rect of the floating window edge-to-edge, behind the file
list rows and the tree rows, behind the row hover highlights, and
behind the selection accent overlay (which sits on top of the
yellow with `Color.accentColor.opacity(0.18)` per the existing tree
row rendering in `DirectoryFilenamePanel.walkerTreeView`).

Yellow is **only** applied inside the floating window. The inline
file panel (`DirectoryFilenamePanel` docked in the main window) keeps
its `Color(NSColor.windowBackgroundColor)` so the inline path stays
on theme. This makes the floating window visually distinguishable at
a glance, which matters when both renderings of the same data are
open at once.

Implementation contract: the yellow is applied as a `.background`
modifier on the root `VStack` of `FileTreeFloatingView`, not on the
window's `backgroundColor`. Two reasons:

1. The window's `backgroundColor` only paints the area not covered
   by the content view; setting it does not guarantee the tree
   rows themselves render on yellow.
2. The `.background` modifier composes with the SwiftUI `List` /
   `OutlineGroup` styling — the rows render their default chrome on
   top of the yellow without us fighting `NSTableView` defaults.

The yellow is not configurable in v1. The bright-yellow color is the
spec contract.

### 3C.4 Wiring

The window is owned by a single `MainActor`-isolated controller
mirroring the existing `AboutWindowController` and
`ReleasesWindowController` patterns:

```swift
@MainActor
final class FloatingFileTreeWindowController {
    static let shared = FloatingFileTreeWindowController()

    private var window: NSWindow?

    var isVisible: Bool { window?.isVisible == true }

    func show(state: AppState) { /* create-or-foreground, see §3C.5 */ }
    func hide()               { window?.orderOut(nil) }
    func toggle(state: AppState) {
        if isVisible { hide() } else { show(state: state) }
    }
}
```

The View menu item calls `toggle(state:)`. The
`AboutAppDelegate.applicationDidFinishLaunching` opens the window
once via `show(state:)` so the default-on contract from §3C.1 is met.

### 3C.5 Window construction

* **Style mask:** `[.titled, .closable, .miniaturizable, .resizable]`.
* **Initial size:** 360 × 600 (portrait — a navigator shape).
* **Min size:** 240 × 320.
* **Position:** centered on the screen the cursor is on at first
  launch; on subsequent toggles, restored to its last position via
  `NSWindow.setFrameAutosaveName("ImageGlass.FloatingFileTree")`.
* **Floating level?** No. The window uses `NSWindow.Level.normal`. A
  true floating-above-all level would steal focus from full-screen
  viewer mode and surprise the user; it is documented as out of
  scope.
* **Release on close:** false (`isReleasedWhenClosed = false`) so the
  controller can re-show the same window without rebuilding it.
* **Restorable:** false. State restoration would conflict with the
  default-on contract (a closed-by-user window would re-open as
  closed on relaunch, contradicting §3C.1's "next launch re-opens
  it").

### 3C.6 Content composition

```
┌───────────────────────────────── File Tree ─┐  ← title bar
│ ⌕ filter            ↻                       │  ← compact header
├─────────────────────────────────────────────┤
│   ▣ 001.dng                                 │
│   ▣ 002.dng                                 │  ← file list (top half)
│   ▣ 003.dng                                 │
├─────────────────────────────────────────────┤
│ ▼ /Users/bryan/Pictures/tour/beach          │
│     001.jpg                                 │  ← file tree
│     002.jpg                                 │     (bottom half)
│ ▼ /Users/bryan/Pictures/tour/mountains      │
│     peak.heic                               │
└─────────────────────────────────────────────┘
   ← every row sits on the bright-yellow background →
```

A vertical split: file list on top (flat union of every visible file,
matching `DirectoryFilenamePanel.flattenVisible`), file tree on
bottom (one `Section` per root with an `OutlineGroup`, matching
`DirectoryFilenamePanel.walkerTreeView`). The split is a SwiftUI
`VSplitView` (AppKit-backed) with a 50/50 default ratio that the user
can drag. Both halves bind to `state.selectedFile`, so a click in
either half loads the image in the main viewer canvas (§4.3
"selection drives the viewer canvas").

### 3C.7 Empty state

When `directories.yaml` has no roots **and** the active scope has no
resolved files, the floating window renders the same
`emptyState` view that `DirectoryFilenamePanel` uses (§3A.10
fallback) — an `[+ Add directory]` button that opens an `NSOpenPanel`
folder picker and routes through `DirectoriesStore.shared.addRoot` +
`DirectoryTreeWalker.shared.scheduleWalk`. The yellow background
remains visible behind the empty-state SF Symbol + label.

### 3C.8 MCP

The floating window is **observable** through MCP via a single tool:

| Tool | Effect |
| --- | --- |
| `set_floating_file_tree_visible(visible)` | When `true`, calls `FloatingFileTreeWindowController.shared.show(state:)`. When `false`, calls `hide()`. Returns the new `isVisible`. |

No additional MCP tools are needed: every file-list / file-tree
read/write tool already in §3A.10 and §8 operates on the same
underlying `directories.yaml` and `AppState`, so the floating window
re-renders without a tool dedicated to it.

### 3C.9 Acceptance criteria (in addition to §11)

The floating window ships when **all** of the following pass — these
are the spec-compliance proofs the implementation must satisfy:

1. **Default-on.** Launching the app with no prior state opens the
   floating window automatically.
2. **Detached.** The floating window has its own title bar and
   traffic lights, is reported as a separate `NSWindow` in
   `NSApp.windows`, and can be moved independently of the main
   viewer window.
3. **Menu toggle exists.** A View-menu item named *Show Floating
   File Tree* (or *Hide Floating File Tree* when open) is present
   and bound to `⌃⌥⌘F`.
4. **Toggle works in both directions.** With the window open, the
   menu item closes it. With the window closed, the menu item opens
   it (creating a fresh window or bringing the cached one forward).
5. **Yellow background.** The content rect of the floating window
   renders the file list and file tree on `#FFFF00`. The yellow is
   visible behind both populated rows and the empty-state view.
6. **Content parity.** The floating window's tree renders the same
   `state.walkerRoots` data the inline `DirectoryFilenamePanel`
   renders; mutations via MCP (`add_directory`,
   `update_directory_filter`, etc.) update the floating window
   within one debounced FS-events tick (§6.6 = 250 ms).
7. **Selection round-trip.** Clicking a file row in the floating
   window updates `state.selectedFile`, which loads the file in the
   main viewer canvas. Selecting a different file via the main
   viewer's `→` / `←` keys updates the highlighted row in the
   floating window.
8. **No close-on-quit dependency.** Closing the floating window with
   its red traffic light does not quit the app while the main viewer
   window is still open.

These criteria are individually verifiable by inspection or by a
test that drives the menu item + checks `NSApp.windows` + checks
the rendered SwiftUI hierarchy.

## 3D. Selectable Tree View

This section pins down a **user-selectable rendering technology** for
the file tree (§2.4 and §3A.4). The file tree's *data* is fixed —
the `RootDirectory` graph the walker maintains in memory (§3A.4). The
*renderer* that turns that data into on-screen rows is a runtime
choice the user makes from the menu bar. Switching renderers swaps
the SwiftUI view that hosts the tree without re-walking the
filesystem and without resetting the user's selection or scroll
position.

The three rendering paths Apple provides for **native** macOS UI
(where the OS draws and manages the controls, as opposed to
Electron/web-based or fully custom-drawn views) are catalogued in
§3D.1. All three produce a tree the user can browse with the same
keyboard, mouse, and accessibility behavior — only the underlying
framework differs.

### 3D.1 The three rendering technologies

| Renderer | Framework | Macro-control | Suited to |
| --- | --- | --- | --- |
| **AppKit** | AppKit + Objective-C-bridged Swift, hosted in SwiftUI via `NSViewRepresentable` | `NSOutlineView` inside `NSScrollView`, driven by `NSOutlineViewDataSource` / `NSOutlineViewDelegate`. Disclosure triangles, drag-and-drop, sidebar styling are all native AppKit behavior. | The classic, battle-tested macOS path. Finder uses `NSOutlineView` in its list-view sidebar. Closest match to native macOS feel; most direct control over drag-and-drop and context menus. |
| **SwiftUI** | SwiftUI declarative views | `OutlineGroup` inside a `List`, or a `List` with the `children:` keypath modifier. Hierarchy, expansion/collapse, and disclosure rendered automatically. | The modern, declarative path. Least code; tightest integration with SwiftUI state and `@Observable`. Apple's preferred framework for new macOS apps. |
| **Catalyst-styled** | AppKit + SwiftUI rendering UIKit-flavored chrome (see §3D.7 — this is **not** a true Catalyst target build; ImageGlass_Mac is and remains an AppKit/SwiftUI Mac-native app) | Custom `List` styled to mimic `UITableView` with `UICollectionView` compositional-layout-style outline rows — flat indented rows, iOS-style disclosure chevrons, iOS row insets, no AppKit disclosure triangle. | A visual alternative for users who prefer the iPadOS / Catalyst look. Useful side-by-side comparison reference. Stylistic only — no functional gain over AppKit. |

The fourth option — fully custom-drawn (Core Graphics, no native
control) — is explicitly excluded; it would not feel like a
"first-class macOS app" per the project goal.

### 3D.2 The View ▸ Tree View submenu

The renderer choice is exposed through a new submenu on the View
menu in the menu bar:

```
View
  Show Panel Column                    ⌃⌥P
  Show Floating File Tree              ⌃⌥⌘F
  ─────
  Tree View ▸
    ✓ AppKit  (NSOutlineView)           ⌃⌥⌘1
      SwiftUI (OutlineGroup)            ⌃⌥⌘2
      Catalyst (UIKit-styled)           ⌃⌥⌘3
```

Item behavior:

* **Exactly one item is checkmarked** at any time — the currently
  active renderer. The checkmark is rendered with the standard
  `NSMenuItem.state = .on` glyph (`✓`).
* Selecting an unchecked item:
  1. Sets it as the active renderer in the shared `TreeRenderTechnology`
     setting (§3D.4).
  2. Moves the checkmark to the newly selected item.
  3. Triggers a SwiftUI view-tree swap in both the inline file panel
     (`DirectoryFilenamePanel`) and the floating file tree window
     (`FileTreeFloatingView` — §3C). The walker / data layer is
     untouched.
* Selecting the already-checkmarked item is a no-op.
* Each item has a hotkey: `⌃⌥⌘1` (AppKit), `⌃⌥⌘2` (SwiftUI),
  `⌃⌥⌘3` (Catalyst). The triple-modifier form keeps the renderer
  toggle out of the way of viewer / panel shortcuts.

Submenu placement — under the View menu, right below the
*Show Floating File Tree* item (§3C.2). This co-locates every
file-tree-presentation control in one menu.

### 3D.3 Default renderer

The factory default is **SwiftUI**. It is the lowest-friction path
for a fresh install: no `NSViewRepresentable` bridging, the
shortest code path, and the framework Apple is investing in. The
default is overridable both by the user (via the submenu) and by
Claude Code (via the MCP tool in §3D.6).

If `UserDefaults` has never recorded a choice (first launch), the
app initializes the setting to `.swiftUI` and writes it back so
subsequent launches read a stable value.

### 3D.4 Setting model

The renderer choice lives in a single source-of-truth setting:

```swift
enum TreeRenderTechnology: String, CaseIterable, Codable, Sendable {
    case appKit    = "appkit"      // NSOutlineView via NSViewRepresentable
    case swiftUI   = "swiftui"     // OutlineGroup inside List
    case catalyst  = "catalyst"    // UIKit-styled rendering in AppKit/SwiftUI

    var displayName: String {
        switch self {
        case .appKit:   return "AppKit"
        case .swiftUI:  return "SwiftUI"
        case .catalyst: return "Catalyst"
        }
    }
}
```

The setting is held on `AppState` as a `@MainActor`-isolated
property and mirrored to `UserDefaults` under the key
`ig.tree_render_tech`. Persistence is automatic — every write to
the property flushes to `UserDefaults` synchronously, the same
pattern `panelLayout` and `panelViewMode` use today.

### 3D.5 Renderer contract

Every renderer is a SwiftUI `View` that takes the same input and
emits the same callbacks. The interface is:

```swift
protocol FileTreeRenderer: View {
    init(state: AppState)
    // Implementations read state.walkerRoots for the data,
    // bind to state.selectedFile for selection, and update
    // state.selectedFile on click. They do not own scroll
    // position; SwiftUI's List/ScrollView state handles that.
}
```

The single switch point is in `DirectoryFilenamePanel.walkerTreeView`
(and `FileTreeFloatingView`'s equivalent half), which picks the
concrete renderer:

```swift
switch state.treeRenderTechnology {
case .appKit:   AppKitOutlineTreeView(state: state)
case .swiftUI:  SwiftUIOutlineTreeView(state: state)   // existing FileListTreeView refactored
case .catalyst: CatalystStyledTreeView(state: state)
}
```

Switching the enum re-renders the host view; the inactive
renderers' state (expanded-folder set, AppKit datasource cache) is
dropped, by design — a fresh renderer always starts from the
walker's current `walkerRoots` snapshot. The user's `selectedFile`
survives because it lives on `AppState`, not in the renderer.

### 3D.6 MCP tool

A single read/write tool exposes the setting:

| Tool | Effect |
| --- | --- |
| `get_tree_render_tech()` | Returns `"appkit"` / `"swiftui"` / `"catalyst"`. |
| `set_tree_render_tech(value)` | Sets the active renderer. Accepts the same three strings. Invalid values return `invalid_value`. Has the same effect as selecting the menu item — the checkmark moves and the view swaps. |

The tool is journaled to `log.log` per
[error_handling.mdx](./error_handling.mdx) §4.2 like every other
write tool.

### 3D.7 The Catalyst renderer — implementation honesty

ImageGlass_Mac is a Mac-native AppKit/SwiftUI app, not a Mac
Catalyst app. Mac Catalyst is a *whole-app* build target — the
entire process must be UIKit-based — so a real Catalyst `UITableView`
or `UICollectionView` cannot be embedded inside this app's
`NSWindow`. The "Catalyst" renderer in §3D.1 is therefore a
**UIKit-styled** renderer: AppKit/SwiftUI primitives configured to
match the Catalyst look. Specifically:

* No `NSOutlineView` disclosure triangle; the row leads with an
  iOS-style chevron (`chevron.right` rotated 90° when expanded) at
  the iPad chevron size.
* Row insets, font (`SF Pro Text` in `.body` size), and row height
  (44 pt) match Catalyst defaults.
* Selection background is iOS-style: a tinted system fill that
  spans the row width with rounded corners on the outside, not the
  AppKit accent-color slab.
* No native AppKit drag handle; drag-and-drop is wired via SwiftUI
  `.onDrag` / `.onDrop` modifiers (which work but feel more
  iPad-like than Finder-like).

This is intentional and called out in the submenu's tooltip:
*"Catalyst — UIKit-styled rendering. Visual alternative to AppKit
and SwiftUI; no functional difference."*

A future major version may add a true Catalyst build target as a
separate distribution; if it does, this renderer becomes the
native renderer for that build, and the submenu collapses to a
single available option per build target.

### 3D.8 Accessibility, keyboard, and selection parity

All three renderers must behave identically from the user's
perspective on the dimensions below. The contract is:

| Dimension | All three renderers |
| --- | --- |
| Keyboard navigation | ↑ / ↓ moves selection one row; → expands; ← collapses; Return loads the selection into the viewer canvas. |
| VoiceOver | Each row announces filename, kind (directory vs image / SVG / video), and depth. Disclosure state is announced as "expanded" / "collapsed". |
| Selection model | Single-select drives `state.selectedFile`. Multi-select is **out of scope** for the file tree in v1 — multi-select lives in the grid and details modes (§4.3). |
| Expand-all / Collapse-all | ⌥-click on a disclosure triangle (or its renderer-equivalent) expands / collapses the entire subtree. |
| Click target | Whole row is clickable, not just the chevron. The chevron and the label are visually distinct but share the row's click area. |
| Right-click menu | Context menu contents (Reveal in Finder, Remove Directory, Edit Filter…) are identical across renderers. |

The three SwiftUI views share these behaviors via a small set of
helpers — `treeRowHandlers(state:)`, `treeKeyboardHandlers(state:)`
— so the contract is enforced by composition, not by re-implementation
in each renderer.

* §3A.4 — the in-memory `RootDirectory` / `DirectoryNode` graph

* §3C.6 — `FileTreeFloatingView`'s composition; the floating

* [menus.mdx](./menus.mdx) §4.4 — full View-menu layout; the
  *Tree View ▸* submenu is added there.
* [settings.mdx](./settings.mdx) §4.10 — Gallery settings, where

### 3D.10 Acceptance criteria (in addition to §11)

The Selectable Tree View ships when all of the following pass:

1. **Three menu items exist.** View ▸ Tree View has exactly three
   items: AppKit, SwiftUI, Catalyst — in that order — with the
   hotkeys ⌃⌥⌘1 / ⌃⌥⌘2 / ⌃⌥⌘3.
2. **One checkmark always.** Exactly one of the three items has a
   checkmark glyph; the others do not. The checkmark is on the
   item matching `state.treeRenderTechnology`.
3. **Toggle round-trips.** Selecting an unchecked item updates the
   checkmark, swaps the rendered view in both the inline panel
   and the floating window, and survives an app relaunch.
4. **Data parity.** The same `state.walkerRoots` graph renders in
   each renderer; toggling does not lose or duplicate rows.
5. **Selection parity.** Selecting a file in any renderer updates
   `state.selectedFile` and loads the file in the viewer canvas
   (§4.3).
6. **Default is SwiftUI.** A fresh `UserDefaults` (no prior key
   stored) leaves the SwiftUI item checkmarked on first launch and
   writes `"swiftui"` to `ig.tree_render_tech`.
7. **MCP round-trip.** `set_tree_render_tech("appkit")` moves the
   checkmark to AppKit and swaps the renderer within one main-actor
   tick; `get_tree_render_tech()` returns `"appkit"`.
8. **No re-walk on swap.** Switching renderers does not increment
   any root directory's `last_walked` value and does not trigger
   an `FSEventStream` callback.

## 4. UI Mechanics

### 4.1 Panel chrome

The panel toolbar (top of the panel, ~36 pt high):

```
[Scope name ▾]  [⌕ filter]  [sort: Date ▾]  [▣ ░ ⊟ ┣ ⫯ view]  [↻ refresh]  [✎ edit scope]
```

* **Scope name** — popover with the list of saved scopes and an
  `[+ New scope…]` entry; clicking a name swaps the active scope.
* **Filter** — `TextField`; matches against filename and (in v1.1)
  date / size tokens.
* **Sort** — `Menu`; reuses `viewer.image_order` choices plus
  `dimensions`.
* **View toggle** — five buttons (or a `Picker`) for the five modes.
* **Refresh** — manual re-evaluation. Visually animated (spinning)
  while a re-walk is in progress.
* **Edit scope** — opens the `scope_editor` panel (which can be
  tabbed with this panel).

### 4.2 Item rendering

A `FileEntry` is rendered with:

```swift
struct FileEntry: Hashable, Sendable {
    var path: URL
    var size: Int64
    var modified: Date
    var created: Date
    var dimensions: CGSize?      // nil until resolved
    var exifDateTaken: Date?     // nil if not present or not yet resolved
    var typeIdentifier: UTType
    var isAnimated: Bool
    var isMultiFrame: Bool
    var isRAW: Bool
    var isHDR: Bool
    var isFavorite: Bool         // user-tagged via the panel
}
```

Visual elements:

* **Thumbnail**: 64 / 96 / 128 / 192 / 256 / 384 / 512 px square
  depending on `gallery.thumb_size` and the mode (strip uses one of
  the smaller; grid uses the configured size; details uses 24 pt;
  tree uses 18 pt).
  Generated via `CGImageSourceCreateThumbnailAtIndex` with
  `kCGImageSourceCreateThumbnailFromImageAlways = true` and
  `kCGImageSourceThumbnailMaxPixelSize = <target>`. Cached on disk
  under `~/Library/Caches/com.starbuck.imageglass/thumbnails/`
  keyed by source path + mtime + target size.

  ```swift
  func makeThumbnail(for url: URL, maxPixelSize: Int) throws -> CGImage? {
      let opts: [CFString: Any] = [
          kCGImageSourceCreateThumbnailFromImageAlways: true,
          kCGImageSourceThumbnailMaxPixelSize: maxPixelSize,
          kCGImageSourceCreateThumbnailWithTransform: true
      ]
      guard let src = CGImageSourceCreateWithURL(url as CFURL, nil) else { return nil }
      return CGImageSourceCreateThumbnailAtIndex(src, 0, opts as CFDictionary)
  }
  ```

* **Filename label** — truncated middle, system caption font.
* **Badges** — small SF Symbol indicators in the corners:

  | Badge | SF Symbol | Meaning |
  | --- | --- | --- |
  | Animated | `play.circle.fill` | Multi-frame animation (GIF / WebP / APNG) |
  | RAW | `camera.aperture` | Camera RAW |
  | HDR | `sun.max.circle.fill` | HDR JPEG / HEIC / AVIF |
  | Multi-frame | `square.stack.3d.up.fill` | Multi-page TIFF / multi-page PDF |
  | Favorite | `star.fill` | User-tagged |
  | Unread | `circle.fill` | New since last view (v1.1) |

  Badges are SF Symbols, not emoji, so they respect monochrome
  rendering, accessibility colors, and the user's "Increase contrast"
  setting.

* **Non-image files** — when the scope includes non-image files
  (PDF, text, video), the thumbnail is replaced by the file-type
  icon from `NSWorkspace.shared.icon(forFile:)`. The panel does not
  attempt to thumbnail unsupported formats; that prevents long
  stalls on huge files.

### 4.3 Selection model

* **Single-select** drives the viewer canvas: selecting any file
  whose `kind` is `.image`, `.svg`, or `.video` loads it into the
  viewer immediately. This is the canonical "click a thumbnail"
  behavior, and it applies to every selection mechanism — mouse
  click, keyboard arrow, MCP `select_file`, and the auto-select
  triggered by the walker (see
  [use_cases/mcp_file.mdx](../docs/use_cases/mcp_file.mdx) §2 and §10).
  Per-kind behavior on load:
  * `.image` — decoded via ImageIO and rendered fit-to-window on the
    `MTKView` canvas.
  * `.svg` — handed to the SVG preview surface ([svg.mdx](./svg.mdx))
    — `WKWebView` for animated content, Core Graphics static for the
    rest. The canvas swaps in the SVG view.
  * `.video` — handed to the video preview surface
    ([videos.mdx](./videos.mdx)) — AVKit `AVPlayerView` in the
    canvas. Playback does **not** start automatically on selection;
    the user presses Space (or the play button) to begin. Auto-play
    on selection would surprise the user with sound.
  Selection never silently fails: if the file is unreadable, the
  viewer renders its standard error placeholder and writes an
  `error.err` record with the file URL.
* **Multi-select** (Shift / Cmd-click) enables batch operations:
  * Delete (⌫) — moves to Trash via `FileManager.trashItem(at:resultingItemURL:)`.
  * Copy paths (⌘C in plain selection mode) — writes paths to the
    pasteboard.
  * Copy image data (⌥⌘C) — copies the first selection's image
    bytes.
  * Move / Copy to folder — opens a chooser.
  * Export… — passes to the export flow.
  * Batch crop — opens the crop tool with `tools.crop.persistent_selection = true` ([crop](./crop.mdx) §6.2).

### 4.4 Drag and drop

* **Drag out** — `NSItemProvider` file-promise items. The user can
  drag a selection into Finder, into another app, or into a
  multi-file-accepting upload field. Promises mean the file is not
  copied until the destination accepts.
* **Drag in** — accepts file URLs from Finder / other apps. Drop on
  the panel adds the URL(s) as a new source criterion to the active
  scope (with the user's confirmation if the scope is a saved
  named scope). Drop on an empty panel creates a new implicit
  scope.

## 5. Sort and Filter

### 5.1 Sort modes

The sort modes match `viewer.image_order` plus `dimensions`:

| Sort | Implementation |
| --- | --- |
| `name` | Natural sort (locale-aware, numeric-aware). Falls back to ASCII when filenames are not text. |
| `size` | `URLResourceKey.fileSize`. |
| `modified` | `URLResourceKey.contentModificationDate`. |
| `created` | `URLResourceKey.creationDate`. |
| `exif_date_taken` | EXIF `DateTimeOriginal` from `CGImageSourceCopyPropertiesAtIndex`. Files without EXIF sort after files with EXIF (configurable). |
| `extension` | UTI's preferred extension. |
| `random` | Stable random with the scope name as the seed (so the order is reproducible across launches until the scope is re-evaluated). |
| `dimensions` | Composite of width × height. Files without resolved dimensions are loaded asynchronously and re-inserted when ready. |

Direction is `asc` or `desc`. Headers in Details mode show the
direction with a `↑ / ↓` indicator.

### 5.2 Filter

The filter field is a free-form text input plus, in v1.1, a small
token language:

| Token | Effect |
| --- | --- |
| `*.jpg` | Filename glob. |
| `>5mb` | File size greater than 5 MB. |
| `wider:3000` | Dimensions width ≥ 3000 px. |
| `taller:2000` | Dimensions height ≥ 2000 px. |
| `after:2026-05-01` | Modified after this date. |
| `before:2026-06-01` | Modified before this date. |
| `kind:raw` | UTI conforms to `public.camera-raw-image`. |
| `kind:hdr` | EXIF / metadata flags HDR. |
| `fav:` | Favorite-tagged. |

Bare text (no `:` or `>`) is treated as a substring match against
the filename. Tokens compose with implicit AND.

## 6. Performance

### 6.1 Lazy decode

Only visible items have thumbnails decoded. Off-screen items use a
placeholder (a faint gray rectangle plus the file-type icon).
Visibility is tracked via the SwiftUI `onAppear` / `onDisappear`
hooks for grid/strip and via `NSCollectionView`'s prefetch APIs
for the rare cases where `LazyVGrid` is replaced by
`NSCollectionViewRepresentable`.

### 6.2 Bounded concurrency

Thumbnail decoding runs in a `ThumbnailCache` actor with a
`AsyncSemaphore` bounded by `ProcessInfo.processInfo.activeProcessorCount`:

```swift
actor ThumbnailCache {
    private let semaphore = AsyncSemaphore(value: ProcessInfo.processInfo.activeProcessorCount)
    private var memoryLRU: LRUCache<ThumbKey, CGImage> = .init(maxBytes: 256 * 1024 * 1024)
    private let diskRoot: URL

    func thumbnail(for entry: FileEntry, size: Int) async -> CGImage? {
        let key = ThumbKey(path: entry.path, mtime: entry.modified, size: size)
        if let hit = memoryLRU[key] { return hit }
        if let disk = await readDisk(key: key) {
            memoryLRU[key] = disk
            return disk
        }
        await semaphore.acquire()
        defer { Task { await semaphore.release() } }
        guard let img = try? makeThumbnail(for: entry.path, maxPixelSize: size) else { return nil }
        memoryLRU[key] = img
        Task.detached(priority: .utility) { await self.writeDisk(key: key, image: img) }
        return img
    }
}
```

### 6.3 LIFO queue

When the user scrolls fast, the queue of pending thumbnails
prioritizes the **most recently requested** items so the visible
viewport always wins. Scrolled-past items are dropped from the
queue without being decoded.

### 6.4 LRU eviction

* **Memory LRU** — capped by total bytes (256 MB default).
* **Disk LRU** — capped by total bytes (`gallery.disk_cache_mb`,
  default 400 MB; absolute cap `advanced.thumb_cache_mb`, default
  1024 MB). Eviction runs on a background `NSBackgroundActivityScheduler`
  job every 6 hours.

### 6.5 Paged scope walking

Scopes with > 50 000 files:

1. The walker yields the first 5 000 entries as soon as they are
   enumerated; the panel renders them while the walker continues.
2. Subsequent batches of 5 000 are appended; the panel uses a
   diff-friendly `IdentifiedArray` so SwiftUI does not re-render the
   whole list.
3. The resolved file list is streamed to disk in chunks, not held
   entirely in memory.
4. The panel toolbar shows a thin progress bar across the bottom
   of the toolbar until the walk completes.

### 6.6 FSEvent debounce

`FSEventStream` events are debounced 250 ms before being dispatched
to the panel. Bulk file operations (e.g., a `cp -R` of 500 files)
collapse to a single re-evaluation rather than 500.

### 6.7 Memory budget

Targets, measured on an M3 Pro:

| Scope size | Idle memory | Active-scroll memory |
| --- | --- | --- |
| 100 files | ~40 MB | ~80 MB |
| 5 000 files | ~80 MB | ~200 MB |
| 50 000 files | ~150 MB | ~400 MB |
| 250 000 files | ~300 MB | ~600 MB |

Above 250 000 files the panel shifts to `NSCollectionView` for the
grid mode automatically.

### 7.2 ScopeWalker

```swift
actor ScopeWalker {
    func walk(_ scope: Scope) -> AsyncThrowingStream<FileEntry, Error> {
        AsyncThrowingStream { continuation in
            Task.detached(priority: .utility) {
                let keys: [URLResourceKey] = [.isDirectoryKey, .fileSizeKey,
                                              .contentModificationDateKey,
                                              .creationDateKey,
                                              .typeIdentifierKey]
                for criterion in scope.criteria {
                    guard let enumerator = FileManager.default.enumerator(
                        at: criterion.root,
                        includingPropertiesForKeys: keys,
                        options: criterion.includeHidden ? [] : [.skipsHiddenFiles]
                    ) else { continue }

                    for case let url as URL in enumerator {
                        if Task.isCancelled { continuation.finish(); return }
                        if matches(url, criterion: criterion) {
                            continuation.yield(makeEntry(url, keys: keys))
                        }
                    }
                }
                continuation.finish()
            }
        }
    }
}
```

## 8. MCP Integration

The MCP server exposes these tools. All read tools are R-only; all
write tools mutate scope JSON via atomic-rename writes and
notify the panel via `FSEventStream`.

| Tool | What it does |
| --- | --- |
| `list_scopes` | Return every scope name + summary. |
| `get_active_scope` | Return the name of the currently active scope (or `null` for implicit). |
| `set_active_scope(name)` | Set the active scope. |
| `get_scope(name)` | Return the full scope JSON. |
| `create_scope(name, criteria, sort?, filter?, description?)` | Create a new scope file. |
| `update_scope(name, patch)` | Patch fields of an existing scope. |
| `delete_scope(name)` | Delete a scope. Refuses to delete the active scope; caller must `set_active_scope` first. |
| `evaluate_scope(name)` | Force re-walk. Returns the resolved count and the wall-clock duration. |
| `list_files_in_scope(name, offset?, limit?)` | Return the resolved file list, paginated. |
| `select_file(path)` | Set the panel's selection to this single file (also loads it in the viewer). |
| `select_files(paths)` | Set the panel's selection to a list. |
| `set_view_mode(mode)` | Switch the panel between strip / grid / details / tree / column. |
| `set_sort(by, direction)` | Set the sort. |
| `set_filter(text)` | Set the filter text. |

Each tool's full JSON-RPC schema lives in
`Source/MCPSchemas/file_list/*.json` and is registered with the
MCP server at panel registration time.

### 8.1 Push events

The MCP server also pushes events that clients can subscribe to:

* `scope_evaluated` — payload: scope name, file count, elapsed_ms.
* `scope_changed` — payload: scope name; emitted on any field change.
* `selection_changed` — payload: list of paths.
* `view_mode_changed` — payload: new mode.

Clients use these to keep their own UI / state in sync without
polling.

## 9. Workflow Scenarios

### 9.1 Preview.app refugee (T1)

User installs the app, opens `~/Desktop/photo.heic` with Cmd-O.
* Implicit scope created from `~/Desktop`.
* Panel docks at the bottom in `strip` mode (Settings default).
* User scrolls horizontally; arrow keys advance through photos.
* No file panel work required from the user.

### 9.2 Photographer with multi-source scope (T2a)

* Photographer creates scope `2026-shoots` via Settings (`[+ New
  scope…]` from the scope picker), pointing at one local and one
  external-drive directory.
* Sets extension filter to `nef, cr3, dng`.
* Picks sort = EXIF Date Taken (descending).
* Switches the panel to `grid` mode, size L.
* When the external drive is unplugged, the panel shows the local
  files only; reconnecting the drive triggers a 250 ms-debounced
  re-walk that re-includes them.

### 9.3 Claude Code rotates scopes (T4)

* User says "use my review queue."
* Claude Code calls `set_active_scope("review-queue")`.
* Panel switches; viewer loads the first file.
* User finishes review; tells Claude Code "now switch to last
  week's exports."
* Claude Code: `set_active_scope("exports-2026-w22")`.
* Zero GUI clicks.

### 9.4 Designer drag-out to Sketch (T2b)

* Designer in Details mode, sorted by Modified descending.
* Multi-selects three icon PNGs (Cmd-click).
* Drags the selection out of the panel onto a Sketch artboard.
* Sketch accepts the file promises; the files are not duplicated
  until Sketch reads them.

### 9.5 Tree-mode disambiguation (T3)

* User has two folders named `IMG_0001.jpg` in different source
  directories.
* In `details` mode the rows look identical.
* User switches to `tree` mode; each file is visibly under its
  source directory.
* User right-clicks the wrong one → Move to Trash.

## 10. Open Questions

* **Vision-backed search (v2).** "Find photos that look like this"
  via on-device Vision embeddings is a natural addition once the
  thumbnail cache is in place. Out of scope for v1.
* **Per-file rating UI.** Should ratings live in the file panel or
  in the metadata panel? Tentative: metadata. The file panel renders
  the rating as a small badge but does not allow editing.
* **Auto-discovered scopes.** Should the app surface a "Recents"
  scope or a "Photos library" scope automatically? Tentative: no
  automatic scopes in v1 — but the Quick Setup flow ([Settings §2.7](./settings.mdx#27-quick-setup-first-run))
  could offer to create them.
* **NAS / S3 / cloud scopes.** Scopes today assume local filesystem
  (or mounted volumes). A future "Cloud" scope kind would walk
  remote storage. Out of scope for v1; spec'd here for awareness.

## 11. Acceptance Criteria

The file panel ships when all of the following pass:

* **Five modes load correctly** for a 5 000-file scope with mixed
  formats (JPEG, HEIC, PNG, DNG, MP4, PDF, TXT).
* **Live updates**: copying 500 files into a watched directory
  triggers exactly one re-walk within 500 ms.
* **Thumbnail throughput**: ≥ 200 thumbnails / second on M3 Pro for
  cached items, ≥ 50 / second for first-time generation.
* **Memory cap**: a 250 000-file scope holds idle at ≤ 350 MB.
* **MCP round-trip**: `create_scope` → `evaluate_scope` →
  `set_active_scope` → `list_files_in_scope` returns the expected
  list and the GUI reflects every change within 500 ms.
* **Sandbox**: a sandboxed build with a security-scoped bookmark
  correctly enumerates a user-granted directory across an app
  relaunch.
* **Crash safety**: kill the process mid-scope-write 100 times; on
  relaunch every scope is either the prior valid state or its
  `.bak` and we recover.
* **Accessibility**: VoiceOver announces each item by filename, size,
  and modified date; the view-mode toggle is keyboard-traversable.
