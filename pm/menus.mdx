# macOS Menu Bar, Hotkeys, and the Current Image

> **Document type:** Product specification.
> **Audience:** Engineering (implementation) and Product (text,
> grouping, keyboard shortcuts).
> **Owners:** PM for menu text, grouping, and discoverability; CTO for
> the SwiftUI `CommandMenu` / AppKit `NSMenu` bridging and shortcut
> conflicts.
> **Status:** v1, **as-built**. §3–§4 and §7 describe the menus and
> shortcuts actually wired in `ImageGlassApp.swift`. Aligns with
> `hotkeys.mdx`, `dir_ui.mdx`, `mcp.mdx`, `local_storage.mdx`, and
> `use_cases/slideshow.mdx`.

---

## 1. Purpose

This document specifies the **full menu surface** of ImageGlass_Mac:
the macOS menu bar and the keyboard shortcuts attached to each command.
It is the canonical reference for:

1. Which menus exist, in which order, and what each item does.
2. The **keyboard shortcut for every menu item** — the app's contract
   is that every actionable command is reachable by a hotkey **without
   opening the menu**, and that **no two commands share a chord**.
3. The **current image** concept — the single cursor every surface
   reads — and the *Copy Current Image Path* command built on top of it.

We follow Apple's macOS Human Interface Guidelines so the app feels
like it shipped with the OS, not like a ported Windows application.

---

## 2. Design Principles

1. **Native first.** Standard menus (App, File, Edit, View, Window,
   Help) appear in their standard positions with their standard
   shortcuts. The user's muscle memory works on day one.
2. **One verb per menu item.** Item text reads as an active verb
   phrase: *Add Directory…*, *Re-evaluate Active Scope*, *Reveal in
   Finder*.
3. **Ellipses (`…`) for any item that opens a dialog or asks for
   further input.** Items that act immediately do not get an ellipsis.
4. **Capitalization follows macOS title case.**
5. **Group related commands with separators.**
6. **Every actionable item carries a hotkey (§2.1).**
7. **No shortcut collisions (§2.2).** Custom shortcuts never collide
   with each other or with system-reserved shortcuts (Spotlight `⌘Space`,
   Mission Control, the Edit-menu standards).

### 2.1 Every Actionable Item Has a Hotkey

Every menu item that performs an action **must** be reachable by a
keyboard shortcut, so a power user never has to open a menu to invoke a
command. There are exactly three documented exemptions, because a
static per-item shortcut is either impossible or actively bad UX for
them:

* **Enumeration / radio submenus** — *Zoom Mode* (6 modes), *Color
  Channel* (5), *Color Picker Format* (7). These are mode pickers, not
  commands; the frequently-used members already have their own chords
  or bare keys (e.g. *Actual Size* `⌘0`, *Zoom to Fit* `⌘9`), and
  binding a global chord to every leaf would exhaust the namespace and
  invite accidental mode-flips. The parent submenu is one keystroke
  away in the menu bar.
* **Dynamic list items** — *Open Recent ▸ \<file\>*, *Registered
  Directories ▸ \<root\>*, *Reopen/Forget Closed Window ▸ \<window\>*.
  Their membership changes at runtime, so no static shortcut is
  meaningful.
* **Apple-owned standard items** — *About*, *Services*, *Hide*, *Quit*,
  the system *Window* items, and *Help ▸ \<search\>* keep their OS
  conventions.

Every other item — including toggles — carries a chord. Where a command
also has a modifier-less **bare-letter** viewer key (from `hotkeys.mdx`:
`C` center, `M` normalize, `Z` fit, `W` width, `N`/`P` next/previous,
`S` slideshow, `I` include, `⇧P` copy-path), the bare key is the
in-canvas twin and the menu chord is the focus-context-free twin. Both
route to the same action.

### 2.2 No Collisions

Two commands may never resolve to the same physical chord. The complete
map in §7 is the single source of truth; it is verified to be
collision-free. The three collisions that existed in the first cut were
resolved as recorded in §7.1.

## 3. Menu Bar Layout (As-Built)

The menu bar contains the following menus, left to right. SwiftUI places
the standard menus (App, File, Edit, View, Window, Help) in their
canonical positions and inserts our `CommandMenu`s (Viewer, Navigate,
Crop, Directories, Layout) after them.

1. **ImageGlass** (application menu)
2. **File**
3. **Edit**
4. **View**
5. **Viewer** — image display, zoom, rotate, frames, slideshow *(new)*
6. **Navigate** — previous / next image *(new)*
7. **Crop** — the crop tool *(new)*
8. **Directories** — scope roots + include state *(new)*
9. **Layout** — panel-layout presets *(new)*
10. **Window**
11. **Help**

Menus marked *(new)* are fork additions not present in upstream
ImageGlass. Wiring lives in `code/Sources/ImageGlass/ImageGlassApp.swift`
(the `.commands { }` builder and the `@CommandsBuilder` properties
`viewerMenuCommands`, `cropMenuCommands`, `directoriesMenuCommands`,
`layoutMenuCommands`, `windowMenuCommands`).

## 5. Current Image and the Copy-Path Command

### 5.1 Copy Current Image Path

*File ▸ Copy Current Image Path* (`⌃⌘C`) copies the absolute POSIX path
of the image currently on the viewer canvas to the clipboard as plain
text.

* It operates on `AppState.currentImage` (§6), resolved through
  `FileActions.resolveTarget(selectedFile:panelCursor:)` so a
  panel-cursor parked on a real file row wins over a stale viewer
  selection.
* A missing file still copies the path and returns `ok=true note=missing`
  (the user may want the path even after a move).
* Implemented in `code/Sources/ImageGlass/FileActions.swift`, method
  `FileActions.copyFilePath(state:source:)`; wired in the File menu in
  `ImageGlassApp.swift`. The bare `⇧P` viewer key (`HotkeyHandlers.swift`,
  `handleCopyPathKey`) is the in-canvas twin.

## 6. The Current Image — Single Global Cursor

The app has exactly **one** "current image" at any moment. It is exposed
as `AppState.currentImage` (a thin, observable alias over
`AppState.selectedFile`), an absolute POSIX path (or `~/`-prefixed), or
`nil` when the viewer is empty. Every surface reads and writes the same
value:

* The **viewer canvas** loads the file at `currentImage`.
* The **directory / filename panel** highlights the row equal to
  `currentImage` and auto-expands its ancestors.
* The **slideshow** reads it on every tick (`slideshow.mdx` §0A).
* The **second viewer** and **floating file tree** mirror it.

**The cursor updates as the user cycles.** Any gesture that moves through
images — `←`/`→`, bare `N`/`P`, a panel-row click, a slideshow advance,
a drag-drop, or an MCP `select_file` — writes the newly-visible image's
path into `currentImage` (via `selectedFile.didSet`), so the "current
image" is always whatever is on screen right now. Menu items that act on
the image (Copy Current Image Path, Print, Copy Image, Move to Trash,
Fit Window to Image) all read this one value and disable themselves when
it is `nil`.

Implemented in `code/Sources/ImageGlass/AppState.swift`
(`AppState.currentImage` / `AppState.selectedFile`). Persistence of the
value across quit/relaunch is specified in `local_storage.mdx`
§"Last Viewed Image".

### 7.3 Bare-Letter Viewer Keys (from `hotkeys.mdx`)

These fire only in the viewer/panel focus context and are **suppressed
whenever `⌘`/`⌥`/`⌃` is held**, so they never shadow a menu chord:
`C` center · `M` normalize · `Z` fit · `W` width · `+`/`-` zoom ·
`N`/`P` next/previous image · `⇧P` copy current image path · `S`
slideshow · `I` cycle include · arrows navigate/pan. They are the
in-canvas twins of the menu chords above.

## 8. Localization Notes

* All menu strings live in a single `Menus.strings` file so they can be
  localized without code changes.
* Ellipses are the Unicode `…` (U+2026).
* Modifier glyphs (`⌘`, `⇧`, `⌥`, `⌃`) come from the system.

---

## 9. Non-Goals (v1)

* **Rebindable shortcuts.** The map in §7 is fixed in v1; a
  *Help ▸ Keyboard Shortcuts…* viewer and user rebinding are deferred.
* **Per-scope menu customization.** The menu bar is identical regardless
  of active scope.
* **Context menus** are specified in `right_click.mdx`, not here.

---

## 10. Acceptance Criteria for v1

1. Every menu and item in §4 is wired to a working command; disabled
   items grey out per the noted rules.
2. Every chord in §7 resolves to exactly one command, with no
   duplicate and no system-shortcut conflict. Automated check: collect
   every `.keyboardShortcut(...)` in `ImageGlassApp.swift`, assert the
   set of (key, modifiers) pairs has no repeats.
3. Every actionable item (excluding the §2.1 exemptions) has a chord.
4. *File ▸ Copy Current Image Path* copies the absolute path of the
   image on the canvas; it disables when the viewer is empty.
5. The current image (`AppState.currentImage`) tracks the on-screen
   image as the user cycles, and is restored on relaunch per
   `local_storage.mdx` §"Last Viewed Image".
