description: Scenario-by-scenario specification for slideshow mode. Pressing `S` (no modifier) on the main viewer toggles slideshow on or off; while running, the controller advances to the next image after the configured interval, re-applies the slideshow zoom mode to every advanced image, and loops at the end of the file list. The interval is sourced from `settings.slideshow.interval_seconds` (default 4.0 s) and is editable in two places that share the same binding — a horizontal slider AND an editable text box — inside Settings (⌘,) ▸ Slideshow ▸ Timing. Items whose include state is *exclude* or *inherit-exclude* are skipped from navigation entirely. The bare-letter hotkeys `N` (next image) and `P` (previous image) use the same cursor and the same order, both while a slideshow is running and while it isn't. Each scenario lists ordered user actions, the expected on-screen and on-disk result, and an explicit verify step that proves the slideshow advanced, looped, re-zoomed, and persisted correctly. Every state change is journaled to `log.log` so an external observer can replay the session.

# Use Case — Slideshow Mode

This page is a **scenario-by-scenario specification** for the
slideshow feature of ImageGlass_Mac. It is written as a sequence of
user actions: *press this key, then this, then this*, and at each
step it spells out the on-screen, on-disk, and in-state result so a
human (or an XCUITest run) can verify the step actually worked.

It is the user-facing companion to:

* [Hotkeys](./hotkeys.mdx) §7 — keyboard map; `S` is the
  no-modifier slideshow toggle introduced here.
* [Settings](../settings.mdx) §4.6 (Slideshow) — the `slideshow.*`
  JSON keys this scenario reads and mutates, especially
  `slideshow.interval_seconds`.
* [Panels](../panels.mdx) §7.5 — the *Slideshow* layout preset
  the controller switches to when slideshow starts.
* [Videos](../videos.mdx) §11.2 — the focus-context rule that
  keeps `Space` bound to *Play / Pause* when a video is current
  and falls back to *Slideshow* when an image is current. `S` is
  the **unconditional, focus-context-free** toggle introduced here;
  it does not displace the existing `Space` or `⌥⌘S` bindings.
* [Error Handling](../error_handling.mdx) §4.2 — the `log.log`
  record format every entry in this scenario must produce.

Nothing in this spec changes the slideshow window's rendering
pipeline, the `SlideshowController` actor model, or the
`SlideshowSettings` JSON schema. It adds:

1. A no-modifier `S` toggle on the main viewer.
2. An "appropriate zoom mode" rule that re-applies the slideshow
   zoom to **every** image the controller advances to, not just
   the first.
3. A side-by-side **Slider + editable number field** in
   Settings ▸ Slideshow ▸ Timing for the interval, replacing the
   slider-only row in the current Settings scene.

## 0. Common Preconditions

These hold for every scenario unless a scenario explicitly overrides
them.

* The app is installed, signed, and at least one previous launch has
  happened (so `~/Library/Application Support/ImageGlass_Mac/`
  exists with `settings.json` and the `logs/` directory).
* At least one root directory in `directories.yaml` has been walked
  and contains ≥ 3 image files whose `kind == .image`. The reference
  fixture is `~/Pictures/tour/` (the same fixture used by
  [`mcp_file.mdx`](./mcp_file.mdx) §0), with images sorted
  depth-first lexicographically:

  ```
  tour/beach/001.jpg
  tour/beach/002.jpg
  tour/beach/notes_old.png
  tour/mountains/peak.heic
  tour/mountains/peak.jpg
  ```

  Five `.image` files in deterministic order. The non-media file
  `tour/README.txt` never enters the list (built-in file-kind
  filter, [list_of_files.mdx §3A.3](../list_of_files.mdx#3a3-built-in-file-kind-filter)).
* The viewer is the focused window. The status bar reads `5 files`,
  `001.jpg` is selected, and the viewer canvas shows that image.
  The panel may be in any view mode; this spec does not depend on
  panel layout.
* `settings.json` is at its defaults for the `slideshow.*` block —
  in particular `interval_seconds = 4.0` and `loop = true`.
  Scenario 4 mutates `interval_seconds`; scenario 7 mutates `loop`.
* "Verify" means: read the named file with `cat`, read the named
  pixel region on screen, observe the named in-memory field, or
  read the named log line. Each verify step is a check a human or
  an XCUITest run can perform.

Two related shortcuts continue to work alongside the new `S`
binding, and are NOT redefined by this spec:

| Existing shortcut | Action | Owner spec |
| --- | --- | --- |
| `⌥⌘S` | View ▸ Start Slideshow (unconditional, menu-driven) | [videos.mdx §11.2](../videos.mdx#112-resolution) |
| `Space` | View ▸ Slideshow when the file list is focused and the current file is an image; Video ▸ Play/Pause when a video is current; SVG ▸ Play/Pause when an animated SVG is current. | [videos.mdx §11.2](../videos.mdx#112-resolution), [svg.mdx §11](../svg.mdx#11-keymap-summary) |
| `S` *(this spec)* | Toggle slideshow — unconditional from the main viewer scene, focus-aware (suppressed when a text field is first responder). | This file. |

The three are independent. `S` is the **quick toggle**: one key,
no modifier, no menu walk. `⌥⌘S` is the **menu-bound shortcut**
power users expect to find under View. `Space` is the
**context-sensitive shortcut** that gives way to video and SVG
playback when those kinds are current.

## 0A. The Currently Selected Image — Single Global Cursor

The app has exactly **one** "currently selected image" at any time.
That cursor lives on `AppState.selectedFile` (an absolute POSIX
path or `nil` when nothing is loaded) and every UI surface reads
the **same** value:

* The **main viewer canvas** loads the file at `selectedFile`.
* The **slideshow window's canvas** (when running) loads the file
  at `selectedFile` — the slideshow does **not** maintain a
  separate cursor.
* The **directory / filename panel** highlights the row whose
  full path equals `selectedFile`, and (per `selectedFile.didSet`)
  the panel auto-expands every ancestor folder of that row so the
  highlight is actually rendered. The highlight tracks the cursor
  whether the cursor moved because the user clicked the row, hit
  `↑` / `↓` / `N` / `P`, dropped a file from Finder, the slideshow
  advanced, or an MCP `select_file` call landed.
* The **arrow-key tree cursor** (`treeNav.activeRow`) is mirrored
  to `selectedFile` on every selection change, so an immediate
  `↑` / `↓` press after a click resumes navigation from the
  clicked row.

### 0A.1 The rule, exact

Any code path that mutates the cursor writes through
`AppState.selectedFile` (and only that). No view, controller, or
timer holds a private "current index" that can drift from the
cursor. In particular:

1. The **slideshow controller** reads `appState.selectedFile`
   **at every tick** to decide which file is "current" and what
   the next file should be. It does not cache the index from
   `start()` or from the previous advance.
2. The **N / P hotkeys** (§10A) call `appState.selectNext()` /
   `appState.selectPrevious()`, which read `selectedFile` and
   write the new value back.
3. The **panel click handler** sets `appState.selectedFile = path`
   and nothing else.

The consequence: any UI gesture that moves the cursor is
**immediately visible** to the next slideshow advance, the next
`N` / `P` press, and every panel highlight, with no extra wiring.

### 0A.2 Verify the single-cursor invariant

* Click the row for `tour/mountains/peak.heic` in the panel.
* On screen: the row highlights blue; the viewer canvas reloads
  with `peak.heic`; the breadcrumb (if visible) updates.
* In state: `AppState.selectedFile == ".../tour/mountains/peak.heic"`
  and `treeNav.activeRow == ".../tour/mountains/peak.heic"`.
* Press `↓` once. The panel highlight and the viewer both advance
  to the next visible row, which **must** be the next item in the
  file tree's top-down order (§1A) — not whatever was "next" in
  a stale `resolvedFiles` index.

## 1A. Ordering — Navigation Follows the File Tree's Visible Order

The list the slideshow walks (and that `N` / `P` step through) is
the **file tree's visible, top-down order**. Concretely:

* Walker roots in `directories.yaml` order (top-to-bottom in the
  Directories panel).
* Within each root, a **depth-first** descent: every file in a
  folder is visited before stepping into the next sibling folder.
* Within a folder, **lexicographic** order of file names (the
  same order the directory panel renders).
* Folders themselves are **not** navigation targets — only files
  produce a slide. Empty folders and folders whose only contents
  are non-image files are stepped past silently.

The implementation handle for this list is
`AppState.orderedNavigationFiles`, which delegates to
`DirectoryFilenamePanel.flattenVisible(walkerRoots)` when walker
roots exist and falls back to the legacy `resolvedFiles` only
when no walker roots are configured.

### 1A.1 Worked example

Reference fixture: `~/Pictures/tour/` from §0. Extended for this
section to make the ordering rule observable:

```
tour/
├── beach/
│   ├── 001.jpg
│   ├── 002.jpg
│   ├── 003.jpg
│   └── notes_old.png
├── desert/
│   ├── 011.jpg
│   ├── 012.jpg
│   └── 013.jpg
└── mountains/
    ├── peak.heic
    └── peak.jpg
```

The navigation list, top-down, is exactly:

```
beach/001.jpg
beach/002.jpg
beach/003.jpg
beach/notes_old.png
desert/011.jpg
desert/012.jpg
desert/013.jpg
mountains/peak.heic
mountains/peak.jpg
```

A slideshow that starts on `beach/002.jpg` advances through
`003.jpg`, then `notes_old.png`, then crosses into `desert/`
starting with `011.jpg`, and so on. It does **not** jump from
`002.jpg` directly to `mountains/peak.heic` (that would be
out-of-order with respect to the visible tree).

### 1A.2 Skip rule — exclude / inherit-exclude

A file is in the navigation list **iff** its tree node's
`passesFilter` flag is true. The walker computes `passesFilter`
from the file's include-state plus its ancestor folders'
include-states, per `include_checks.mdx` §3:

| Include state | Effective | In nav list? |
| --- | --- | --- |
| `include` (explicit) | yes | **yes** |
| `inherit` with an `include` ancestor | yes | **yes** |
| `inherit` with no override anywhere | depends on root default | follows root |
| `exclude` (explicit) | no | **no — skipped** |
| `inherit` with an `exclude` ancestor | no | **no — skipped** |

Slideshow advance, `N`, `P`, `↑`, `↓`, and the folder-jump
shortcuts all walk the **same** filtered list, so a cycle through
the slideshow visits exactly the files the panel highlights as
included — never a skipped file.

When the user toggles a file or a folder's include state
**during** a running slideshow (the `I` hotkey from
`include_checks.mdx` §4), the navigation list is recomputed
before the next advance. If the **currently displayed** file
just became excluded, the slideshow keeps it on screen for the
remainder of the current countdown (the user can still see what
they just excluded) and steps to the next included file on the
next tick.

### 1A.3 Verify the ordering rule

* Use the §1A.1 fixture. Click `beach/002.jpg` in the panel.
  Press `S`. Wait `interval_seconds`.
* Expected: the next slide is `beach/003.jpg`. Wait again — next
  is `beach/notes_old.png`. Wait again — next is
  `desert/011.jpg` (crossing folders only after exhausting
  `beach/`).
* If the next slide is `mountains/*` or any `desert/` file before
  `beach/` is exhausted, the advance is using the wrong list and
  §1A is failing. Verify `AppState.orderedNavigationFiles` is
  what the controller reads.

## 1. First-Time Slideshow Start — `S` Toggles On

### 1.1 Preconditions

* §0 holds verbatim. The viewer shows `001.jpg`. The status bar
  reads `5 files`. No slideshow window is open.

### 1.2 Actions, in order

1. With the main viewer focused (no text field is first
   responder), press `S` once.

### 1.3 Expected result

* `SlideshowController.shared.isRunning` flips from `false` to
  `true`.
* The controller reads `settings.slideshow.interval_seconds` (4.0)
  as the **interval** for the run, and `viewer.slideshowSeconds`
  is updated to that same value so the in-window stepper and the
  countdown overlay observe the live source of truth.
* The controller opens the slideshow window (per
  [Slideshow Window](../panels.mdx#75-slideshow-controlcmd5)),
  switching the layout to the *Slideshow* preset. The main window
  is hidden if `settings.layout.hide_main_in_slideshow == true`
  (default).
* The slideshow window's viewer canvas loads `001.jpg`. The
  canvas re-applies the **slideshow zoom mode** — `.fit` (Zoom
  to Fit) by default, the same `ViewerState.zoomMode = .fit`
  already set in `SlideshowController.start`. The image is
  rendered scaled-to-fit on the Metal canvas, centered, with no
  pan offset.
* The countdown overlay reads `4.0s` and starts ticking down at
  10 Hz toward the next advance.
* The main viewer (if still visible) shows its inert state — no
  countdown, no zoom change. The slideshow's `ViewerState` is a
  separate instance so the main window's zoom / pan / rotation
  are untouched.

### 1.4 Verify

* On screen: a new window titled `ImageGlass Slideshow` is key.
  Its canvas shows `001.jpg` fit-to-window, centered, on the
  configured slideshow background (default black). A countdown
  badge in the top-right reads a value between `4.0s` and `0.0s`.
* In state:
  * `SlideshowController.shared.isRunning == true`.
  * The slideshow window's `ViewerState.zoomMode == .fit`.
  * The slideshow window's `ViewerState.slideshowSeconds == 4.0`.
  * The main window's `ViewerState` is unchanged from before
    the press (same `zoomMode`, same `panOffset`).
* On disk:

  ```sh
  tail -n 1 ~/Library/Application\ Support/ImageGlass_Mac/logs/log.log
  ```

  contains a line of the form

  ```
  ts=… tool=slideshow.toggle on=true interval=4.0 source=key:S corr=… ok=true
  ```

  The `source=key:S` field is the audit fingerprint of the `S`
  binding; menu-driven starts (`⌥⌘S`) record `source=menu:View`,
  and MCP-driven starts record `source=mcp:set_slideshow`. The
  three sources share a common log shape so a debugger can
  collate them.

## 2. Slideshow Advance — Next Image After the Interval

### 2.1 Preconditions

* Scenario 1 completed. The slideshow is running, showing `001.jpg`,
  with `interval_seconds = 4.0`.

### 2.2 Actions, in order

1. Wait. Do not press any key, do not move the mouse.
2. After ≥ 4 seconds (one full interval), observe.

### 2.3 Expected result

* Exactly at the wall-clock interval boundary
  (`startInstant + 4.0 s`, within ≤ 50 ms scheduler jitter), the
  controller reads `AppState.selectedFile` (`001.jpg`), looks it
  up in `AppState.orderedNavigationFiles` (§1A), and advances
  `AppState.selectedFile` to the next item in that list
  (`002.jpg`). The controller **does not** use a cached index
  from `start()` — every tick rereads `selectedFile`, so a click
  in the directory panel during the countdown re-anchors the
  cursor and the next advance steps from the clicked file
  (see §2A).
* The slideshow window's `ImageCanvasView` reloads with `002.jpg`.
* The canvas re-applies the slideshow zoom mode (`.fit`) — see
  §6 for the exact rule. Even if scenario 2.5 had panned or
  zoomed the prior image, the new image is presented fit-to-window
  again.
* The countdown overlay snaps back to `4.0s` and begins counting
  down toward the *next* advance.
* The status bar (inside the slideshow window) updates to
  `002.jpg` and the file metadata.

### 2.4 Verify

* On screen: at ~4 s the displayed image visibly changes from
  `001.jpg` to `002.jpg`. The countdown re-arms to `4.0s`.
* In state:
  * `AppState.selectedFile` is now the URL of `002.jpg`.
  * The slideshow's `ViewerState.zoomMode == .fit` (unchanged
    — but verifiably re-applied; see §6).
  * The slideshow's `ViewerState.panOffset == .zero` (any prior
    user pan was cleared by the zoom-mode reset).
* On disk:

  ```sh
  grep "app=slideshow.advance" ~/Library/Application\ Support/ImageGlass_Mac/logs/log.log | tail -n 1
  ```

  reads

  ```
  ts=… app=slideshow.advance from=…/001.jpg to=…/002.jpg interval=4.0 zoom_mode=fit corr=…
  ```

  The `interval=` field records the interval that was *just
  elapsed* (4.0), not the next-advance budget. `zoom_mode=fit`
  records the zoom mode the new image is rendered in — proof
  that §6 fired.

### 2.5 Continues — successive advances

The controller continues to advance once per interval:
`002.jpg → notes_old.png → peak.heic → peak.jpg → …`. After
`peak.jpg`, the loop behavior of scenario 7 takes over.

## 2A. Click During Slideshow Re-Anchors the Cursor

This is the bug-driven contract that prompted §0A and §1A.
A user click in the directory / filename panel during a running
slideshow **must** move the slideshow cursor, and the **next**
advance must step from the clicked file, not from where the
controller was before the click.

### 2A.1 The rule, exact

1. The directory panel's row click handler runs
   `state.selectedFile = path` and nothing else (see
   `DirectoryFilenamePanel.swift` row tap binding).
2. `AppState.selectedFile.didSet` mirrors the new path into
   `treeNav.activeRow` and reveals every ancestor folder so the
   row is highlighted.
3. The slideshow controller's tick handler — on every fire —
   reads `appState.selectedFile` and computes the new "next" by
   `orderedNavigationFiles.firstIndex(of: selectedFile) + 1`.
   The previous "next" (the one the user would have seen if they
   hadn't clicked) is discarded.
4. The countdown is **not** restarted by the click. The user is
   choosing **which** image will display next, not adding time
   to the current slide. (If the user also wants to pause and
   inspect, they can press `S` to stop, then `S` again to
   restart.)

The same rule applies to MCP `select_file` calls and to drops
from Finder while a slideshow is running: any path that lands on
`AppState.selectedFile` becomes the slideshow's new anchor for
the next advance.

### 2A.2 Worked example — the illustration from the bug report

Use a 5-folder fixture, 10 images each, named `dirN/img-K.jpg`
where N is the folder index 1…5 and K is the global image index
1…50. The navigation list (per §1A) is `img-1.jpg, img-2.jpg, …,
img-50.jpg` in order.

* Start the slideshow on `img-1.jpg`. Let it run until
  `img-15.jpg` is on screen and the countdown reads ~2.0 s.
* Click `img-2.jpg` in the panel.

Expected, step by step:

* Immediately: the viewer canvas swaps to `img-2.jpg`. The panel
  highlight moves to the `img-2.jpg` row. `AppState.selectedFile
  == ".../dir1/img-2.jpg"`.
* Countdown continues from ~2.0 s — the click did not restart
  the timer.
* At countdown == 0, the controller reads `selectedFile`
  (`img-2.jpg`), finds it at index 1 in `orderedNavigationFiles`,
  and advances to `orderedNavigationFiles[2]` — which is
  `img-3.jpg`.
* Next advance: `img-4.jpg`. Then `img-5.jpg`. The slideshow has
  effectively rewound to image 2 and proceeds from there. It
  does **not** resume at `img-16.jpg` (the position before the
  click) — that position was discarded the moment the click
  changed `selectedFile`.

### 2A.3 Verify

* On screen: the post-click slide is `img-3.jpg`, not
  `img-16.jpg`.
* In state, immediately after the advance:
  `AppState.selectedFile == ".../dir1/img-3.jpg"`.
* On disk:

  ```
  ts=… app=slideshow.advance from=…/dir1/img-2.jpg to=…/dir1/img-3.jpg interval=4.0 zoom_mode=fit corr=…
  ```

  Note `from=…/img-2.jpg` — the audit log proves the controller
  stepped from the **clicked** file, not from `img-15.jpg`.

### 2A.4 Edge case — click the file already on screen

If the user clicks the row that is already current, `didSet`
short-circuits (`oldValue != selectedFile` is false) and nothing
happens. The slideshow continues from the same position.

### 2A.5 Edge case — click a file currently excluded

If the user clicks a row whose `passesFilter == false` (only
possible if the panel is showing folder skeletons that contain
excluded files via a future surface — today, excluded files are
not rendered as clickable rows), the click is honored:
`selectedFile` becomes the clicked path and the viewer displays
it. The **next advance**, however, sees that the file is not in
`orderedNavigationFiles` (`.firstIndex == nil`), falls into the
"current not in list" branch, and re-enters the list at index 0.
This is the same behavior as opening an ad-hoc file via drag-
drop or `L` (load from clipboard) while a slideshow is running:
the slideshow re-anchors to the start of the navigable list on
the next tick.

## 3. Slideshow Stop — `S` Toggles Off

### 3.1 Preconditions

* Scenario 2 completed, or scenario 1 just completed. A slideshow
  is currently running.

### 3.2 Actions, in order

1. With the slideshow window focused (or the main viewer; `S` is
   bound on both), press `S` once.

### 3.3 Expected result

* `SlideshowController.shared.stop()` is called.
* `isRunning` flips to `false`. The advance timer is invalidated.
* The slideshow window closes; the main window returns to key
  status. The main window's layout is restored to whatever preset
  was active before slideshow started.
* `AppState.selectedFile` retains whatever image was on screen at
  stop time. **The main viewer's canvas is not reloaded** and the
  main viewer's `ViewerState.zoomMode` is unchanged (no surprise
  zoom snap).
* The countdown overlay disappears.

### 3.4 Verify

* On screen: the slideshow window is gone. The main viewer is key
  and shows the file that was on screen at stop time.
* In state:
  * `SlideshowController.shared.isRunning == false`.
  * The main window's `ViewerState.zoomMode` equals its
    pre-slideshow value (unchanged).
  * `AppState.selectedFile` equals the last file the slideshow
    displayed.
* On disk:

  ```
  ts=… tool=slideshow.toggle on=false interval=4.0 source=key:S corr=… ok=true
  ts=… app=slideshow.stop reason=user_toggle advances=N elapsed_s=… corr=…
  ```

  `advances=N` is the count of successful advances during the run
  (4 if scenarios 1 → 2 → 2.5 ran end-to-end). `elapsed_s=` is the
  wall-clock duration from start to stop, used to verify the
  ~4 s/interval pacing in §11.

## 3A. Reliable Toggle — `S` Always Flips Play/Stop

This section is the contract for the bug the user reported: *"the
slideshow was stopping on its own, and pressing `S` didn't reliably
turn it back off."* The rule is now unambiguous:

> **`S` (and `Space`, and View ▸ Start/Stop Slideshow, and MCP
> `set_slideshow`) is a true toggle. If any slideshow is running,
> the toggle STOPS it. If none is running, the toggle STARTS one.
> There is no third outcome and no way for a bare `S` to spawn a
> second run.**

### 3A.1 Why it used to be unreliable

Every toggle surface funnels into
`SlideshowController.toggle(appState:source:windowID:)`
(`code/Sources/ImageGlass/Viewer/SlideshowWindow.swift`). When the
keypress carries **no** explicit `windowID` (every bare `S` / `Space`
does), the controller resolves the target with
`resolveTargetWindowID()` = `WindowRegistry.frontmostWindowID`. If the
run was keyed under a *different* window id than the one currently
resolved as frontmost — the frontmost changed since the run started,
or MCP started the run under a different resolved target — then
`isRunning(windowID: target)` was `false`, so the toggle took the
`start` branch and opened a **second** run instead of stopping the
visible one. The user experienced this as "`S` doesn't stop it."

### 3A.2 The rule, exact

`toggle(appState:source:windowID:)` now decides in this order:

1. If the resolved `target` window is running a slideshow → **stop**
   it (`reason=user_toggle`). This is the normal single-window case.
2. Else, if the caller passed **no** explicit `windowID` **and** any
   window is running a slideshow → **stop that run**
   (`runs.keys.sorted().first`). A bare `S` therefore always stops a
   visible slideshow regardless of which window the run is keyed
   under.
3. Else → **start** a run on `target`.

Implemented in
`SlideshowController.toggle(...)`. The explicit-`windowID` path
(MCP targeting a specific window, multi_window.mdx §7.2) is
untouched — only the no-window keyboard/menu path gains the
"stop whatever is running" fallback.

### 3A.3 Verify

* Start a slideshow with `S`. Click a Finder window, then click back
  onto the ImageGlass window (this can re-resolve the frontmost id).
  Press `S`. Expected: the slideshow **stops**. It must not open a
  second run or leave the first running.
* On disk: exactly one `slideshow.toggle on=false … source=key:S`
  line follows, and there is **no** second `slideshow.toggle on=true`
  line for that press.

## 3B. A Transient Empty List Pauses — It Never Auto-Stops

The second half of the "stops on its own" bug. The advance loop
reads `AppState.orderedNavigationFiles` **live on every tick** (§0A.1,
§2.3). That list is recomputed from the walker snapshot, so a scope
re-evaluation, a directory refresh, or an FSEvents-driven rewalk can
leave it **transiently empty** for a frame or two — even though the
user did nothing and files still exist on disk.

### 3B.1 Old behavior (the bug)

`advance()` hit `files.isEmpty` and called
`stop(reason: "no_in_scope_files")`, tearing the whole run down. To
the user the slideshow "just stopped by itself" mid-show.

### 3B.2 New behavior

An empty navigation list at advance time is treated as a **paused
tick**, not a stop:

1. The run stays alive; `SlideshowController.runs[windowID]` is
   **not** removed.
2. The countdown is re-armed for another full interval via
   `scheduleTick(windowID:every:)`.
3. On the next tick the list is re-read. As soon as **any** in-scope
   file is present, the slideshow advances normally.

Implemented in `SlideshowController.advance(windowID:)` — the
`files.isEmpty` branch now re-arms the tick instead of calling
`stop`.

### 3B.3 The only things that stop a running slideshow

After this spec, a running slideshow stops **only** for one of:

| Cause | `reason=` | Section |
| --- | --- | --- |
| User toggles it off (`S` / `Space` / menu / MCP) | `user_toggle` | §3, §3A |
| End of list with `loop == false` | `end_of_list` | §7.7 |
| App quit / window close | *(no stop line; state flushed)* | §14 |

A transient or even total empty list no longer appears in this
table. Excluding every file (include_checks.mdx §10) now **pauses**
the show on its current frame; it resumes when a file is re-included.
This supersedes the old `reason=no_in_scope_files` auto-stop
(include_checks.mdx §10.3 is updated in lockstep).

### 3B.4 Verify

* Start a slideshow over `~/Pictures/tour/`. From an MCP client, run
  `refresh_directory` (forces a rewalk) a few times while the show
  runs. Expected: the slideshow keeps advancing; it does **not**
  stop. No `slideshow.stop reason=no_in_scope_files` line appears.
* Exclude every root (`Change Include ▸ Change Include Off`,
  include_checks.mdx §7.3). Expected: the current image stays on
  screen, the countdown keeps re-arming, and no stop line is written.
  Re-include a root and the show advances again within one interval.

## 4. Change Interval From Settings — Slider + Editable Number Field

This is the only place a user adjusts the interval persistently.
The Settings panel exposes the value with two controls that share
**one** binding: a horizontal slider for coarse change, and an
editable text field with stepper for exact entry.

### 4.1 Layout — what the row looks like

The Slideshow tab's Timing section, top row:

```
Interval (s)   [───────●──────────]   [  4.0 ▾ ]   s
                slider (1…600)         text field
                                      (stepper)
```

Three on-screen widgets bound to **one** model field —
`settings.slideshow.interval_seconds`:

* A `Slider` with `in: 1...600`.
* A `TextField` with a `Stepper` that increments/decrements by
  `0.5` seconds and accepts direct numeric entry.
* A static label `s` for unit.

The slider's thumb position and the text field's number stay in
sync to four significant digits at all times.

### 4.2 Preconditions

* §0 holds. The slideshow is **not** running (scenario 3 has run,
  or §0 is fresh). `settings.slideshow.interval_seconds == 4.0`.
* The user opens Settings via `⌘,` (or App ▸ Settings…). The
  Slideshow tab is the active tab (or the user clicks the
  *Slideshow* tab — `play.rectangle` icon).

### 4.3 Actions, in order

1. Drag the **slider** thumb from its current position (≈ 4 s)
   rightward until the text field reads `12.0`. Release the
   mouse.
2. Click into the **text field**. Use Cmd-A to select the
   contents, type `7`, press Return.
3. Click the stepper's up arrow twice (each click increments
   by 0.5).
4. Press `⌘W` to close the Settings window (or click red traffic
   light).

### 4.4 Expected result

After step 1 (slider drag):

* The text field updates **live** to track the slider —
  `4.0 → 5.5 → 7.2 → … → 12.0` as the thumb moves.
* `settings.slideshow.interval_seconds == 12.0` after release.
* The change is **debounced 300 ms** then written to
  `settings.json` atomically (write to a `.tmp` then rename).

After step 2 (typed value):

* The slider thumb snaps to the `7.0` position.
* `settings.slideshow.interval_seconds == 7.0`.
* After the 300 ms debounce, `settings.json` is rewritten.

After step 3 (stepper +0.5 × 2):

* The text field reads `8.0`. The slider thumb shifts
  accordingly.
* `settings.slideshow.interval_seconds == 8.0`. Written to
  disk after debounce.

After step 4 (close):

* No additional writes happen — the value was already on disk.
* The Slideshow tab's state survives a relaunch of the app.

### 4.5 Verify

* On screen: the slider thumb, the text field, and the stepper
  arrows always agree. There is no flash of an old value when
  any one widget updates.
* On disk, after step 1:

  ```sh
  grep '"interval_seconds"' ~/Library/Application\ Support/ImageGlass_Mac/settings.json
  ```

  reads `"interval_seconds": 12.0`.

* On disk, after step 3:

  ```sh
  grep '"interval_seconds"' ~/Library/Application\ Support/ImageGlass_Mac/settings.json
  ```

  reads `"interval_seconds": 8.0`.

* On disk: `log.log` contains, for each finalized write,

  ```
  ts=… app=settings.write key=slideshow.interval_seconds old=4.0 new=12.0 source=gui:settings corr=…
  ts=… app=settings.write key=slideshow.interval_seconds old=12.0 new=7.0 source=gui:settings corr=…
  ts=… app=settings.write key=slideshow.interval_seconds old=7.0 new=8.0 source=gui:settings corr=…
  ```

  Live drag does **not** produce a log line per thumb tick; only
  the debounced settle-value does.

### 4.6 Bounds and input validation

* The slider's range is `1.0 … 600.0` seconds, matching the
  existing slider in `SettingsScene.swift` and the
  [settings.mdx §4.6](../settings.mdx#46-slideshow) contract.
* The text field accepts any non-negative number entered. On
  commit (Return / blur / stepper), the value is clamped to
  `[1.0, 600.0]`. Values below 1.0 snap to 1.0; values above
  600.0 snap to 600.0; non-numeric input (e.g. `abc`) snaps back
  to the prior value and a one-line audit record is written
  with `ok=false err=invalid_input`.
* The stepper increment is `0.5 s`. Holding the up/down arrow
  repeats at the system-standard auto-repeat rate.
* Zero and negatives are not representable through the GUI.
  `SettingsValidation.swift` already coerces
  `interval_seconds <= 0` to `5.0` on load — that path is the
  safety net for hand-edited `settings.json`, not the GUI.

### 4.7 Mid-show changes

If the slideshow is **running** while the user opens Settings and
adjusts the slider / text field, the change applies on the **next**
interval boundary, not the current one:

* The controller observes the live `settings.slideshow.interval_seconds`
  field.
* The current countdown continues with the old interval until it
  hits zero and advances.
* The new countdown that follows the advance uses the new
  interval.

This avoids visually jarring mid-countdown snaps (e.g. dragging
the slider from 4 s to 30 s mid-countdown shouldn't extend the
currently-displayed slide; the user is editing the *next* slide's
budget).

### 4.8 Verify mid-show change

* Preconditions: slideshow running with `interval_seconds = 4.0`.
* Action: at ~`startInstant + 1.5 s`, with 2.5 s remaining on the
  countdown, drag the slider to 10.0.
* Expected: the countdown continues from 2.5 → 0, advances, then
  re-arms to 10.0 — visible jump in the countdown badge at the
  advance boundary. `settings.slideshow.interval_seconds == 10.0`
  after the 300 ms debounce.

## 5. `S` Is Focus-Aware — Suppressed in Text Inputs

A single-key, unmodified shortcut would steal `s` from anyone
typing in a text field. The binding is therefore **focus-aware**:
it fires only when the first responder is **not** a text input.

### 5.1 The rule, exact

`S` (no modifier) triggers `SlideshowController.toggle()` only if
**all** of these hold:

1. The key-down event arrives at the main viewer scene
   (`ImageGlassApp.Scene`'s key-event responder chain).
2. The current first responder is not an `NSTextField`,
   `NSTextView`, `NSSearchField`, `WKWebView`, or anything that
   reports `acceptsFirstResponder == true && isEditable == true`
   for text input.
3. There is no modifier pressed (`event.modifierFlags &
   .deviceIndependentFlagsMask` contains none of
   `.command / .option / .control / .shift`).
4. The key code matches `S` (`kVK_ANSI_S`).

Conditions 3 + 4 keep the existing `⌘S` (Save), `⇧⌘S` (Save As),
and `⌥⌘S` (View ▸ Start Slideshow) bindings untouched —
they take precedence because they carry modifiers and are
registered as menu-key equivalents, which the responder chain
checks first.

### 5.2 Preconditions

* §0 holds. The user is **not** in a text input.

### 5.3 Actions, in order

1. Click into a Settings text field (scenario 4 has a
   "interval seconds" text field — use that, with Settings open
   on the Slideshow tab).
2. Type `s`.

### 5.4 Expected result

* The character `s` is inserted at the cursor in the text field.
* No slideshow is started.
* No `slideshow.toggle` log line is written.
* `SlideshowController.shared.isRunning` is unchanged.

### 5.5 Verify

* On screen: the text field now contains the prior interval
  value with `s` appended (e.g. `4.0s`). The value is invalid;
  on Return / blur the input-validation path of §4.6 snaps it
  back. While editing, no slideshow window opens.
* On disk: no new `slideshow.toggle` line in `log.log`.
* In state: `SlideshowController.shared.isRunning == false`
  (assuming preconditions).

### 5.6 Verify the opposite — `S` outside text

* Click the slideshow tab body (not a field) to remove the field
  as first responder. Press `S`. Slideshow starts. This is just
  scenario 1 re-run from inside Settings to prove §5's rule fires
  only when text input is active, not whenever Settings is open.

## 6. Appropriate Zoom Mode — Re-Applied on Every Advance

This is the core visual rule the user asked for: every image the
slideshow advances to is presented in the **slideshow zoom mode**,
regardless of what the user did to the previous image during its
on-screen window.

### 6.1 The rule

Each time the controller advances:

1. The next file's `URL` is selected.
2. The slideshow window's `ViewerState.zoomMode` is set to the
   **slideshow zoom mode** — `.fit` (Zoom to Fit) for v1.
3. The slideshow window's `ViewerState.panOffset` is reset to
   `.zero`.
4. The slideshow window's `ViewerState.rotationQuarterTurns`,
   `flipHorizontal`, `flipVertical` are reset to defaults.
5. The slideshow window's `ViewerState.lockedZoom` is reset to
   `1.0` (no effect while `zoomMode == .fit`; matters if a
   future user-issued zoom in scenario 6.5 promoted the mode to
   `.lock`).
6. The canvas reload draws the new image fit-to-window.

`Zoom to Fit` is the default because it preserves aspect ratio,
centers the image, and matches the existing `SlideshowController`
behavior (`viewerState.zoomMode = .fit` on `start`). The reset
extends that one-time start setting to **every advance**.

### 6.2 The knob

A future settings key (`slideshow.zoom_mode`, see
[settings.mdx §4.6](../settings.mdx#46-slideshow)) can override
the default. For v1 the value is hard-coded to `.fit`. If/when
the knob lands, the controller reads it on every advance — no
restart required.

### 6.3 Preconditions

* Scenarios 1 + 2 completed. The slideshow is running, showing
  `002.jpg`, with interval `4.0`.

### 6.4 Actions, in order

1. While `002.jpg` is on screen, press `+` three times in the
   slideshow window to zoom in. This promotes
   `ViewerState.zoomMode` to `.lock` with `lockedZoom ≈ 1.728`
   (`1.2³`) per [hotkeys.mdx §5](../hotkeys.mdx#5-zoom-and-window-controls).
2. Press `→` once to pan right by 15% of the viewport
   ([hotkeys.mdx §4.3](../hotkeys.mdx#43-pan)).
3. Wait until the next advance fires (~ 4 s after `002.jpg` was
   loaded).

### 6.5 Expected result

* At the advance boundary the controller swaps in `notes_old.png`.
* The slideshow window's `ViewerState.zoomMode` is **forced back
  to `.fit`**.
* `panOffset` is **reset to `.zero`**.
* `lockedZoom` is reset to `1.0`.
* `notes_old.png` is drawn fit-to-window, centered.
* The user's manual zoom-in + pan from §6.4 are **discarded** —
  they applied only to the displayed instance of `002.jpg`, not
  to any subsequent image.

### 6.6 Verify

* On screen: `notes_old.png` is rendered fit-to-window with no
  pan offset; the user's prior zoom-in / pan are gone.
* In state, immediately after the advance:
  * `slideshowViewerState.zoomMode == .fit`
  * `slideshowViewerState.panOffset == .zero`
  * `slideshowViewerState.lockedZoom == 1.0`
  * `slideshowViewerState.rotationQuarterTurns == 0`
* On disk: the `slideshow.advance` log line reads
  `zoom_mode=fit`, regardless of any prior `lock` mode the user
  set:

  ```
  ts=… app=slideshow.advance from=…/002.jpg to=…/notes_old.png interval=4.0 zoom_mode=fit corr=…
  ```

### 6.7 What is NOT reset

* `settings.slideshow.*` keys (the source of truth) are
  untouched.
* The main viewer's `ViewerState` is untouched (the slideshow
  has its own `ViewerState` instance, per scenario 3.3).
* `colorChannel`, `colorFormat`, `showColorPicker` — color
  inspection options carry across advances. Resetting these
  would be surprising for a user comparing color across many
  images.
* `smoothInterpolation` — the user's choice of sharp vs. smooth
  scaling is preserved across advances.

## 10A. `N` / `P` — Next / Previous Image (Bare Letters)

`N` (next image) and `P` (previous image) are the bare-letter
companions to `↓` / `↑`. They use the same single global cursor
(§0A), the same navigation list (§1A), and the same skip rule
(§1A.2) as the slideshow, so the behavior is consistent across
all three navigation surfaces.

These two bindings replace earlier assignments on the same keys:
the v0 `N`-for-Normalize-zoom moves to `M`, and the v0
`P`-for-Copy-Path moves to `⇧P` (Shift + P). The hotkeys spec
(`hotkeys.mdx` §5 + §5A) is updated in lockstep.

### 10A.1 The rule, exact

A bare `N` press, when no modifier is held and no text input is
first responder (focus-aware per §5):

1. Calls `appState.selectNext(wrap: settings.viewer.wrap_at_ends)`.
2. `selectNext` reads `appState.orderedNavigationFiles` (§1A),
   finds `selectedFile` in the list, advances the index by 1,
   and writes the new path back to `selectedFile`.
3. The viewer canvas reloads with the new file; the panel
   highlight moves; the breadcrumb updates. All three surfaces
   stay in sync via the same `selectedFile.didSet` plumbing
   used by every other selection mutation.
4. If the **slideshow is running**, the press counts as user
   navigation: the new `selectedFile` becomes the new anchor for
   the next slideshow advance (§2A), and the countdown is
   **not** restarted. This is symmetric with a panel-row click.
5. If `selectedFile` is `nil` (empty viewer), `selectNext`
   selects the first file of the navigation list.
6. If `selectedFile` is the last file:
   * `wrap_at_ends == true` (default): wrap to the first file.
   * `wrap_at_ends == false`: no-op; the cursor stays put.

`P` is the symmetric previous-image binding — same list, same
focus rule, same wrap behavior, decrementing the index instead.

### 10A.2 Excluded items are skipped

`orderedNavigationFiles` already filters out files whose
`passesFilter == false` (excluded explicitly, or excluded by
inheritance from a folder, per §1A.2). `N` and `P` therefore
step **only** through included files. There is no separate skip
loop in the hotkey handler — the list is the filter.

A concrete check: in the §1A.1 fixture, if the user excludes
the entire `desert/` folder (set its include state to *exclude*
via the `I` hotkey on the folder row), then `orderedNavigationFiles`
becomes `[beach/001.jpg, beach/002.jpg, beach/003.jpg,
beach/notes_old.png, mountains/peak.heic, mountains/peak.jpg]`.
Pressing `N` from `beach/notes_old.png` advances to
`mountains/peak.heic` — `desert/*.jpg` is skipped.

### 10A.3 Focus-awareness

Same rule as `S` (§5): a key-down with no modifier on
`kVK_ANSI_N` or `kVK_ANSI_P` fires the navigation **only** when
the current first responder is not an editable text input. While
typing in the Settings search field or any other text field,
`n` and `p` insert their characters normally and do not move
the cursor.

### 10A.4 Verify

Use the §1A.1 fixture. Click `beach/002.jpg`.

* Press `N`. Expected: `selectedFile` becomes `beach/003.jpg`;
  the viewer and panel highlight move there.
* Press `N` again. Expected: `selectedFile` becomes
  `beach/notes_old.png`.
* Press `N` again. Expected: `selectedFile` becomes

## 11. Pacing — Drift and Stability

The interval is wall-clock-based, not "interval-from-last-tick"-based,
so a slow image-decode does not silently extend the run. The
acceptance bar is: over a 10-advance run, the wall-clock time
from start to the 10th advance is within ±200 ms of `10 ×
interval_seconds`.

### 11.1 Preconditions

* `interval_seconds = 4.0`.
* `~/Pictures/tour/` extended to ≥ 12 image files (so the
  5-file fixture wraps without affecting the measurement).

### 11.2 Actions, in order

1. Note `startInstant`.
2. Press `S`. Slideshow starts.
3. Let it run for 11 advances (so 10 measured intervals).
4. Press `S` to stop. Note `stopInstant`.

### 11.3 Expected result

* `stopInstant − startInstant` is in the range
  `[10 × 4.0 − 0.2, 10 × 4.0 + 0.2] = [39.8 s, 40.2 s]`.

### 11.4 Verify

* `log.log` contains the `app=slideshow.stop` line with
  `elapsed_s=` between 39.8 and 40.2.
* `log.log` contains 10 `slideshow.advance` lines (or 11 if
  the §7 loop boundary fired; the count is exact and
  acceptance-testable).

## 14. Local Storage — YAML-Backed Slideshow State

The user asked for two things beyond the toggle fix: *"we want a local
storage folder with YAML files for settings … our own directory … save
a lot of settings, like whether we are in slideshow mode or not, so we
restart and reload the state we had before."* This section is the
contract for that persistence.

### 14.1 The directory — our own, already ours

ImageGlass_Mac keeps all of its plain-text state under **one directory
it owns**:

```
~/Library/Application Support/ImageGlass_Mac/
```

defined by `AppPaths.macAppSupportDir`
(`code/Sources/ImageGlassCore/Paths.swift`, `macAppName =
"ImageGlass_Mac"`). This is the "local storage folder" — see
[local_storage.mdx](../local_storage.mdx) for the full layout. Every
file in it is human-readable YAML the user (or the MCP server) can open
and edit. The files relevant to the slideshow:

| File | What it holds |
| --- | --- |
| `settings.yaml` | App-level defaults, incl. the `slideshow.*` block (interval, loop, fullscreen, countdown, background, …). |
| `settings_window_<N>.yaml` | Per-window workspace state, incl. the per-window `slideshow:` block that records whether a slideshow was running at quit. |
| `directories.yaml` | Walker roots + `include_overrides` (what the slideshow walks). |

This mirrors how the **sister apps** persist state: each ships a
`justfile` and keeps its state as flat files under a per-app directory
(see the repo `CLAUDE.md` "Local Web Apps" and "Sister Web Apps"
tables). ImageGlass_Mac follows the same convention — one owned
directory, plain-text files, atomic writes.

### 14.2 The slideshow settings that persist

Two kinds of slideshow state are saved, in two different files:

1. **Configuration** (how the slideshow behaves) lives in the
   app-level `slideshow.*` block in `settings.yaml` /
   `SlideshowSettings`
   (`code/Sources/ImageGlassCore/Config/Settings.swift`):
   `interval_seconds`, `interval_to_seconds`, `use_random_interval`,
   `loop`, `fullscreen`, `show_countdown`, `hide_main_window`,
   `notify_every`, `notify_sound`, `background`, and
   `enabled_on_launch`. Documented in
   [settings.mdx §4.6](../settings.mdx#46-slideshow). These are
   written atomically by `SettingsStore` on every debounced change
   (§4.4).

2. **Mode / running state** (were we in slideshow mode?) lives in the
   per-window `slideshow:` block of `settings_window_<N>.yaml`
   (`WindowSlideshowState`,
   `code/Sources/ImageGlassCore/Config/WindowScopedSettings.swift`):

   ```yaml
   slideshow:
     was_running_on_quit: true    # were we mid-show when the app quit?
     current_index: 12            # which file the show was on
   ```

### 14.3 Save on quit

On clean quit (`NSApplication.willTerminateNotification`),
`WindowStateController.persistSlideshowStateOnQuit()` walks every open
window and calls `WindowState.persistSlideshow()`, which copies the
live `WindowState.slideshow.isRunning` flag (set by
`SlideshowController.start` / `.stop`) into
`slideshow.was_running_on_quit`, plus `current_index`, and flushes the
per-window YAML atomically.

So: quit while a slideshow is playing → `was_running_on_quit: true`.
Press `S` to stop, then quit → `was_running_on_quit: false`.

### 14.4 Restore on launch

At launch, after `bootstrap()` resolves the scope and the navigation
list is populated, `WindowStateController.restoreSlideshowIfNeeded()`
reads the target window's persisted flag. If
`slideshow.was_running_on_quit == true`, it calls
`SlideshowController.shared.start(source: "launch:restore")` so the app
comes back up **in the same slideshow mode the user left it in** — the
show is already playing, on the same interval, from the restored
selection.

This intentionally **reverses** the old "slideshow mode is not
persisted" rule (the previous
[local_storage.mdx §4.2.1](../local_storage.mdx#421-session-state--save-on-quit)
discarded it). The user explicitly wants the mode to survive a
restart, and it now does.

Best-effort guarantee: if the walk has not finished and
`orderedNavigationFiles` is still empty when the restore fires,
`start()` no-ops with a `no_files_available` audit line and the user
simply presses `S` to begin. No crash, no half-started run.

### 14.5 Verify the round-trip

* Press `S` to start a slideshow. Let it advance to `002.jpg`. Quit
  the app (`⌘Q`).
* On disk:

  ```sh
  grep -A2 'slideshow:' ~/Library/Application\ Support/ImageGlass_Mac/settings_window_1.yaml
  ```

  reads `was_running_on_quit: true` and a non-zero `current_index`.
* Relaunch. Expected: the slideshow is **already running** — the
  countdown badge is visible and the image advances after the
  interval, with no key press. `log.log` shows a
  `slideshow.toggle on=true … source=launch:restore` line.
* Press `S` (it stops — §3A), then quit and relaunch. Expected: the
  app comes up **stopped**; `was_running_on_quit: false` was written
  and no restore fired.

## 15. End-to-End Acceptance

A run of this spec passes when **all of**:

* §0A — the app has exactly one cursor (`AppState.selectedFile`);
  every selection-changing surface (panel click, `↑` / `↓` /
  `N` / `P`, drag-drop, MCP `select_file`, slideshow advance)
  writes through that field, and the directory panel highlight,
  the viewer canvas, and the slideshow window all read from it.
* §1A — the navigation list `AppState.orderedNavigationFiles`
  is the file tree's depth-first top-down visible order. Files
  whose `passesFilter == false` (excluded or inherit-excluded)
  are absent from the list, so slideshow / `N` / `P` step past
  them without a separate skip loop.
* §1 — pressing `S` with a non-empty file list starts the
  slideshow, opens the slideshow window, applies the slideshow
  zoom mode (`.fit`), and reads `interval_seconds` from
  `settings.json`. `slideshow.toggle on=true source=key:S` is
  in `log.log`.
* §2 — exactly at `start + interval_seconds` (±50 ms), the
  controller reads `AppState.selectedFile`, finds it in
  `AppState.orderedNavigationFiles`, advances to the next item
  in **that** list (not the legacy `resolvedFiles`), and re-arms
  the countdown. A `slideshow.advance` line is written with
  `zoom_mode=fit`.
* §2A — a click on a panel row during a running slideshow
  re-anchors `AppState.selectedFile` to the clicked file, and
  the **next** advance steps from the clicked file (not from
  the position the timer would have advanced to). The countdown
  is not restarted. The `slideshow.advance` audit line proves
  it via the `from=` field — `from` is the **clicked** file,
  not the pre-click position.
* §3 — pressing `S` while running stops the slideshow, closes
  the slideshow window, leaves `AppState.selectedFile` on the
  last-displayed image, and leaves the main viewer's
  `ViewerState` untouched. `slideshow.toggle on=false` is in
  `log.log`.
* §3A — `S` (and `Space`, menu, MCP with no explicit window) is a
  dependable play/stop toggle: if any run is active it stops it
  rather than spawning a second run, even when the frontmost
  window id has drifted from the run's window id.
* §3B — a transient or total empty `orderedNavigationFiles` at
  advance time **pauses** (re-arms the tick) instead of tearing
  the run down. The only stops are user-toggle, end-of-list with
  loop off, and quit. No `slideshow.stop reason=no_in_scope_files`
  is ever written.
* §4 — Settings ▸ Slideshow ▸ Timing exposes a **Slider** and an
  **editable text field with stepper** that share the same
  `settings.slideshow.interval_seconds` binding. Edits in
  either control update the other live. Debounced writes land
  in `settings.json`; only settle-values are journaled. The
  field clamps to `[1.0, 600.0]` and snaps non-numeric input
  back to the prior value with `ok=false err=invalid_input`.
* §4.7 — a mid-show interval change applies on the **next**
  advance, not the current countdown.
* §5 — `S` is focus-aware: it does NOT fire while a text input
  is first responder. `⌘S`, `⇧⌘S`, `⌥⌘S` (modifier presence)
  are unaffected.
* §6 — every advance re-applies the slideshow zoom mode and
  resets pan / rotation / flip / lockedZoom on the slideshow's
  `ViewerState`. Manual zoom or pan applied to one image is
  **discarded** on the next advance. The `slideshow.advance`
  line records `zoom_mode=fit` proof.
* §7 — at end of list, `loop=true` wraps to file #1 (advance
  line carries `wrap=true`); `loop=false` stops automatically
  (`slideshow.stop reason=end_of_list`).
* §8 — pressing `S` with an empty file list is a no-op: no
  window opens, `isRunning` stays `false`, one
  `slideshow.toggle on=true ok=false err=no_files_available`
  audit line is written.
* §9 — a one-image list still produces advance events that run
  the §6 zoom-reset on the single image.
* §10 — `⌥⌘S` (menu), `Space` (context-sensitive), `⌘S` /
  `⇧⌘S` (Save / Save As) continue to work without conflict.
  `source=` fields in `log.log` distinguish the three slideshow
  trigger paths.
* §10A — bare `N` and `P` step `AppState.selectedFile` through
  `AppState.orderedNavigationFiles` (next / previous). The keys
  are focus-aware (suppressed in editable text inputs), respect
  `settings.viewer.wrap_at_ends`, skip excluded and
  inherit-excluded files, and behave identically whether or
  not a slideshow is running. The earlier `N`-for-Normalize
  binding moves to `M`; the earlier `P`-for-Copy-Path binding
  moves to `⇧P`.
* §11 — over 10 advances, wall-clock elapsed is within ±200 ms
  of `10 × interval_seconds`.
* §12 — MCP `set_slideshow` produces results identical to the
  `S` key, differentiated only by `source=mcp:set_slideshow`
  in `log.log`. A one-shot `interval` argument runs without
  persisting; a `set_setting` call persists.
* §14 — slideshow state persists to the app's own YAML local
  storage directory (`~/Library/Application Support/ImageGlass_Mac/`).
  Configuration lives in `settings.yaml`/`SlideshowSettings`; the
  running-mode flag lives in each `settings_window_<N>.yaml`
  (`slideshow.was_running_on_quit` + `current_index`), written on
  quit by `WindowStateController.persistSlideshowStateOnQuit()` and
  read at launch by `restoreSlideshowIfNeeded()`. Quitting mid-show
  and relaunching brings the slideshow back up running
  (`source=launch:restore`); quitting stopped brings it back
  stopped.

If any step fails, the failing line in `log.log` (with its `corr=`
correlation id) is the first place to look — `grep corr=<id>
log.log` returns the audit line(s) interleaved with the matching
error record, reproducing the call in full.
