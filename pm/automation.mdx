# Automation, Panels, and Local Storage

This page describes, in detail, the architectural features that define
ImageGlass_Mac as a programmable image viewer: MCP integration, modular UI
panels, scope controls, and Local Storage. The high-level charter lives in
`overview.mdx`; this page is the deep dive.

---

## Why These Features?

ImageGlass_Mac extends the standard image-viewer experience for
**AI-driven and workflow-driven** use:

* The user defines *scopes* (which directories / files matter right now).
* The viewer reads those scopes from disk-backed Local Storage.
* An MCP server exposes Local Storage to external tools.
* Claude Code (or any MCP client) edits scopes, swaps directory sets, and
  re-triggers evaluation without the user opening the GUI.

The result: ImageGlass_Mac is both a viewer **and** a programmable
file-list surface that an AI assistant can drive.

---

## 1. MCP Support

### Goal

Embed ImageGlass_Mac with a **Model Context Protocol** server so that
external MCP clients — Claude Code in particular — can read and modify the
viewer's configuration over a well-defined protocol.

### Scope

* MCP exposes the Local Storage data model (see §4) as the primary surface.
* MCP is the **only sanctioned automation contract**. Ad-hoc IPC, file
  scraping, or other hooks are not part of the design.
* MCP must be stable enough that a Claude Code workflow can rely on it
  across releases.

### Non-goals

* The MCP server is **not** a remote-control surface for arbitrary GUI
  actions in v1. It targets scopes + Local Storage first; viewer-state RPCs
  can be added later if needed.

## 2. Modular UI Panels (New Column)

### Goal

The main window includes a dedicated **column** that hosts modular UI
panels.

### Initial Panel: Directory / Filename Panel

The first panel that ships is the **Directory / Filename panel**. It shows
the list of images and other files that are currently in scope.

It has two view modes:

| Mode | What it shows |
| --- | --- |
| **List view** | A flat list of filenames in scope. |
| **File-tree view** | The same files arranged in their directory hierarchy. |

The two views are **the same data**, just rendered differently. The data
source is always the resolved file list from Local Storage (§4).

### Panel Architecture

* The column is **panel-based**. Additional panels can be added later
  (e.g., a scope-editor panel, a metadata panel, a recent-runs panel).
* Adding a new panel must not require modifying the core viewer.
* Panels can be shown, hidden, or reordered by the user.

---

## 3. Scope Controls

### Goal

Give the user explicit, first-class control over **what files are in scope**.

### What a Scope Can Express

* One or more **directories** to include.
* One or more **directory hierarchies** (recurse).
* **Inclusion criteria**: glob patterns, file extensions, name filters.
* **Exclusion criteria**: ignore patterns, hidden files, etc.
* **Named rule sets** that can be referenced and reused.

### Where Scope Lives

Scope is persisted in **Local Storage** (§4) as plain text files. There is
no hidden binary scope state.

### Who Edits Scope

Either:

* The user, via the UI (eventually a scope-editor panel).
* Claude Code or another MCP client, via the MCP server (§5).

---

## 4. Local Storage Feature

> **Feature name:** Local Storage.
> **Form:** Plain text files in a known local-filesystem directory.

### What It Stores

Each Local Storage entry records:

* The **last time** the scope was evaluated / loaded.
* The list of **images and other files** that were resolved into the file
  list on that run.
* The list of **source directories or criteria** that drove the scope.

### Runtime Behavior

1. Local Storage holds a list of **directories or criteria**.
2. ImageGlass_Mac walks the matching **directory hierarchies**.
3. It finds the files that satisfy the scope.
4. That resolved list is what shows up in the Directory / Filename panel
   (§2).
5. The run is recorded (timestamp + resolved file list) back into Local
   Storage.

### Why Plain Text?

Plain text on disk is a **deliberate architectural decision**:

* It lets the MCP server read/write the same files the GUI uses.
* It lets Claude Code diff and edit scope definitions like any other code.
* It lets the user inspect what the viewer "thinks" is in scope without
  launching the GUI.
* It makes the system debuggable, scriptable, and version-controllable.

A binary settings blob would close off all of those workflows.

---

## 5. MCP-Driven Editing of Local Storage

Because Local Storage is plain text **and** is fronted by the MCP server,
Claude Code can:

* **Read** the current scope definitions.
* **Modify** scope definitions in place.
* **Swap in** new directory lists.
* **Change inclusion / exclusion criteria**.
* **Trigger re-evaluation** so the panel reflects the new scope.

All of this happens **without the user editing settings inside the GUI**.

### Typical Workflow

1. User asks Claude Code something like:
   *"Show me all PNGs under `~/screenshots/2026-06/` plus my Figma exports
   directory, but exclude anything with `_old` in the name."*
2. Claude Code calls the MCP server.
3. The MCP server updates the Local Storage scope file.
4. ImageGlass_Mac re-evaluates the scope.
5. The Directory / Filename panel updates to show the resolved file list.
